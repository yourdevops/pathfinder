# Phase 06.4: CI Step Identity and Change Tracking - Research

**Researched:** 2026-02-16

## Summary

This phase addresses four gaps (GAP-01 through GAP-04) from the CI Workflows remediation plan. The work is entirely within the existing Django/GitPython stack and requires no new library dependencies. The core changes are: (1) adding `slug` and `status` fields to the `CIStep` model, (2) switching from shallow clone to full clone (or sufficient depth) in `scan_steps_repository` to enable per-file commit SHA computation via `git log`, (3) implementing SHA-based change detection with field-level comparison, and (4) replacing hard-delete with archival (soft-delete) including UI warnings and auto-cleanup.

All existing CI Steps, Workflows, and related data were deleted, so schema changes apply to empty tables. No data migration or backfill is needed -- fresh `makemigrations` and `migrate` are sufficient.

**Primary recommendation:** Implement in 3 plans: (1) model changes + migration, (2) rewrite `scan_steps_repository` for per-file SHA, slug derivation, collision detection, change detection, and archival, (3) update views/templates for archived step filtering and step update warnings.

### Pattern 1: Slug Derivation

**What:** Derive a URL-safe slug from `x-pathfinder.name` using Django's built-in `slugify`.
**When to use:** During step scanning, when creating/updating CIStep records.

```python
# Source: Django docs, verified via testing
from django.utils.text import slugify

def derive_step_slug(pathfinder_name: str, directory_name: str) -> str:
    """Derive slug from x-pathfinder.name, falling back to directory_name."""
    name = pathfinder_name or directory_name
    slug = slugify(name)
    if not slug:
        slug = slugify(directory_name)
    return slug
```

**Django `slugify` behavior (verified):**

| Input | Output |
|-------|--------|
| `"Test with Pytest"` | `test-with-pytest` |
| `"Docker Build"` | `docker-build` |
| `"Multiple   Spaces"` | `multiple-spaces` |
| `"Special/Characters!@#"` | `specialcharacters` |
| `"Under_score_test"` | `under_score_test` (keeps underscores) |
| `""` (empty) | `""` (empty -- needs fallback) |

Note: Django's `slugify` keeps underscores. The design doc examples only show spaces-to-hyphens. This is acceptable since underscores are valid in slugs and the design doc does not explicitly prohibit them.

### Pattern 2: Per-File Commit SHA via GitPython

**What:** Get the last commit SHA that modified a specific file.
**When to use:** During step scanning, for each discovered step file.

```python
# Source: GitPython docs + verified via testing
# CRITICAL: Requires full clone, NOT shallow clone (depth=1)

def get_per_file_sha(repo: git.Repo, file_path: str) -> str:
    """Get the last commit SHA that modified a specific file.

    Args:
        repo: GitPython Repo object (must not be shallow)
        file_path: Relative path from repo root, e.g., 'test/pytest/action.yml'

    Returns:
        40-character hex SHA string
    """
    return repo.git.log('-1', '--format=%H', '--', file_path).strip()
```

**Critical finding (verified):** In a shallow clone (depth=1), `git log -1 --format=%H -- <path>` returns the HEAD SHA for ALL files because only one commit exists. Per-file SHA requires the full history (or at least enough depth to reach each file's last-modified commit).

### Pattern 4: Soft-Delete via Status Field

**What:** Mark removed steps as "archived" instead of deleting them.
**When to use:** During step scanning when a step file is no longer found in the repository.

```python
# Replace hard delete:
#   CIStep.objects.filter(...).exclude(...).delete()
# With:
#   CIStep.objects.filter(...).exclude(...).update(status="archived")

# Cleanup: delete archived steps with zero references
CIStep.objects.filter(
    status="archived",
).exclude(
    id__in=CIWorkflowStep.objects.values_list('step_id', flat=True)
).delete()
```

### Anti-Patterns to Avoid

- **Do NOT use shallow clone for per-file SHA**: Shallow clones (depth=1) make `git log -- <path>` useless; all files show the same HEAD SHA. Use full clone or `--unshallow`.
- **Do NOT use `unique_together` for the engine+slug constraint**: Django docs recommend `UniqueConstraint` in `Meta.constraints` over the deprecated `unique_together`. The existing `["repository", "directory_name"]` constraint should also be migrated to `UniqueConstraint` style eventually, but that is out of scope.
- **Do NOT query CIWorkflowVersion manifest content to find step references**: The FK path is `CIWorkflowStep.step -> CIStep`. Checking if an archived step is "referenced by a workflow version" means checking if any `CIWorkflowStep` record points to it. The manifest content is text, not a queryable FK.
- **Do NOT add `has_step_updates` BooleanField to CIWorkflow**: The REMEDIATION.md suggests this, but it creates stale-flag problems. Instead, compute step update warnings at query time by checking if any step in the workflow has a `commit_sha` newer than when the workflow was last edited.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Slug generation | Custom regex replacer | `django.utils.text.slugify` | Handles Unicode, edge cases, hyphen collapsing |
| Per-file git SHA | Manual git subprocess call | `repo.git.log('-1', '--format=%H', '--', path)` | GitPython already wraps git CLI; handles errors, encoding |
| Soft-delete framework | Custom manager/mixin | Simple `status` CharField + queryset filter | Only one model needs it; full soft-delete library is overkill |
| Change classification | Complex diff algorithm | Simple field-value comparison (old != new) | Fields are JSONField/CharField; equality check is sufficient |

**Key insight:** Every piece of this phase uses existing Django and GitPython primitives. No new libraries or custom frameworks are needed.

## Common Pitfalls

### Pitfall 1: Shallow Clone Breaks Per-File SHA

**What goes wrong:** `git log -1 --format=%H -- <path>` returns HEAD SHA for all files in a shallow clone.
**Why it happens:** Shallow clone (depth=1) has exactly one commit; every file's "last modified commit" is the single available commit.
**How to avoid:** Change `clone_repo_shallow(depth=1)` to a full clone (remove `depth` and `single_branch` params) or fetch with `--unshallow` after initial clone.
**Warning signs:** All steps show the same `commit_sha` value (which equals the repo HEAD).
**Tradeoff:** Full clone is slower and uses more disk for large repos. For typical steps repos (small, few hundred files), this is negligible.

### Pitfall 2: Slug Collision Across Repositories

**What goes wrong:** Two different steps from different repositories derive the same `(engine, slug)` pair.
**Why it happens:** Different repos can have steps with the same `x-pathfinder.name` for the same engine.
**How to avoid:** At import time, check for existing slug+engine before creating. If collision exists from a DIFFERENT repository, skip the step and log a warning. If same repository, this is an update.
**Warning signs:** IntegrityError on the unique constraint.

### Pitfall 3: Archived Step Still Referenced by Workflows

**What goes wrong:** Trying to hard-delete an archived step that is still referenced by `CIWorkflowStep`.
**Why it happens:** `CIWorkflowStep.step` has `on_delete=models.PROTECT`, which raises `ProtectedError` on delete.
**How to avoid:** The cleanup task MUST check for references before deleting: `CIStep.objects.filter(status="archived").exclude(id__in=CIWorkflowStep.objects.values_list('step_id', flat=True)).delete()`.
**Warning signs:** `django.db.models.deletion.ProtectedError` at runtime.

### Pitfall 4: discover_steps Returns Relative Directory Path, Not File Path

**What goes wrong:** Using `directory_path` directly in `git log -- <path>` without appending the engine file name.
**Why it happens:** `discover_steps` returns `directory_path` (e.g., `test/pytest`), but git log needs the file path (e.g., `test/pytest/action.yml`).
**How to avoid:** Construct the file path: `f"{raw_step['directory_path']}/{engine_file_name}"`.
**Warning signs:** `git log` returns empty string (no file at that path).

### Pitfall 5: Stale `has_step_updates` Flag

**What goes wrong:** A boolean flag on `CIWorkflow` indicating "steps have updates" never gets cleared.
**Why it happens:** No clear trigger to reset the flag -- editing the workflow? Publishing a version? Manual dismiss?
**How to avoid:** Instead of a persistent flag, compute the warning at view time. Compare step `updated_at` timestamps against workflow `updated_at` or version `published_at`. Or, store a `last_change_type` on CIStep that gets set during scan and displayed in the UI without a workflow-level flag.
**Recommendation:** Store `last_change_type` (null/interface/metadata) on CIStep itself, set during scan. The workflow detail view checks if any of its steps have `last_change_type = 'interface'` and `updated_at > workflow.updated_at`.

### Pitfall 6: Name Change = Slug Change = New Step

**What goes wrong:** Renaming `x-pathfinder.name` for an existing step changes its slug, which means it's treated as a new step (old slug gets archived, new slug gets created).
**Why it happens:** Step identity is `(engine, slug)`, and slug derives from name.
**How to avoid:** This is BY DESIGN per the spec: "Slug change = new step (old removed)". The scan logic should handle this correctly: old slug is not found -> archived; new slug is created as new. Log a warning about the rename.
**Warning signs:** A step appears to be "archived" and a "new" step appears with similar content.

@task(queue_name="steps_scan")
def cleanup_archived_steps() -> dict:
    """Delete archived steps that are not referenced by any workflow."""
    from core.models import CIStep, CIWorkflowStep

    referenced_step_ids = CIWorkflowStep.objects.values_list('step_id', flat=True)
    deleted_count, _ = (
        CIStep.objects
        .filter(status="archived")
        .exclude(id__in=referenced_step_ids)
        .delete()
    )
    return {"deleted": deleted_count}

## Key Design Decisions

### 1. Full Clone vs. Unshallow

**Recommendation: Full clone (no `depth` parameter).**

Options:
- **Full clone** (`git.Repo.clone_from(url, dir, branch=branch, single_branch=True)` without `depth`): Simple, one operation, all history available.
- **Shallow then unshallow** (`clone --depth=1` then `fetch --unshallow`): Two operations, more complex, no real benefit since we need all history anyway.
- **Deeper shallow** (`clone --depth=N`): Unreliable -- we don't know how deep history goes for each file.

Full clone with `single_branch=True` is the best balance: fetches all history for only the default branch.

### 2. Where to Store Change Classification

**Recommendation: `last_change_type` field on `CIStep`.**

Options:
- **`has_step_updates` on `CIWorkflow`**: Stale flag problem (when to clear it?).
- **`last_change_type` on `CIStep`**: Set during scan, queryable from workflow detail view. Reset to null on next scan if unchanged.
- **Separate changelog model**: Overkill for R1; save for R3 (StepSyncEntry).

The `last_change_type` on CIStep is simple and avoids stale-flag issues. The workflow detail view queries: "Do any of my steps have `last_change_type='interface'` and `updated_at > my.updated_at`?"

### 3. Collision Detection Strategy

**Recommendation: Skip-and-warn per design doc.**

When a slug already exists for the same engine from a DIFFERENT repository:
1. Skip the step (do not create/update)
2. Log a warning with the conflicting repository name
3. The warning is visible in scan logs (future R3 StepSyncEntry)

When a slug exists for the same engine from the SAME repository: this is a normal update path.

### 4. `file_path` Field on CIStep

**Recommendation: Add `file_path` CharField to store the relative path from repo root.**

Currently only `directory_name` (basename) is stored. But per-file SHA requires the full path (`test/pytest/action.yml`), and the manifest generation also uses `directory_name` in `uses` references (`ci-steps/{directory_name}`). Storing `file_path` enables:
- Accurate per-file SHA computation on subsequent scans
- The design doc's step discovery: "Steps are matched to existing Catalog entries by `x-pathfinder.name`, not by file path. If a step is moved to a different directory, Pathfinder recognizes it as a new version of the same step."

The `file_path` updates on each scan regardless (tracking moves).

## Open Questions

1. **Clone Performance for Large Repos**
   - What we know: Full clone is needed for per-file SHA. Steps repos are typically small (a few hundred files).
   - What's unclear: If a customer has a very large steps repo, full clone could be slow.
   - Recommendation: Use `single_branch=True` for full clone (fetch only default branch history). If performance becomes an issue, consider `--filter=blob:none` (treeless clone) in a future optimization.

2. **Step Update Warning Persistence**
   - What we know: We need to show warnings on workflow detail when steps have interface changes.
   - What's unclear: When exactly should the warning clear? When the workflow is re-saved in the composer? When a new version is published?
   - Recommendation: Clear `last_change_type` on CIStep at the start of each scan (reset all active steps to null, then set for changed steps). The warning shows until the next scan finds no change. Separately, editing and re-saving a workflow in the composer naturally picks up the latest step data (via draft version regeneration), which is the intended resolution path per the design doc.

3. **Archived Step in Existing Workflow Draft**
   - What we know: The composer filters out archived steps from the step picker. But what about a draft workflow that already contains an archived step?
   - What's unclear: Should the workflow be broken or just show a warning?
   - Recommendation: Show a warning badge on the step in the workflow detail view ("This step is archived"). The composer edit mode should still show the archived step as "already in workflow" with a warning. This matches the design doc: "Workflows referencing it display a warning prompting the author to replace the archived step."
