Add outputs_schema to the CIStep model, extend the plugin interface with output reference formatting/parsing methods, update sync to extract and store outputs, add outputs display to step detail page, and extend manifest generation to include step IDs.

Purpose: This is the backend foundation for step output wiring. Without outputs_schema on CIStep, the composer cannot display outputs. Without format_output_reference(), there are no copy-paste references. Without step IDs in manifests, output references in the YAML will not resolve at runtime.

Output: CIStep model with outputs_schema, plugin interface with 3 new methods, GitHub plugin implementation, sync parsing outputs, step detail showing outputs, manifests with step IDs.
</objective>

**CIStep model (core/models.py):**
Add `outputs_schema` JSONField to CIStep, placed right after `inputs_schema`:
```python
outputs_schema = models.JSONField(
    default=dict,
    help_text="Output declarations from step definition (e.g., action.yml outputs block)",
)
```
Run `uv run python manage.py makemigrations core` then `uv run python manage.py migrate`.

**CICapableMixin (plugins/base.py):**
Add three new abstract methods after the existing `resolve_artifact_ref` method:

1. `format_step_id(self, step_slug: str) -> str` -- Return the engine-native step ID derived from slug. Raise NotImplementedError.

2. `format_output_reference(self, step_slug: str, output_name: str) -> str` -- Return the engine-native output reference string for copy-paste. Raise NotImplementedError.

3. `parse_output_reference(self, value: str) -> dict | None` -- Parse an input value to check if it matches the engine's output reference pattern. Returns dict with `step_slug` and `output_name` if matched, None otherwise. Raise NotImplementedError.

**GitHubPlugin (plugins/github/plugin.py):**
Implement the three new methods:

1. `format_step_id(self, step_slug)`: Return `step_slug` as-is (GitHub Actions step IDs are lowercase with hyphens).

2. `format_output_reference(self, step_slug, output_name)`: Return `f"${{{{ steps.{step_slug}.outputs.{output_name} }}}}"` (engine-native GitHub Actions expression syntax).

3. `parse_output_reference(self, value)`: Use regex pattern `r'^\$\{\{\s*steps\.([a-z0-9][a-z0-9-]*)\.outputs\.([a-z0-9][a-z0-9_-]*)\s*\}\}$'` to match and extract step_slug/output_name. Return dict or None.

Add a module-level compiled regex `_GH_OUTPUT_REF_RE` above the class for reuse.

Update `parse_step_file()` to include outputs:
```python
"outputs": file_content.get("outputs", {}),
```
Add this line after `"inputs": file_content.get("inputs", {}),` in the return dict.

Update `generate_manifest()` to add step `id` to each user-composed step entry:
After `step_entry = {"name": step.name, ...}` and before `if ws.input_config:`, add:
```python
step_entry["id"] = self.format_step_id(step.slug)
```
This inserts `id` into the manifest YAML for each step. The auto-injected checkout step does NOT get an id (it has no outputs consumers need).
