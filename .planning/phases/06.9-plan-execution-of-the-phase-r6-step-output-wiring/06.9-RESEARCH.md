# Phase 06.9: Step Output Wiring - Research

**Researched:** 2026-02-17
**Domain:** CI workflow step output wiring (model, sync, composer UX, manifest generation)
**Confidence:** HIGH

## Summary

This phase implements GAP-21 from REMEDIATION.md: allowing CI steps to declare outputs, workflow composers to wire one step's output to another step's input via copy-paste of engine-native references, and CI plugins to translate those references during manifest generation.

The existing codebase provides a strong foundation. The `CIStep` model already has `inputs_schema` (JSONField) and the `CIWorkflowStep.input_config` (JSONField) already stores per-step input overrides. The workflow composer already has Alpine.js infrastructure for step configuration expansion, drag-and-drop reordering, runtime constraint validation (both client-side and server-side), and the `copyBtn` Alpine component is already registered for clipboard operations. The `generate_manifest()` method on `GitHubPlugin` already iterates `workflow_steps` in order and builds the YAML structure -- it needs extension to add step `id` attributes and resolve output references.

**Primary recommendation:** Add `outputs_schema` JSONField to CIStep, extend `parse_step_file()` to extract outputs, rename the composer "Configure inputs" to "Configure" with Inputs/Outputs tabs, add real-time inline validation for output references, and extend `generate_manifest()` to assign step IDs and resolve `$ref` entries to engine-native syntax. The reference format approach uses engine-native strings (e.g., `${{ steps.package-docker.outputs.image-ref }}`) displayed to users, keeping the plugin responsible for the exact format.

### Pattern 1: `outputs_schema` JSONField Structure
**What:** Mirror the existing `inputs_schema` structure for outputs.
**When to use:** Parsing outputs from action.yml during sync, displaying in catalog and composer.

The `inputs_schema` field stores the raw `inputs:` block from action.yml:
```python
# Current inputs_schema structure (from action.yml):
{
    "image-name": {
        "description": "Container image name",
        "required": true,
        "type": "string"
    },
    "dockerfile": {
        "description": "Path to Dockerfile",
        "required": false,
        "default": "Dockerfile"
    }
}
```

The `outputs_schema` should mirror this, storing the raw `outputs:` block:
```python
# Proposed outputs_schema structure (from action.yml):
{
    "image-ref": {
        "description": "Full image reference with tag",
        "value": "${{ steps.meta.outputs.tags }}"  # engine-native value expression
    },
    "image-digest": {
        "description": "Image digest (sha256)",
        "value": "${{ steps.push.outputs.digest }}"
    }
}
```

The `value` field stores the engine-native expression that the step implementation uses internally. This is opaque metadata from Pathfinder's perspective -- it is not used for wiring validation or manifest generation. Only the output names (keys) and descriptions matter for the composer UX.

**Confidence:** HIGH -- directly mirrors the existing inputs_schema pattern.

### Pattern 2: Plugin Methods for Output Reference Formatting
**What:** Two new methods on `CICapableMixin` to format step IDs and output references.
**When to use:** Composer UI (displaying copy-paste references) and manifest generation.

```python
# In plugins/base.py CICapableMixin:

    For GitLab CI: '$IMAGE_REF'
    For Bitbucket: '$IMAGE_REF'
    For Jenkins: '${env.IMAGE_REF}'

    raise NotImplementedError
```

```python
# In plugins/github/plugin.py GitHubPlugin:
```

**Confidence:** HIGH -- follows existing plugin interface patterns.

### Pattern 3: Output Reference Detection (Claude's Discretion)
**What:** Detect whether a user-entered value in an input field is a static value or an output reference.
**When to use:** Real-time validation on input change in the composer.

**Recommendation:** Use engine-specific regex patterns provided by the plugin.

For GitHub Actions, the pattern is:
```
\$\{\{\s*steps\.([a-z0-9-]+)\.outputs\.([a-z0-9-]+)\s*\}\}
```

This captures: `${{ steps.<step-slug>.outputs.<output-name> }}`

The detection approach:
1. The plugin provides a `parse_output_reference(value: str) -> dict | None` method
2. Returns `{"step_slug": "...", "output_name": "..."}` if it's a valid reference pattern
3. Returns `None` if it's a static value
4. Client-side mirrors this with a JavaScript regex in the Alpine component

```python
# In plugins/base.py CICapableMixin:
def parse_output_reference(self, value: str) -> dict | None:
    """Parse an input value to check if it's an output reference.

    Returns dict with step_slug and output_name if it matches
    the engine's output reference pattern, None otherwise.
    """
    raise NotImplementedError
```

For GitHub Actions:
```python
import re

_GH_OUTPUT_REF_RE = re.compile(
    r'^\$\{\{\s*steps\.([a-z0-9][a-z0-9-]*)\.outputs\.([a-z0-9][a-z0-9-]*)\s*\}\}$'
)

def parse_output_reference(self, value: str) -> dict | None:
    m = _GH_OUTPUT_REF_RE.match(value.strip())
    if m:
        return {"step_slug": m.group(1), "output_name": m.group(2)}
    return None
```

Client-side equivalent for Alpine (registered in the `workflowComposer` component):
```javascript
_parseOutputRef: function(value) {
    var m = value.trim().match(
        /^\$\{\{\s*steps\.([a-z0-9][a-z0-9-]*)\.outputs\.([a-z0-9][a-z0-9-]*)\s*\}\}$/
    );
    if (m) return {stepSlug: m[1], outputName: m[2]};
    return null;
}
```

**Confidence:** HIGH -- simple regex, well-defined GitHub Actions syntax.


**Important CSP constraint:** The `copyBtn` Alpine.data component takes a text parameter at initialization time. Since the output reference string depends on the step slug (which is dynamic), we need to either:
1. Pre-compute the reference strings and pass them via the step data from the server, or
2. Use a new method `formatOutputRef(step, outputName)` in the `workflowComposer` component and nest `copyBtn` within it.

**Recommendation:** Pre-compute output references on the server side and include them in the step data passed to Alpine. The view passes a `step_output_refs_map` (step UUID -> {output_name -> reference_string}) via `json_script`, and each step in the composer carries its pre-computed references. This avoids CSP issues with dynamic `copyBtn` initialization.

```python
# In WorkflowComposerView._build_all_steps_context():
step_output_refs_map = {}
for s in all_steps:
    if s.outputs_schema:
        refs = {}
        for output_name in s.outputs_schema:
            refs[output_name] = ci_plugin.format_output_reference(s.slug, output_name)
        step_output_refs_map[str(s.uuid)] = refs
```

Then in the Alpine component, each step carries `output_refs`:
```javascript
this.steps.push({
    // ... existing fields ...
    outputs_schema: step.outputs_schema || {},
    output_refs: this.outputRefsMap[step.id] || {},
});
```

**Confidence:** HIGH -- follows existing patterns (copyBtn is already used with server-computed values).

### Pattern 5: Real-time Validation for Output References (Claude's Discretion)
**What:** Validate pasted output references in real-time as users type/paste into input fields.
**When to use:** On every input change event in the composer.

Validation checks (client-side):
1. Does the value match the output reference pattern? (regex)
2. If yes, does the referenced step slug exist in the current workflow steps?
3. If yes, does the referenced step appear BEFORE the current step in order?
4. If yes, does the referenced output name exist in that step's `outputs_schema`?

```javascript
validateInputValue: function(stepIndex, inputName, value) {
    var ref = this._parseOutputRef(value);
    if (!ref) return null;  // Static value, no validation needed

    // Find referenced step in current workflow
    var refStepIndex = -1;
    var refStep = null;
    for (var i = 0; i < this.steps.length; i++) {
        if (this.stepSlugMap[this.steps[i].id] === ref.stepSlug) {
            refStepIndex = i;
            refStep = this.steps[i];
            break;
        }
    }

    if (!refStep) {
        return "Step '" + ref.stepSlug + "' not found in this workflow";
    }
    if (refStepIndex >= stepIndex) {
        return "Step '" + ref.stepSlug + "' must appear before this step";
    }
    var outputs = refStep.outputs_schema || {};
    if (!outputs[ref.outputName]) {
        return "Output '" + ref.outputName + "' not found on step '" + ref.stepSlug + "'";
    }
    return null;  // Valid
}
```

For tracking validation errors per input, add a `validation_errors` dict to each step:
```javascript
this.steps.push({
    // ... existing fields ...
    validation_errors: {}  // {input_name: "error message" or null}
});
```

The inline error display:
```html
<p x-show="step.validation_errors[entry.name]"
   class="text-xs text-red-400 mt-0.5"
   x-text="step.validation_errors[entry.name]"></p>
```

**Re-validation on reorder:** After drag-and-drop reorder completes, iterate all steps and re-validate all input_config values that contain output references. This catches broken ordering.

```javascript
// In onDrop(), after updating orders:
this.revalidateAllOutputRefs();
```

**Confidence:** HIGH -- mirrors the existing runtime constraint checking pattern (client-side + server-side).

### Pattern 6: Server-Side Validation (Claude's Discretion)
**What:** Validate output references in `WorkflowComposerView.post()` before saving.
**When to use:** On every composer save (both create and edit).

```python
def _validate_output_references(steps_data, ci_plugin=None):
    """Validate all output references in step input configs.

    For each step's input_config values, detect output references
    and validate:
    1. Referenced step exists in the workflow
    2. Referenced step appears before the consuming step
    3. Referenced output exists in the step's outputs_schema
    """
    errors = []
    # Build slug -> (index, CIStep) mapping
    step_map = {}  # slug -> (index, CIStep)
    ordered_steps = []

    for i, entry in enumerate(steps_data):
        try:
            ci_step = CIStep.objects.get(uuid=entry.get("id"))
            step_map[ci_step.slug] = (i, ci_step)
            ordered_steps.append((i, ci_step, entry.get("input_config", {})))
        except CIStep.DoesNotExist:
            continue

    for step_index, ci_step, input_config in ordered_steps:
        for input_name, value in input_config.items():
            if not isinstance(value, str):
                continue
            # Try to parse as output reference
            ref = ci_plugin.parse_output_reference(value) if ci_plugin else None
            if not ref:
                continue
            ref_slug = ref["step_slug"]
            ref_output = ref["output_name"]

            if ref_slug not in step_map:
                errors.append(
                    f'"{ci_step.name}" input "{input_name}": '
                    f'step "{ref_slug}" not found in workflow'
                )
                continue
            ref_index, ref_step = step_map[ref_slug]
            if ref_index >= step_index:
                errors.append(
                    f'"{ci_step.name}" input "{input_name}": '
                    f'step "{ref_slug}" must appear before this step'
                )
                continue
            outputs = ref_step.outputs_schema or {}  # NEW FIELD
            if ref_output not in outputs:
                errors.append(
                    f'"{ci_step.name}" input "{input_name}": '
                    f'output "{ref_output}" not found on step "{ref_slug}"'
                )
    return errors
```

**Confidence:** HIGH -- follows the exact pattern of `_validate_step_order()`.

### Pattern 7: Manifest Generation with Step IDs and Output Resolution
**What:** Extend `generate_manifest()` to assign `id` to each step and resolve output references in `input_config`.
**When to use:** Every time a manifest is generated (draft preview, publish, push).

Current manifest step generation (from `plugins/github/plugin.py`):
```python
step_entry = {
    "name": step.name,
    "uses": uses_ref,
}
if ws.input_config:
    step_entry["with"] = ws.input_config
```

Updated to include `id` and resolve references:
```python
step_entry = {
    "name": step.name,
    "id": self.format_step_id(step.slug),  # NEW
    "uses": uses_ref,
}
if ws.input_config:
    # Resolve output references in input_config values
    resolved_config = {}
    for key, value in ws.input_config.items():
        # Output references are already stored as engine-native strings
        # (e.g., "${{ steps.package-docker.outputs.image-ref }}")
        # No resolution needed -- they pass through as-is
        resolved_config[key] = value
    step_entry["with"] = resolved_config
```

**Key insight:** Since the user decision is to use engine-native format in the composer (copy-paste), the `input_config` values already contain the engine-native expression strings. The manifest generator just needs to:
1. Add `id` to each step (derived from slug)
2. Pass `input_config` values through as-is (they are already engine-native)

This means `input_config` stores strings like `"${{ steps.package-docker.outputs.image-ref }}"` directly, not `{"$ref": "package-docker.image-ref"}`. The REMEDIATION.md mentioned a `$ref` format, but the CONTEXT.md decision to use engine-native copy-paste format supersedes this. The engine-native string IS the stored value.

**However**, for server-side validation, we need to be able to parse these engine-native strings back to (step_slug, output_name). This is what `parse_output_reference()` does.

**Confidence:** HIGH -- simple extension of existing manifest generation.

### Anti-Patterns to Avoid
- **Building a custom dropdown/select UI for output wiring:** The user decided on copy-paste of engine-native references. Do NOT build a picker/dropdown for selecting outputs from preceding steps.
- **Aggregating outputs from all preceding steps:** The Outputs tab shows only the current step's own outputs. Do NOT build a panel that shows "available outputs from above."
- **Storing an intermediate `$ref` format:** Since the user chose engine-native format, store the actual engine-native string in `input_config`. Do NOT add an intermediate `{"$ref": "slug.output"}` format that requires translation.
- **Creating a separate JS file for Alpine logic:** Per CLAUDE.md, Alpine logic for the composer stays in the `workflowComposer` Alpine.data() registration in the template's `<script>` block. Do NOT create separate JS files.
- **Using arrow functions or semicolons in Alpine directives:** The CSP build only supports single expressions. All multi-step logic must be in the registered `workflowComposer` component methods.

## Common Pitfalls

### Pitfall 1: CSP Build Expression Limits
**What goes wrong:** Writing Alpine directives with semicolons or multi-step logic inline.
**Why it happens:** Natural JavaScript habits conflict with the CSP build's single-expression parser.
**How to avoid:** All multi-step logic (validation, tab switching, reference formatting) goes into `workflowComposer` Alpine.data() methods. Inline directives only call single methods: `@click="setConfigTab(index, 'outputs')"`.
**Warning signs:** Console errors about expression parsing in Alpine.

### Pitfall 2: Dynamic copyBtn Initialization
**What goes wrong:** Trying to use `x-data="copyBtn(dynamicExpression)"` where `dynamicExpression` is computed at runtime. The CSP build cannot evaluate dynamic expressions in `Alpine.data()` initialization.
**Why it happens:** Output reference strings depend on the step slug which is known at render time but changes per step.
**How to avoid:** Pre-compute all output reference strings server-side and pass them via `json_script`. Each step in the Alpine data carries its own `output_refs` map. Use `x-data="copyBtn(step.output_refs[entry.name])"` -- this works because `step.output_refs[entry.name]` is a simple property access, not a computed expression.
**Warning signs:** copyBtn showing empty text or throwing errors.

### Pitfall 3: Manifest Hash Stability
**What goes wrong:** Adding `id` to steps changes the generated YAML, which changes the manifest hash. Existing published versions would no longer match builds.
**Why it happens:** The manifest hash includes the full YAML content including step IDs.
**How to avoid:** This is expected and acceptable. All existing CI data was deleted (per REMEDIATION.md: "No data migrations are needed"). New manifests will have step IDs from the start. When editing an existing workflow, the draft version will have the new format with IDs, and publishing creates a new authorized version with the new hash.
**Warning signs:** None -- this is by design.

### Pitfall 4: Output Reference Validation After Reorder
**What goes wrong:** User pastes an output reference, then drags the referenced step below the consuming step. The reference becomes invalid but shows no error.
**Why it happens:** Input values are only validated on change, not on reorder.
**How to avoid:** Call `revalidateAllOutputRefs()` after every drag-and-drop reorder. This iterates all steps and re-validates all input_config values that match the output reference pattern.
**Warning signs:** User creates an invalid workflow by reordering steps after wiring outputs.

### Pitfall 5: Step Slug Map Not Available Client-Side
**What goes wrong:** Client-side validation needs to map step slugs to step indices, but the Alpine component only has step UUIDs and names.
**Why it happens:** The slug is on the CIStep model, not currently passed to the composer's Alpine data.
**How to avoid:** Include `slug` in the step data passed to Alpine. Both the `initial_steps_json` (for existing steps) and the `addStep` method need to carry the slug. Also pass a `stepSlugMap` (UUID -> slug) for looking up slugs from UUIDs.
**Warning signs:** Validation cannot find referenced steps by slug.

### Pitfall 6: Outputs on Steps That Don't Declare Them
**What goes wrong:** Showing an Outputs tab on steps that have no outputs (like setup steps).
**Why it happens:** Not checking for empty outputs_schema before showing the tab.
**How to avoid:** Only show the Outputs tab button when `hasOutputs(step)` returns true (non-empty outputs_schema). The existing pattern uses `hasInputs(step)` for the Configure button -- apply the same conditional.
**Warning signs:** Empty Outputs tab on setup steps.

```python
import re

_GH_OUTPUT_REF_RE = re.compile(
    r'^\$\{\{\s*steps\.([a-z0-9][a-z0-9-]*)\.outputs\.([a-z0-9][a-z0-9-]*)\s*\}\}$'
)

def parse_output_reference(self, value: str) -> dict | None:
    m = _GH_OUTPUT_REF_RE.match(value.strip())
    if m:
        return {"step_slug": m.group(1), "output_name": m.group(2)}
    return None
```

## Open Questions

1. **Should the Configure button/gear icon show when a step has outputs but no inputs?**
   - What we know: Currently `hasInputs(step)` controls the Configure button visibility. After this phase, we need "Configure" visible if the step has inputs OR outputs.
   - What's unclear: Whether to show the button only when inputs exist (current) or also when outputs exist.
   - Recommendation: Show the Configure button (now just gear icon for "Configure") whenever the step has inputs OR outputs. Change `x-show="hasInputs(step)"` to `x-show="hasInputs(step) || hasOutputs(step)"`. If only outputs exist, default the tab to 'outputs'.

2. **Step slug availability in the `stepsJson()` serialization**
   - What we know: The hidden form field serializes `steps` via `stepsJson()`. The server currently looks up steps by UUID. But the server-side validation also needs slugs.
   - What's unclear: Whether to include slugs in the serialized JSON or re-fetch them server-side.
   - Recommendation: Include slug in the serialized step data (it's already available in the Alpine state). The server can use it for validation but still authoritative-checks via UUID lookup.

3. **How to handle the `title` attribute on the Configure button**
   - What we know: Currently it says "Configure inputs". With the rename, it should say "Configure".
   - Recommendation: Change `title="Configure inputs"` to `title="Configure"`. Simple.
