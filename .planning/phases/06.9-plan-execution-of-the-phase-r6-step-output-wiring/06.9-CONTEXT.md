# Phase 6.9: Step Output Wiring - Context

**Gathered:** 2026-02-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Steps declare outputs in the catalog (parsed during sync). The workflow composer lets users wire one step's output to another step's input via copy-paste of engine-native references. CI plugins translate these references to engine-native syntax during manifest generation. Addresses GAP-21 from REMEDIATION.md.

</domain>

<decisions>
## Implementation Decisions

### Composer wiring UX
- Rename "Configure inputs" to just "Configure" on each step in the composer
- Configure panel expands with two tabs: **Inputs** (default) and **Outputs**
- Outputs tab shows only the current step's own outputs (not aggregated from all preceding steps)
- Output references displayed in **engine-native format** (e.g., `${{ steps.package-docker.outputs.image-ref }}` for GitHub Actions)
- The exact reference format is a **plugin responsibility** — each CI plugin provides the copy-paste string for its engine
- Each output reference has a **copy icon button** (reuse existing `copyBtn` Alpine component pattern)
- Users manually paste output references into downstream steps' input fields

### Output display in catalog
- Step detail page shows Inputs and Outputs as **separate sections** (distinct cards/blocks)
- Each output displays the same info format as inputs (name, description, type if available)
- Steps catalog list (table view) has **no output indicator** — outputs only visible on step detail page
- Output changes (removed/renamed) rely on the **existing interface change badge system** — no extra output-specific warnings

### Validation feedback
- **Real-time validation** on input change — validate as soon as user pastes/types an output reference
- **Inline error under input** — red text below the field explaining the issue (e.g., "Step 'package-docker' must appear before this step", "Output 'image-ref' not found on step 'package-docker'")
- Step reorder that breaks existing output wiring: **allow the reorder, show inline errors** on broken references — user must fix them
- **Client + server validation** — Alpine.js validates in real-time for UX; server validates on save for safety (matches existing runtime constraint checking pattern)

### Engine scope
- GitHub Actions is the priority engine for output reference resolution in manifest generation (per REMEDIATION.md)
- Other engines (GitLab CI, Bitbucket, Jenkins) follow the same pattern but are lower priority — design the plugin interface to support them

### Claude's Discretion
- How to detect that a pasted value is an output reference vs a static value (pattern matching on engine-native syntax)
- Alpine.js component structure for the Configure tabs and real-time validation
- Server-side validation implementation details
- `outputs_schema` JSONField structure and sync parsing logic
- Step `id` assignment strategy in manifest generation

</decisions>

<specifics>
## Specific Ideas

- "Let's use GH native `${{ steps.package-docker.outputs.image-ref }}`. Make the exact format a responsibility of the Plugin, not core!" — engine-native references, plugin-driven format
- "Rename the step's 'Configure inputs' to be just 'Configure'. When clicking on it, it expands with an Inputs tab (default) and Outputs (in a ready-to-copy format, that can be put into the downstream Steps' Inputs)" — explicit copy-paste workflow, not dropdown pickers
- Outputs displayed "Same as Inputs" on step detail page — consistent parameter presentation

</specifics>
