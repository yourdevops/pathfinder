#### 2. Patch Auto-Update End-to-End Flow

**Test:**
1. Create a service with auto-update enabled
2. Publish a patch version (e.g., 1.0.0 -> 1.0.1) for the workflow
3. Observe GitHub repo for PR creation
4. Merge the PR and trigger a build
5. Verify service status transitions: pending_pr -> synced

**Expected:**
- Flash message shows "Auto-update queued for N service(s)" on publish
- PR appears in service repo with branch `pathfinder/ci-manifest`
- PR title/body indicate manifest update to new version
- After merge, build verification updates service status to "synced"

**Why human:** Involves external GitHub API, webhook delivery, real repo state, and async task execution timing.

#### 3. Digest PR Reuse on Multiple Updates

**Test:**
1. Publish patch version 1.0.1 (creates PR)
2. Before merging, publish patch version 1.0.2
3. Observe that PR is updated with new commit (no second PR created)

**Expected:**
- Only one PR exists on branch `pathfinder/ci-manifest`
- PR commit history shows two commits (one for 1.0.1, one for 1.0.2)
- Both manifest updates present in the PR

**Why human:** Requires observing GitHub PR state across multiple publish events.

#### 4. Version Cleanup Execution

**Test:**
1. Change retention period to 1 day in CI Configuration settings
2. Create and revoke a version
3. Wait 2 days (or adjust server time)
4. Click "Run Cleanup Now" in settings
5. Verify old version's manifest_content is cleared but hash remains

**Expected:**
- Flash message: "Version cleanup has been queued"
- After task executes, old version record exists but manifest_content is empty
- manifest_hash field still populated
- Last cleanup timestamp updates in settings UI

**Why human:** Requires time manipulation, observing background task execution, and verifying database state.

#### 5. Cleanup Button Enqueues Task (Not Synchronous)

**Test:**
1. Add logging to `cleanup_old_versions()` function

3. Observe that flash message appears immediately
4. Check background worker logs for cleanup execution

**Expected:**
- Flash message appears instantly (not delayed by cleanup execution)
- Cleanup logs appear in background worker, not request/response cycle
- Settings page remains responsive during cleanup

**Why human:** Verifying async task execution vs synchronous call requires observing worker logs and timing.
