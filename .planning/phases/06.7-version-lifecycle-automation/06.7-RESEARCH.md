# Phase 06.7: Version Lifecycle Automation - Research

**Researched:** 2026-02-16
**Domain:** CI Workflow version auto-update, PR digest management, version retention/cleanup
**Confidence:** HIGH

## Summary

Phase 06.7 addresses GAP-11 (Auto-Update on Version Publish) and GAP-12 (Version Cleanup and Retention) from REMEDIATION.md. When a CI Workflow version is published, services using that workflow should automatically receive updated manifests via PR (patch bumps only). Old versions should be cleaned up per retention policy.

The codebase already has all foundational pieces: `CIWorkflowVersion` with status lifecycle, `push_ci_manifest` task for PR-based manifest delivery, `verify_build` with sync detection (`pending_pr` to `synced`), the `semver` library for version comparison, and the `django-tasks` + `django-scheduled-tasks` infrastructure for background and cron tasks. The implementation primarily involves connecting these pieces: a post-publish auto-update task, a PR digest mechanism (reusing existing branches/PRs), a service-level auto-update toggle, global retention settings, and a daily cleanup task.

**Primary recommendation:** Build on existing `push_ci_manifest` task for auto-updates, refactor its branch naming from `pathfinder/ci-manifest-{service.name}` to the fixed `pathfinder/ci-manifest` per repo, and add PyGithub `repo.get_pulls(head=...)` for digest PR detection.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Auto-update policy:**
- Patch-only auto-update: only patch bumps (same major.minor) trigger auto-push to services
- Minor/major version bumps require manual service update
- Service-level boolean toggle: "Auto-update on patch versions" (on/off, default on)
- No "pinned" concept -- services have a current selected version; the toggle controls whether patch auto-updates happen
- Claude's Discretion: FK timing -- whether to update service's ci_workflow_version immediately or wait for sync detection

**PR management:**
- Auto-updates always go through PR (direct push was deferred)
- Digest PR pattern: if an open PR on branch `pathfinder/ci-manifest` exists, add a new commit updating the manifest file (do NOT force-push -- other code changes on that branch must be preserved)
- Branch name: `pathfinder/ci-manifest` (fixed name per repo, not per-service or per-version)
- If the PR was previously merged (branch deleted), recreate branch from default HEAD, commit new manifest, open new PR

**Retention and cleanup:**
- Global retention settings only (no per-workflow overrides)
- Default retention period: 1 year (365 days) for manifest content
- Manifest content is cleared after retention period; hash is retained for verification
- Version record deletion follows versioning.md rules: no Build references, not latest version, older than retention period
- Deletion guards: prevent CIWorkflowVersion deletion while Build.workflow_version references exist; prevent CIWorkflow deletion while any version has build references

**Update status tracking:**
- New service-level status: `pending_pr` -- means a manifest PR is open and awaiting merge
- Sync detection happens inside existing `verify_build` task: after build verification, if verified version matches the pending version on default branch, transition to `synced`
- Service detail page shows current status only (synced/pending_pr/outdated), no auto-update history log
- Flash message on publish: "Version X.Y.Z published. Auto-update queued for N services."

**Cleanup UI and scheduling:**
- Version cleanup runs as a daily scheduled task (matching steps repo scan pattern)
- Settings page gets a manual "Run cleanup now" button that enqueues the task to background worker and reports status back to UI
- Management command also available for CLI/cron use

- Scheduled task implementation details (queue, frequency)
- Flash message implementation approach
- Settings page layout for retention configuration and cleanup button

### Pattern 1: Post-Publish Auto-Update Task Chain
**What:** After `PublishVersionView` saves the authorized version, enqueue `auto_update_services` task. This task queries services, compares versions via semver, and enqueues individual `push_ci_manifest` calls for eligible services.
**When to use:** Every time a workflow version is published.

```python
# In PublishVersionView.post(), after draft.save():
from core.tasks import auto_update_services
result = auto_update_services.enqueue(
    workflow_id=workflow.id,
    version_id=draft.id,
)
# Count eligible services for flash message
eligible_count = Service.objects.filter(
    ci_workflow=workflow,
    auto_update_patch=True,
    ci_workflow_version__isnull=False,
).count()
messages.success(
    request,
    f"Version {version_number} published. Auto-update queued for {eligible_count} services."
)
```

### Pattern 2: Semver Patch-Only Comparison
**What:** Use `semver.Version.parse()` to compare the service's current pinned version with the newly published version. Only auto-update if it is a patch bump on the same major.minor.
**When to use:** Inside `auto_update_services` task for each service.

```python
import semver as semver_lib

new_v = semver_lib.Version.parse(new_version.version)
current_v = semver_lib.Version.parse(service.ci_workflow_version.version)

# Patch-only: same major, same minor, only patch differs
is_patch = (new_v.major == current_v.major
            and new_v.minor == current_v.minor
            and new_v > current_v)
```

### Pattern 3: Digest PR via Branch Reuse
**What:** Instead of creating a new branch per update, use a fixed branch name `pathfinder/ci-manifest` per repo. If the branch and open PR exist, add a commit to the existing branch. If the branch was deleted (PR merged), recreate from default HEAD.
**When to use:** Inside the refactored `push_ci_manifest` task.

```python
# Check for existing open PR on the digest branch
# PyGithub: repo.get_pulls(state='open', head='owner:pathfinder/ci-manifest')
prs = repo.get_pulls(state='open', head=f'{owner}:pathfinder/ci-manifest')
open_pr = next(iter(prs), None)

if open_pr:
    # Branch exists, PR is open -- just add commit to existing branch
    plugin.update_or_create_file(config, repo_name, manifest_path, content, message,
                                  branch='pathfinder/ci-manifest')
    pr_url = open_pr.html_url
else:
    # No open PR -- create branch from default HEAD, commit, open new PR
    plugin.create_branch(config, repo_name, 'pathfinder/ci-manifest', source_branch)
    plugin.update_or_create_file(config, repo_name, manifest_path, content, message,
                                  branch='pathfinder/ci-manifest')
    pr_result = plugin.create_pull_request(...)
    pr_url = pr_result['html_url']
```

### Anti-Patterns to Avoid
- **Creating per-service branches for auto-update:** Decision locks branch name to `pathfinder/ci-manifest` per repo. Using per-service branches would create branch sprawl and break the digest PR pattern.
- **Force-pushing to the digest branch:** Explicitly prohibited. Other code changes on that branch must be preserved. Always add new commits.
- **Updating the service FK immediately on publish:** Risky because the PR has not been merged yet. The existing sync detection in `verify_build` already handles the `pending_pr` to `synced` transition. Recommend keeping the service pinned to its current version until sync detection confirms the merge.
- **Running cleanup synchronously in the view:** Settings page "Run cleanup now" button should enqueue the task, not run it inline. Cleanup may take time if there are many versions.

## Common Pitfalls

### Pitfall 1: Branch Name Migration
**What goes wrong:** The current `push_ci_manifest` uses `pathfinder/ci-manifest-{service.name}` as branch name. Switching to `pathfinder/ci-manifest` means existing open PRs on the old branch pattern will be orphaned.
**Why it happens:** Branch naming convention change.
**How to avoid:** The refactored task should first check for PRs on the old branch pattern and handle gracefully (log a warning, ignore). New auto-updates always use the new fixed branch name. Since this is early development with no production data, this is low risk.
**Warning signs:** Duplicate PRs in service repos (old pattern + new pattern).

### Pitfall 2: Race Condition in Digest PR Detection
**What goes wrong:** Two auto-update tasks for different services in the same repo run concurrently. Both check for open PRs, both find none, both create branches and PRs.
**Why it happens:** Background tasks execute asynchronously without locking.
**How to avoid:** Process services per-repo sequentially within the auto-update task. Group services by `repo_url`, then process each repo's services in sequence. The `push_ci_manifest` calls for different repos can run in parallel.
**Warning signs:** Duplicate PRs created for the same repo.

### Pitfall 3: Cleanup Deleting Content Needed for Verification
**What goes wrong:** Clearing `manifest_content` on old versions while builds are still being verified against those hashes.
**Why it happens:** The content is cleared but the hash is retained. Build verification uses the hash, not the content, so this is actually safe.
**How to avoid:** Verify that `verify_build` only uses `manifest_hash` for matching (confirmed: line 676 of tasks.py filters by `manifest_hash`, never reads `manifest_content`). Content clearing is safe.
**Warning signs:** None -- this is a non-issue once confirmed, but worth documenting.

### Pitfall 4: FK Update Timing Confusion
**What goes wrong:** Updating `service.ci_workflow_version` to the new version immediately on auto-update, before the PR is merged. This would cause `verify_build` sync detection to expect the new version on the default branch, but the old version is still there.
**Why it happens:** Temptation to update the FK eagerly.
**How to avoid:** Update `service.ci_workflow_version` to the new version immediately (so `verify_build` can detect when the PR is merged and the new version appears on default branch). The existing sync detection logic at `tasks.py:697-705` already checks `version_match.id == service.ci_workflow_version_id` -- so updating the FK immediately is correct. Set `ci_manifest_status = 'pending_pr'` at the same time.
**Warning signs:** Service stuck in `pending_pr` after PR merge if FK was not updated.

### Pitfall 5: PyGithub `get_pulls(head=...)` Requires Owner Prefix
**What goes wrong:** Calling `repo.get_pulls(head='pathfinder/ci-manifest')` returns wrong results.
**Why it happens:** The `head` parameter requires the format `owner:branch_name` (e.g., `myorg:pathfinder/ci-manifest`).
**How to avoid:** Always use the correct format: `repo.get_pulls(state='open', head=f'{owner}:pathfinder/ci-manifest')`. Extract owner from parsed repo URL.
**Warning signs:** Empty results when PRs should exist, or all open PRs returned.

### Pitfall 6: Creating PR When Branch Has No Diff
**What goes wrong:** If the manifest content is identical to what is already on the default branch, creating a PR results in an empty PR or a GitHub API error ("No commits between base and head").
**Why it happens:** Version header change in the manifest means the content is always different (version string changes). But if a service was already on this version somehow, the content could match.
**How to avoid:** The manifest header includes the version string (e.g., `# Version: 1.0.1`), so publishing a new version always produces a different manifest. This is a non-issue for auto-updates triggered by a new publish.

```python
# Source: core/views/ci_workflows.py, PublishVersionView + SuggestVersionView patterns
import semver as semver_lib

def is_patch_bump(current_version_str: str, new_version_str: str) -> bool:
    """Check if new_version is a patch bump from current_version (same major.minor)."""
    try:
        current = semver_lib.Version.parse(current_version_str)
        new = semver_lib.Version.parse(new_version_str)
        return (new.major == current.major
                and new.minor == current.minor
                and new > current)
    except ValueError:
        return False
```

## Discretion Recommendations

### FK Update Timing: Update Immediately
**Recommendation:** Update `service.ci_workflow_version` to the new version immediately when the auto-update PR is created, and set `ci_manifest_status = 'pending_pr'`.

**Rationale:** The existing sync detection logic in `verify_build` (tasks.py:697-705) already checks `version_match.id == service.ci_workflow_version_id`. For sync detection to work correctly when the PR is merged and a new build runs, the service must already point to the new version. If we wait, there is no mechanism to know when to update the FK.

**Flow:**
1. Publish version 1.0.1 (patch bump from 1.0.0)
2. Auto-update task sets `service.ci_workflow_version = v1.0.1`, `service.ci_manifest_status = 'pending_pr'`
3. PR is created/updated in repo
4. Developer merges PR
5. Build triggers, runs on default branch with v1.0.1 manifest
6. `verify_build` finds hash match for v1.0.1, checks `version_match.id == service.ci_workflow_version_id` -- match!
7. Transitions `ci_manifest_status` from `pending_pr` to `synced`

### Scheduled Task Details
**Recommendation:** Use `@cron_task(cron_schedule="0 3 * * *")` on `steps_scan` queue (reuse existing queue since cleanup is similar in nature to scan tasks -- infrequent, batch operations). Run at 03:00 UTC, one hour after the steps repo scan.

### Flash Message Implementation
**Recommendation:** Use Django's `messages.success()` directly in `PublishVersionView.post()`. Count eligible services before enqueuing the task (a simple queryset count is fast and does not need to be deferred). This gives immediate feedback to the user.

### Settings Page Layout
**Recommendation:** Add a new "CI Configuration" section under Settings in the sidebar nav (between "Audit Log" and "Notifications"). The page shows:
1. Retention settings (days input field, save button)
2. Last cleanup run info (timestamp, counts)
3. "Run Cleanup Now" button (HTMX POST that enqueues task and shows toast)

Use the existing `SiteConfiguration` singleton model to add retention fields. This follows the established pattern and avoids creating a separate configuration model.

## Open Questions

1. **Multiple workflows per repo**
   - What we know: The fixed branch name `pathfinder/ci-manifest` is per repo. Multiple services in different projects could share the same repo.
   - What's unclear: If two workflows auto-update for the same repo simultaneously, they write different manifest files (e.g., `ci-python-uv.yml` and `ci-java-maven.yml`) to the same branch. This should work since `update_or_create_file` operates on individual files.
   - Recommendation: This is safe because each workflow has a unique manifest path (`ci-{workflow.name}.yml`). The digest PR collects all manifest updates. No action needed beyond the per-repo sequential processing for race condition prevention.

2. **Management command vs cron_task for CLI cleanup**
   - What we know: User wants both a management command and a daily scheduled task.
   - What's unclear: Whether the management command should run synchronously or also enqueue a task.
   - Recommendation: Management command should call the cleanup function directly (synchronous), while the cron_task enqueues the same function as a background task. This matches Django conventions where management commands are for direct CLI use.
