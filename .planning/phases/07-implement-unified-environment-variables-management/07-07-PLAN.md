**Architecture overview:** Replace the HTMX per-row interaction model with an Alpine.js `envVarEditor` component that manages all current-level env vars as client-side state. Upstream vars remain server-rendered and read-only. The component includes a Save button that POSTs all current-level vars as JSON to a bulk save endpoint.

**Step 1: Register `envVarEditor` Alpine component in `base.html`**

Add the `envVarEditor` component via `Alpine.data()` in the `alpine:init` event listener in `theme/templates/base.html` (where `dropdown` and `copyBtn` are already registered). This is a shared component used across project, environment, and service settings pages, so it belongs in `base.html` per CLAUDE.md rules.

The component manages:
- `vars` array: current-level editable variables `[{key, value, lock, description}]`
- `editingIdx`: index of row currently being edited (-1 = none)
- `addingNew`: boolean, true when add-row is shown
- `newVar`: object for the new variable being added
- `dirty`: boolean, true when changes have been made
- `saving`: boolean, true during save request
- `saveUrl`: the bulk save endpoint URL
- `csrfToken`: CSRF token for the POST

Methods (all using `function()` syntax, NOT arrow functions, per CSP build constraint):
- `initFromData(initialVars, saveUrl, csrfToken)`: Initialize state from server-rendered data
- `startEdit(idx)`: Set `editingIdx = idx`, save backup of original values for cancel
- `cancelEdit()`: Restore from backup, set `editingIdx = -1`
- `saveEdit()`: Validate (key format, not empty), apply changes, set `editingIdx = -1`, set `dirty = true`
- `startAdd()`: Set `addingNew = true`, `newVar = {key: '', value: '', lock: false, description: ''}`
- `cancelAdd()`: Set `addingNew = false`
- `confirmAdd()`: Validate key (uppercase, not empty, not duplicate, not PTF_*), push to `vars`, set `addingNew = false`, set `dirty = true`
- `removeVar(idx)`: Splice from array, set `dirty = true`
- `toggleLock(idx)`: Toggle `vars[idx].lock` (only if value not empty), set `dirty = true`
- `updateKey(val)`: Uppercase and sanitize the key in `newVar`
- `bulkSave()`: POST `vars` array as JSON to `saveUrl` with CSRF token. On success, set `dirty = false`, `saving = false`. On error, show alert. Use `fetch()` API.

**IMPORTANT CSP constraint:** Every Alpine directive in templates must be a single expression. No semicolons, no arrow functions. Complex logic must be in the component methods. For example:
- `@click="startEdit(idx)"` - valid (single method call)
- `@click="editingIdx = idx; backup = vars[idx]"` - INVALID (semicolon)
- `:class="editingIdx === idx ? 'bg-dark-surface' : ''"` - valid (single ternary)

**Step 2: Rewrite `_env_var_container.html`**

Replace the current HTMX-based container with an Alpine-driven container. The template receives `resolved_vars`, `current_level`, `is_editable`, `show_empty_warning`, and `env_var_bulk_save_url` from the view context.

Structure:
```
<div x-data="envVarEditor" x-init="initFromData({{ current_level_vars_json }}, '{{ env_var_bulk_save_url }}', '{{ csrf_token }}')">
  <!-- Heading with count -->

  <!-- Upstream vars (server-rendered, read-only, no Alpine) -->
  {% for var in resolved_vars %}
    {% if var.source != current_level %}
      {% include "_env_var_row.html" with var=var is_editable=False ... %}
    {% endif %}
  {% endfor %}

  <!-- Separator -->

  <!-- Current-level vars (Alpine-driven) -->
  <template x-for="(v, idx) in vars" :key="idx">
    <!-- Display row when editingIdx != idx -->
    <div x-show="editingIdx !== idx">
      [lock icon] [key with tooltip] = [value] [edit btn] [delete btn]
    </div>
    <!-- Edit row when editingIdx === idx -->
    <div x-show="editingIdx === idx">
      [lock toggle] [key input] = [value input] [desc toggle] [save btn] [cancel btn]
    </div>
  </template>

  <!-- Add row (shown when addingNew) -->
  <div x-show="addingNew">
    [key input] = [value input] [desc toggle] [confirm btn] [cancel btn]
  </div>

  <!-- Add variable button (hidden when addingNew) -->
  <button x-show="!addingNew" @click="startAdd()">Add variable</button>

  <!-- Save / Cancel bar (shown when dirty) -->
  <div x-show="dirty" class="...">
    <span>Unsaved changes</span>
    <button @click="bulkSave()" :disabled="saving">Save Changes</button>
  </div>
</div>
```

The upstream vars section still uses Django template `{% include %}` for `_env_var_row.html` with `is_editable=False`. These rows are purely server-rendered, read-only, no HTMX actions.

The current-level vars section uses Alpine `x-for` to render rows from the `vars` array. Each row has display and edit states toggled by `editingIdx`. This is ALL client-side — no HTMX swaps.

**Step 1: Add `EnvVarBulkSaveView` to `core/views/env_vars.py`**

Create a new view that accepts the full array of current-level env vars as JSON and replaces the entity's `env_vars` JSONField entirely.

```python
class EnvVarBulkSaveView(LoginRequiredMixin, ProjectContributorMixin, View):
    """Bulk save all env vars for the current level. Replaces the entity's env_vars entirely."""

    def post(self, request, *args, **kwargs):
        import json

        service_name = kwargs.get("service_name")
        env_name = kwargs.get("env_name")
        service, environment = _get_entity_and_context(self.project, service_name, env_name)
        current_level = _current_level_for(service_name, env_name)
        entity = _get_entity(self.project, current_level, service, environment)

        try:
            new_vars = json.loads(request.body)
        except json.JSONDecodeError:
            return HttpResponseBadRequest("Invalid JSON")

        if not isinstance(new_vars, list):
            return HttpResponseBadRequest("Expected JSON array")

        # Validate each var
        validated = []
        seen_keys = set()
        for var in new_vars:
            key = (var.get("key") or "").strip().upper()
            if not key or not re.match(r"^[A-Z][A-Z0-9_]*$", key):
                return HttpResponseBadRequest(f"Invalid key format: {key}")
            if key.startswith("PTF_"):
                return HttpResponseBadRequest("Cannot save system variables (PTF_* prefix)")
            if key in seen_keys:
                return HttpResponseBadRequest(f"Duplicate key: {key}")
            seen_keys.add(key)

            value = var.get("value", "")
            lock = bool(var.get("lock", False))
            description = (var.get("description") or "").strip()

            # Empty value cannot be locked
            if not value:
                lock = False
            # Environment level never locks
            if current_level == "environment":
                lock = False

            # Check upstream locks — cannot add a var whose key is locked upstream
            # (This check uses resolve_env_vars to get the upstream state)

            validated.append({"key": key, "value": value, "lock": lock, "description": description})

        # Check upstream lock conflicts
        resolved = resolve_env_vars(self.project, service, environment)
        upstream_locked_keys = {
            v["key"] for v in resolved
            if v["lock"] and v["source"] != current_level
        }
        for var in validated:
            if var["key"] in upstream_locked_keys:
                return HttpResponseBadRequest(
                    f"Variable '{var['key']}' is locked at an upstream level and cannot be overridden."
                )

        entity.env_vars = validated
        entity.save(update_fields=["env_vars", "updated_at"])

        return HttpResponse(status=200)  # Alpine fetch checks for 200
```

Use `ProjectOwnerMixin` instead of `ProjectContributorMixin` if the current level is "project" (project-level env var editing is owner-only per existing pattern). However, since the permission mixin is declared at class level, the simplest approach is to keep `ProjectContributorMixin` and add an explicit permission check in `post()` for the project level:

```python
if current_level == "project" and self.user_project_role != "owner":
    return HttpResponse(status=403)
```
