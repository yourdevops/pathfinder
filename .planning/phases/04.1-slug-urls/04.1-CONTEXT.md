# Phase 4.1: Replace UUID URLs with Slugs - Context

**Gathered:** 2026-01-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Transform all entity URLs from UUID-based to human-readable name-based slugs. The `name` field IS the slug — no separate slug field. Names must be slug-safe from creation. User URLs remain UUID-based for privacy.

</domain>

<decisions>
## Implementation Decisions

### URL Transition
- Break immediately — UUID URLs return 404, no redirects or dual access
- Remove UUID fields entirely from models (except User model)
- User URLs keep UUID: `/users/<uuid>/edit/` for privacy
- API endpoints mirror UI: slugs only (e.g., `/api/projects/my-project/`)

### Naming Constraints
- Name = Slug — single `name` field that IS the URL identifier
- Description field for human-friendly display names
- Format: lowercase + hyphens only (no uppercase, underscores, or spaces)
- Length limits (configurable in General settings by admins):
  - Default: 20 chars project, 40 chars service, 20 chars environment (DNS-friendly 63 total)
  - Optional: expand each to 63 chars (hierarchical DNS pattern)
  - Goal: encourage minimalistic, concise naming as user behavior
- Uniqueness: scoped by parent — same name allowed in different projects (e.g., environments)
- Collisions: block with error within same scope ("Name already exists")
- Reserved names: block names that conflict with resource actions (e.g., `delete`, `edit`, `create`, `settings`, `new`)

### Nested URL Structure
- Projects: top-level `/projects/{name}/`
- Environments: nested `/projects/{project}/environments/{name}/`
- Services (Phase 5): nested `/projects/{project}/services/{name}/`
- Blueprints: top-level `/blueprints/{name}/`
- Connections: top-level `/connections/{name}/`
- Groups: top-level `/groups/{name}/`
- Users: keep UUID `/users/{uuid}/edit/`

### Claude's Discretion
- Migration script approach for existing data
- URL pattern ordering to avoid conflicts
- Validation regex implementation
- SlugField vs CharField with validators

</decisions>

<specifics>
## Specific Ideas

- "Name = slug. No separate fields — use 'name'. This is rigid on purpose."
- "For any human-friendly names, there is a Description field"
- DNS-friendly naming encouraged: `<project>-<service>-<environment>` pattern at 63 chars max
- Configurable length limits are an admin option, not user-facing

</specifics>
