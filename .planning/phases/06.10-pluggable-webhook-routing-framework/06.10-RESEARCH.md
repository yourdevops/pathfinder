## Summary

This phase refactors webhook handling from two hardcoded core endpoints (`/webhooks/build/` and `/webhooks/steps-repo/`) into a pluggable framework where plugins own their webhook handlers. The existing plugin system already has a clean autodiscovery pattern and URL mounting via `get_urlpatterns()` -- webhook routes fit naturally into this existing pattern.

The codebase has exactly two webhook handlers in `core/views/webhooks.py`: `build_webhook` (handles GitHub Actions `workflow_run` events) and `steps_repo_webhook` (handles GitHub `push` events for steps repositories). Both are entirely GitHub-specific -- they reference `X-GitHub-Event` headers, use GitHub HMAC-SHA256 signature verification, and parse GitHub-specific payload structures. Moving these to `plugins/github/` is a clean extraction with no entanglement.

**Primary recommendation:** Extend the existing `get_urlpatterns()` pattern to include webhook routes (no new mixin needed). Use a single webhook endpoint per plugin (`/integrations/github/webhook/`) that dispatches internally based on `X-GitHub-Event` header. This matches how GitHub App webhooks work -- a single URL receives all event types, differentiated by headers.

### Pattern 1: Plugin Webhook Registration via Existing URL Pattern

**What:** Plugins already expose URL patterns via `get_urlpatterns()`, and core already mounts them under `/integrations/<plugin_name>/`. Webhook routes are simply additional entries in the plugin's `urls.py`.

**When to use:** Always. This is the simplest approach because the mounting infrastructure already exists.

**Example (plugins/github/urls.py after):**
```python
from django.urls import path
from . import views
from . import webhooks

app_name = "github"

urlpatterns = [
    path("create/", views.GitHubConnectionCreateView.as_view(), name="create"),
    path("manifest/callback/", views.GitHubManifestCallbackView.as_view(), name="manifest_callback"),
    path("installation/callback/", views.GitHubInstallationCallbackView.as_view(), name="installation_callback"),
    path("webhook/", webhooks.github_webhook, name="webhook"),
]
```

This results in the URL `/integrations/github/webhook/` automatically via the existing loop in `pathfinder/urls.py`:
```python
for plugin_name, plugin in registry.all().items():
    patterns = plugin.get_urlpatterns()
    if patterns:
        urlpatterns.append(
            path(f"integrations/{plugin_name}/", include((patterns, plugin_name), namespace=plugin_name))
        )
```

### Pattern 2: Single Webhook Endpoint with Event-Type Dispatch

**What:** GitHub sends all webhook events (push, workflow_run, pull_request, etc.) to a single URL. The event type is in the `X-GitHub-Event` header. The plugin's webhook handler inspects this header and dispatches to internal handlers.

**Why this is correct:** GitHub App webhooks have a single `hook_attributes.url`. Per-repo webhooks also use a single URL per hook. There is no GitHub mechanism for routing different event types to different URLs.

**Example (plugins/github/webhooks.py):**
```python
@csrf_exempt
def github_webhook(request):
    """Single entry point for all GitHub webhook events."""
    if request.method != "POST":
        return HttpResponse(status=200)

    event_type = request.headers.get("X-GitHub-Event", "")

    if event_type == "workflow_run":
        return _handle_workflow_run(request)
    elif event_type == "push":
        return _handle_push(request)
    else:
        return HttpResponse(status=200)  # Ignore unhandled events
```

### Pattern 3: Dynamic Webhook URL Resolution

**What:** Instead of hardcoding webhook URLs as string literals, use Django's `reverse()` to dynamically resolve the webhook URL from the plugin's registered URL name.

**Why critical:** Three places currently hardcode webhook URLs:
1. `core/views/services.py:814` -- `ServiceRegisterWebhookView`
2. `core/tasks.py:1022` -- `push_ci_manifest` task
3. `core/views/ci_workflows.py:93` -- Steps repo webhook registration
4. `plugins/github/views.py:157` -- App manifest `hook_attributes.url`

**Example:**
```python
from django.urls import reverse
webhook_path = reverse("github:webhook")  # returns "/integrations/github/webhook/"
webhook_url = f"{site_config.external_url.rstrip('/')}{webhook_path}"
```

### Anti-Patterns to Avoid
- **Hardcoded URL strings for webhook registration:** Use `reverse()` consistently so URLs are derived from the URL configuration, not duplicated as string literals.
- **Framework-level signature verification:** Each SCM provider has different auth (GitHub uses HMAC-SHA256, GitLab uses token header, Bitbucket uses different scheme). Forcing a common interface adds complexity for no benefit.
- **Creating a new WebhookCapableMixin just for URL registration:** The existing `get_urlpatterns()` already returns URL patterns. Adding webhook routes there is natural. A mixin would add abstraction without functional benefit. (See Discretion recommendation below.)
- **Multiple sub-routes per plugin:** GitHub uses a single URL per webhook config. Creating `/integrations/github/webhook/build/` and `/integrations/github/webhook/push/` would mean configuring multiple webhook URLs on GitHub, which is not how GitHub App webhooks work.

## Discretion Recommendations

### 1. WebhookCapableMixin vs. extending plugin_urlpatterns

**Recommendation: Do NOT use a mixin. Extend existing `get_urlpatterns()`.**

Rationale:
- The existing pattern in `pathfinder/urls.py` already iterates `plugin.get_urlpatterns()` and mounts them under `/integrations/<plugin_name>/`. Adding a `webhook/` path to the plugin's URL patterns is trivially simple.
- A `WebhookCapableMixin` would add a new interface contract (`get_webhook_urlpatterns()`) and require changes to the URL mounting loop -- more code for zero functional benefit.
- The webhook route is just another URL pattern. It does not need special framework support.
- If future use cases need framework-level awareness (e.g., "list all plugins that have webhooks"), a simple check `if 'webhook' in [p.name for p in plugin.get_urlpatterns()]` suffices without a mixin.

**Confidence: HIGH** -- verified by examining the existing URL mounting code in `pathfinder/urls.py` lines 73-81.

### 2. Single URL vs. multiple sub-routes per plugin

**Recommendation: Single URL per plugin (`/integrations/github/webhook/`).**

Rationale:
- GitHub App webhooks are configured with a single `hook_attributes.url`. You cannot specify different URLs for different event types.
- Per-repo webhooks also use a single URL per hook configuration (though you can create multiple hooks).
- GitLab similarly sends all events to one webhook URL, differentiated by headers.
- The `X-GitHub-Event` header cleanly identifies event type for internal dispatch.
- Multiple sub-routes would require registering multiple webhooks per repo, increasing API calls and complexity.

**Confidence: HIGH** -- verified against GitHub App Manifest documentation and existing `configure_webhook()` usage.

### 3. Internal dispatch mechanism

**Recommendation: Simple header-based if/elif dispatch in the webhook view function.**

Rationale:
- There are currently only 2 event types to handle: `workflow_run` and `push`.
- A registry/mapping pattern (e.g., `EVENT_HANDLERS = {"push": handle_push, ...}`) is reasonable but over-engineering for 2 handlers.
- A simple `if event_type == "workflow_run"` is readable, debuggable, and easy to extend.
- If a plugin grows to handle 5+ event types, it can refactor internally -- this is plugin-internal concern, not framework-level.

**Confidence: HIGH** -- pragmatic choice for current scale.

### 4. Core's future webhook routes

**Recommendation: Core registers its own webhook URL patterns the same way plugins do -- via `urlpatterns` in `pathfinder/urls.py`.**

Rationale:
- Core is not a plugin. It should not go through the plugin registry.
- If core needs webhook routes in the future (e.g., `/api/webhooks/slack/`), it adds them directly to `pathfinder/urls.py` or `core/urls.py` like any other Django URL.
- No special framework is needed. The "framework" is Django's URL routing.
- `core/views/webhooks.py` can be repurposed as a utility module if needed, or deleted if all content moves to the plugin.

**Confidence: HIGH** -- there are no current core webhook needs; designing for speculative future use is YAGNI.

## Common Pitfalls

### Pitfall 1: Forgetting to update all hardcoded webhook URL references
**What goes wrong:** After moving webhook handlers to the plugin, some code still constructs the old `/webhooks/build/` URL, causing webhooks to hit 404.
**Why it happens:** Webhook URLs are constructed as string literals in 3 separate locations in core code.
**How to avoid:** Grep for `/webhooks/` across the entire codebase. All instances must be replaced with `reverse()` calls or plugin-aware URL construction.
**Warning signs:** 404 errors in logs for POST requests to old webhook paths.

**Specific locations to update:**
1. `core/views/services.py:814` -- `ServiceRegisterWebhookView.post()` -- builds `/webhooks/build/`
2. `core/tasks.py:1022` -- `push_ci_manifest()` -- builds `/webhooks/build/`
3. `core/views/ci_workflows.py:93` -- `StepsRepoRegisterView.post()` -- builds `/webhooks/steps-repo/`
4. `plugins/github/views.py:157` -- `_build_manifest()` -- builds webhook path for App manifest
5. `core/urls.py:441-444` -- `webhooks_patterns` list definition
6. `pathfinder/urls.py:67` -- includes `webhooks_patterns` under `/webhooks/`

### Pitfall 2: Breaking the GitHub App manifest webhook URL
**What goes wrong:** The GitHub App manifest flow hardcodes the webhook URL derivation, causing newly created Apps to have wrong webhook URL.
**Why it happens:** `plugins/github/views.py:157` constructs `webhook_path` by string manipulation of the callback URL: `callback_path.rsplit("manifest/callback/", 1)[0] + "webhook/"`. This is fragile and assumes URL structure.
**How to avoid:** Use `reverse("github:webhook")` directly to get the canonical webhook path.
**Warning signs:** New GitHub Apps sending webhooks to wrong URL.

### Pitfall 3: Not handling the GitHub App webhook URL update
**What goes wrong:** Existing GitHub Apps continue sending webhooks to the old `/webhooks/build/` URL after the endpoint is removed.
**Why it happens:** The GitHub App's webhook URL is set during creation and not automatically updated.
**How to avoid:** Add a one-time migration step or admin action that uses the GitHub API to update the App's webhook URL. The `PATCH /app/hook/config` endpoint can update an App's webhook URL.
**Warning signs:** Webhook delivery failures in GitHub App settings after deployment.

### Pitfall 4: CSRF protection on webhook endpoints
**What goes wrong:** Webhooks fail with 403 Forbidden because Django's CSRF middleware rejects POST requests without CSRF tokens.
**Why it happens:** Webhook views must be `@csrf_exempt` because external services cannot provide CSRF tokens.
**How to avoid:** Ensure `@csrf_exempt` is applied to the new webhook view in `plugins/github/webhooks.py`. Already handled correctly in the current code.

### Pitfall 5: Import cycles when webhook handlers import core models
**What goes wrong:** Moving webhook handlers to plugins can create circular imports because the handlers import `core.models.Service`, `core.models.StepsRepository`, etc.
**Why it happens:** Plugin modules import core models, and core may import plugin modules during URL discovery.
**How to avoid:** Use lazy imports inside handler functions (the existing code already does this with `from core.models import ...` inside function bodies). Maintain this pattern.
**Warning signs:** `ImportError: cannot import name 'X' from partially initialized module`.

def verify_github_signature(request: HttpRequest, secret: str) -> bool:
    """Verify GitHub webhook HMAC-SHA256 signature."""

### Key Migration Details

**ServiceRegisterWebhookView webhook URL:** Currently at `core/views/services.py:814`, this constructs the webhook URL passed to `plugin.configure_webhook()`. This view is in core but calls into the plugin to actually register the webhook. The webhook URL it constructs must point to the plugin's webhook endpoint. Two approaches:
1. Use `reverse("github:webhook")` -- but this hardcodes "github" in core code, violating Architecture Rule 1.
2. Add a helper method to the plugin (e.g., `plugin.get_webhook_url(site_config)`) that the core view calls. This keeps the URL construction in the plugin.

**Recommendation:** Option 2 -- add `get_webhook_url(external_url: str) -> str` to `BasePlugin` (returns empty string by default), implement in `GitHubPlugin` to return `f"{external_url}{reverse('github:webhook')}"`. Core code calls `plugin.get_webhook_url(site_config.external_url)` without knowing the plugin name.

**Steps repo webhook registration:** Currently in `core/views/ci_workflows.py:93`, this constructs `/webhooks/steps-repo/` when registering steps repos. Since steps repo push events now go to the same plugin webhook endpoint (which dispatches by `X-GitHub-Event`), this should also use the plugin's webhook URL. The events parameter `["push"]` tells GitHub which events to send.

**push_ci_manifest webhook registration:** Currently in `core/tasks.py:1022`, after pushing a CI manifest, this registers a webhook for `workflow_run` events. Same URL change applies.
