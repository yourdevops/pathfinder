# Phase 6.10: Pluggable Webhook Routing Framework - Context

**Gathered:** 2026-02-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Refactor webhook handling from hardcoded core endpoints into a pluggable framework where plugins own their webhook handlers. Core provides routing infrastructure and can register its own webhook routes for future non-plugin use. Current `/webhooks/build/` and `/webhooks/steps-repo/` are GitHub-specific and must move into the GitHub plugin.

</domain>

<decisions>
## Implementation Decisions

### URL structure & routing
- Single webhook URL per plugin at `/integrations/<plugin>/webhook/`
- Plugin dispatches internally based on event type (e.g., GitHub uses `X-GitHub-Event` header to distinguish push, workflow_run, etc.)
- Ingress/firewall rule pattern: allow `/integrations/*/webhook/*`
- Plugins provide webhook URL patterns (similar to existing `plugin_urlpatterns`), core includes them under `/integrations/<plugin>/`
- Common pattern enables generic external traffic rules without per-plugin firewall changes

### Plugin webhook interface
- Plugin-internal signature verification — each plugin handles its own auth inside its webhook handler (no framework-level verify_signature contract)
- Common registration pattern for safety: plugins expose webhook URL patterns that the framework auto-discovers and mounts
- This follows the existing plugin URL autodiscovery pattern, extended to include webhook routes

### Migration of existing endpoints
- Remove `/webhooks/build/` and `/webhooks/steps-repo/` immediately — clean break, no redirects, no legacy code
- GitHub App webhook URL auto-updated via API to point to `/integrations/github/webhook/`
- Per-repo webhooks (registered via ServiceRegisterWebhookView) update on next interaction — when service is modified or webhook re-registered, new URL is used
- GitHub App manifest generation dynamically resolves webhook URL from plugin — no hardcoded paths

### Core vs. plugin ownership
- ALL GitHub-specific webhook handling moves into `plugins/github/` — aligns with Architecture Rule 1 (core must be plugin-agnostic)
- Core retains the routing framework and ability to register its own webhook routes for future use (e.g., Pathfinder API webhooks, Slack notifications)
- `core/views/webhooks.py` becomes the framework module (dispatcher/router), not the handler module

### GitHub App auto-configuration
- During App setup, webhook URL auto-configured to `/integrations/github/webhook/`
- Manifest `hook_attributes.url` resolved dynamically from plugin webhook URL

### Claude's Discretion
- Whether to use WebhookCapableMixin or just extend plugin_urlpatterns for webhook route registration
- Single URL vs. multiple sub-routes per plugin (Claude picks based on how GitHub/GitLab webhooks actually work)
- Internal dispatch mechanism within plugin webhook handler
- How core's future webhook routes are registered (same mechanism or separate)

</decisions>

<specifics>
## Specific Ideas

- The 404 triggering this phase: `POST /integrations/github/webhook/ HTTP/1.1 404 Not Found` — GitHub is already sending to the desired URL pattern
- The user explicitly wants the webhook URL under `/integrations/` not a separate `/webhooks/` prefix — consistency with plugin URL namespace
- Ingress rules should be generic: `allow /integrations/*/webhook/*` covers all current and future plugins

</specifics>
