# Phase 5.2 Plan 01: CI Pairing Models & Project CI Config UI Summary

**One-liner:** ProjectApprovedWorkflow/ProjectCIConfig models with HTMX-powered project settings UI for workflow approval management.

## What Was Done

### Task 1: Add CI pairing models and Service CI fields
- Added `status` field to CIWorkflow model (published/draft, default published)
- Created `ProjectApprovedWorkflow` model (M2M through table linking projects to approved workflows)
- Created `ProjectCIConfig` model (OneToOne per project for default workflow and approve-all toggle)
- Added Service CI fields: `ci_workflow` FK, `ci_manifest_status`, `ci_manifest_pushed_at`, `ci_manifest_pr_url`
- Added `ci_manifest_out_of_date` property on Service
- Added `get_available_workflows_for_project()` helper function
- Registered new models with auditlog
- Migration 0013 created and applied

### Task 2: Build Project CI Configuration UI in settings tab
- Created `ProjectCIConfigForm` and `ApproveWorkflowForm` in ci_workflows forms
- Added `ProjectCIConfigView`, `ProjectApproveWorkflowView`, `ProjectRemoveApprovedWorkflowView` views
- Added URL patterns for ci-config, approve-workflow, remove-workflow endpoints
- Created `_ci_config_section.html` HTMX partial with:
  - Approve All Published toggle (checkbox with custom toggle styling)
  - Default CI Workflow dropdown (filtered to available workflows)
  - Approved Workflows list with remove buttons
  - Add Workflow form (shows only unapproved published workflows)
- Included CI Configuration section in project settings tab between Members and Danger Zone
- All operations use HTMX without page reload
