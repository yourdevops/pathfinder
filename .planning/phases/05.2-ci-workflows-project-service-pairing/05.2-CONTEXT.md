# Phase 5.2: CI Workflows — Project & Service Pairing - Context

**Gathered:** 2026-01-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Connect CI Workflows (built in Phase 5.1) to Services and Projects. Projects maintain an approved list of workflows. Services are assigned one workflow from that list. Pathfinder pushes the generated GitHub Actions manifest to the service's repository via feature branch + PR. No versioning, no auto-update, no deployment capabilities.

</domain>

<decisions>
## Implementation Decisions

### Assignment flow
- CI Workflows are approved at the **project level** first — project admin explicitly adds workflows to an "Approved Workflows" list
- Option for "All published" as a shortcut (approve all published workflows)
- Only **published** workflows can be approved (draft support deferred)
- Service creation wizard includes a workflow selection step — shows only project-approved workflows
- One CI Workflow per service (1:1 relationship)
- Service can change its assigned workflow from the **Service Settings tab** — list filtered to project-approved workflows only
- No fork/upstream/runtime family filtering — project approval list is the single filter
- On workflow change, Pathfinder pushes the updated manifest to the repo via PR

### Manifest push behavior
- **Manual push** — user clicks a "Push manifest" button after assignment; no auto-push on assign
- Push uses **generic git** to a feature branch, then opens a PR via the SCM plugin if supported
- Manifest file path follows the **CI Plugin-defined convention** (e.g., `.github/workflows/` for GitHub Actions)
- Manifest file name is fixed per service so it gets **overwritten on updates** (not new files)
- Show **latest status only** on the service CI tab (synced/out-of-date), not full push history
- Audit trail captures push events for change tracking
- Skip-CI comment on commits **deferred to Phase 6** (no builds yet)

### Project CI defaults
- CI configuration lives in the **Project Settings tab** as a new "CI Configuration" section
- Project admin can set a **default CI Workflow** — pre-selected in service creation wizard but changeable
- Versioning and auto-update settings **deferred** (Phase 5.1 has no versioning)

### Claude's Discretion
- Service CI tab layout and detail level
- Manifest sync status indicator design
- Approved workflows management UI pattern (inline list, modal, etc.)
- How to surface "out-of-date" status when workflow was edited after last push

</decisions>

<specifics>
## Specific Ideas

- The service creation wizard replaces what Blueprints used to provide — CI Workflows are the new onboarding path
- Push to repo should work via generic git operations, with PR creation delegated to the SCM plugin
- The "All published" option on project approval list should be a convenience toggle, not a separate mode

</specifics>

<deferred>
## Deferred Ideas

- Draft workflow support in project approval list — future enhancement
- CI Workflow versioning (semver, patch auto-update) — future phase
- Auto-push manifest on workflow assignment — may revisit based on user feedback
- Skip-CI commit comments — Phase 6 (Builds)
- Auto-update service manifests on workflow patch changes — requires versioning first

</deferred>
