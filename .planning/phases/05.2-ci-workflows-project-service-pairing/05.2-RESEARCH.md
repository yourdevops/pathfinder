# Phase 5.2: CI Workflows -- Project & Service Pairing - Research

**Researched:** 2026-01-31
**Domain:** Django model relationships, Git operations, CI manifest push workflow
**Confidence:** HIGH

## Summary

This phase connects CI Workflows (built in Phase 5.1) to Services and Projects. The core work involves: (1) new Django models for project-approved workflows and service-workflow assignment, (2) extending project settings UI with a CI Configuration section, (3) adding a CI Workflow tab to the service detail page, (4) a background task that pushes the generated GitHub Actions manifest to the service's repository via feature branch + PR, and (5) extending the service creation wizard with a workflow selection step.

The codebase already has all required infrastructure: `CIWorkflow` model with steps and manifest generation, `GitPython` for git operations, `PyGithub` plugin with `create_branch`, `create_file`, and `create_pull_request` methods, `django-tasks` for background job processing, and `formtools.SessionWizardView` for the service creation wizard.

**Primary recommendation:** Use simple Django ForeignKey/M2M relationships to model the approval list and assignment. Leverage existing `scaffold_existing_repository` pattern (clone, branch, commit, push, PR) for manifest push. Extend the existing `SessionWizardView` with an additional wizard step for workflow selection.

## Architecture Patterns

### New Models Required

```
  - workflow (FK -> CIWorkflow)
  - approved_by (CharField, denormalized username)
  - created_at (DateTimeField)
  - unique_together: [project, workflow]

ProjectCIConfig             # OR: fields directly on Project model
  - project (OneToOneField -> Project)
  - default_workflow (FK -> CIWorkflow, null=True)
  - approve_all_published (BooleanField, default=False)

Service model additions:
  - ci_workflow (FK -> CIWorkflow, null=True, blank=True)
  - ci_manifest_status (CharField: synced/out-of-date/never-pushed)
  - ci_manifest_pushed_at (DateTimeField, null=True)
  - ci_manifest_pr_url (URLField, blank=True)
```

**Decision: Inline vs separate model for ProjectCIConfig**

Option A -- Separate `ProjectCIConfig` OneToOne model:
- Cleaner separation of concerns
- Easier to extend later (versioning settings, auto-update flags)
- No migration on existing Project table
- Consistent with how the codebase handles per-entity config

Option B -- Add fields directly to `Project`:
- Fewer queries (no join needed)
- Simpler for just 2 fields

**Recommendation: Option A (separate model).** The CONTEXT.md mentions deferred versioning/auto-update settings, which will eventually need more fields. A separate model keeps Project clean and is consistent with how `SiteConfiguration` works as a dedicated config model.

### CIWorkflow Status Field

The current `CIWorkflow` model has NO status field (no published/draft distinction). The CONTEXT.md says "Only published workflows can be approved" and "Draft support deferred." Two options:

1. Add a `status` field now (`published`/`draft`) -- future-proofs for draft support
2. Treat all workflows as published -- simpler, matches CONTEXT.md "draft support deferred"

**Recommendation:** Add the `status` field now with default `published`. All existing workflows become published. This costs almost nothing and prevents a breaking migration later. The approval list filter simply checks `status='published'`.

### Manifest Push Pattern

The existing `scaffold_existing_repository` function in `core/git_utils.py` already implements the exact pattern needed:

1. Clone repo (shallow)
2. Create feature branch
3. Write file(s)
4. Commit
5. Push feature branch
6. Create PR via SCM plugin

For manifest push, the flow is:

1. Generate manifest YAML via `generate_github_actions_manifest(workflow)`
2. Determine file path: `.github/workflows/{service.handler}.yml` (fixed name per service)
3. Clone service repo using its SCM connection (from project connections)
4. Create branch: `pathfinder/ci-manifest-{service.name}`
5. Write manifest file to `.github/workflows/` directory
6. Commit with message: `chore: update CI workflow manifest for {service.name}`
7. Push branch
8. Create PR via SCM plugin `create_pull_request()`
9. Update service: `ci_manifest_status='synced'`, `ci_manifest_pushed_at=now()`, `ci_manifest_pr_url=pr_url`

**Alternative: Use GitHub API directly (no clone).**
The GitHubPlugin already has `create_branch()`, `create_file()`, and `create_pull_request()` methods. This avoids cloning entirely:

1. `create_branch()` from main
2. `create_file()` to write manifest to branch (GitHub API handles create-or-update)
3. `create_pull_request()` to open PR

**Recommendation: Use the GitHub API approach (no clone).** It is faster, requires no disk I/O, and the methods already exist. The generic git (clone) approach is only needed if we must support non-GitHub SCMs. Since only GitHub is implemented, use the API. If a generic fallback is needed later, the clone-based approach from `scaffold_existing_repository` can be adapted.

**Important caveat:** `create_file()` in PyGithub creates a new file but fails if the file already exists. For updates, `repo.update_file()` is needed (requires the current file SHA). The push task must:
1. Try `repo.get_contents(path, ref=branch)` to check if file exists
2. If exists: `repo.update_file(path, message, content, sha, branch=branch)`
3. If not: `repo.create_file(path, message, content, branch=branch)`

This requires adding an `update_or_create_file()` method to `GitHubPlugin`. The PyGithub `Repository.update_file()` method works exactly like `create_file()` but takes an additional `sha` parameter.

### Out-of-Date Detection

A service's manifest is "out-of-date" when the assigned CIWorkflow's `updated_at` is newer than the service's `ci_manifest_pushed_at`. This is a simple comparison -- no need for a separate field tracking workflow version hash.

```python
@property
def ci_manifest_out_of_date(self):
    if not self.ci_workflow or not self.ci_manifest_pushed_at:
        return False
    return self.ci_workflow.updated_at > self.ci_manifest_pushed_at
```

### Background Task Queue

The existing TASKS config already has queues: `default`, `health_checks`, `repository_scaffolding`, `steps_scan`.

Add a new queue for manifest push operations: `ci_manifest_push`. This keeps manifest operations separate from repository scaffolding.

Alternatively, reuse `repository_scaffolding` since both involve git operations. **Recommendation: Use `repository_scaffolding` queue** to avoid config change and keep git-related tasks together.

### Anti-Patterns to Avoid

- **Don't create a separate CIManifest model** for storing generated YAML. The manifest is always re-generated from the CIWorkflow on demand. Store only the status and timestamps on the Service model.
- **Don't auto-push on workflow assignment.** The CONTEXT.md explicitly says manual push only.
- **Don't implement versioning.** Deferred per CONTEXT.md.
- **Don't track push history.** CONTEXT.md says "latest status only."

## Common Pitfalls

### Pitfall 1: PyGithub create_file vs update_file
**What goes wrong:** Calling `create_file()` on an existing file path raises a 422 error.
**Why it happens:** GitHub API requires the current file SHA for updates.
**How to avoid:** Always check if file exists first with `get_contents()`, then use `update_file()` with the SHA if it does.
**Warning signs:** 422 Unprocessable Entity errors from GitHub API on second push.

### Pitfall 2: SCM Connection Resolution for Service
**What goes wrong:** Service needs an SCM connection to push manifests, but the connection is on the ProjectConnection, not the Service.
**Why it happens:** Services don't directly reference connections. The project-connection link is indirect.
**How to avoid:** Resolve the SCM connection via `ProjectConnection.objects.filter(project=service.project, is_default=True)` or let the user select which connection to use for the push. The service's `repo_url` was originally created using a specific ProjectConnection during scaffolding -- store or resolve this consistently.
**Warning signs:** Manifest push fails because no SCM connection is found.

### Pitfall 3: Wizard Step Ordering with formtools
**What goes wrong:** Adding a step to the wizard breaks the step index calculation.
**Why it happens:** `SessionWizardView` relies on the ordered form_list. Context data and step navigation reference step names and indices.
**How to avoid:** Insert the new step at the correct position in `WIZARD_FORMS`. Update `WIZARD_TEMPLATES`, `STEP_TITLES`, and all step-specific context in `get_context_data()`. The wizard's step nav already handles arbitrary step counts via the loop.
**Warning signs:** "Step not found" errors, wrong template rendered, step progress bar out of sync.

### Pitfall 4: Circular Import with Task Functions
**What goes wrong:** Importing models at module level in tasks.py causes circular imports.
**Why it happens:** Django models depend on settings which may depend on task registration.
**How to avoid:** Use lazy imports inside the task function body, as the existing tasks already do: `from core.models import Service` inside the function.
**Warning signs:** `AppRegistryNotReady` or `ImportError` on startup.

### Pitfall 5: Approved Workflows Queryset Filtering
**What goes wrong:** The "Approve All Published" toggle doesn't work correctly alongside the explicit approval list.
**Why it happens:** Two sources of truth for "which workflows are available."
**How to avoid:** In the queryset that returns available workflows for a project, combine both: `if project_config.approve_all_published: CIWorkflow.objects.filter(status='published')` else `CIWorkflow.objects.filter(project_approvals__project=project, status='published')`. This keeps the logic in one place.
**Warning signs:** Workflows appearing or disappearing unexpectedly when toggling the setting.

### Pitfall 6: Manifest File Name Conflicts
**What goes wrong:** Two services in the same repo overwrite each other's workflow files.
**Why it happens:** Fixed file name per service, but multiple services can share a repository.
**How to avoid:** Use the service handler as the file name: `.github/workflows/{service.handler}.yml`. Since handler is `{project}-{service}` and is unique, this prevents conflicts.
**Warning signs:** Workflow file disappears after pushing manifest for a different service.

### GitHub Plugin: Update or Create File

```python
# plugins/github/plugin.py - new method
def update_or_create_file(
    self, config, repo_name, path, content, message, branch="main"
):
    """Create or update a file, handling both cases."""
    g = self._get_github_client(config)
    repo = g.get_repo(repo_name)

    try:
        # Try to get existing file to get its SHA
        existing = repo.get_contents(path, ref=branch)
        result = repo.update_file(path, message, content, existing.sha, branch=branch)
    except Exception:
        # File doesn't exist, create it
        result = repo.create_file(path, message, content, branch=branch)

    return {
        "path": result["content"].path,
        "sha": result["content"].sha,
        "commit_sha": result["commit"].sha,
    }
```

### Background Task: Push Manifest

```python
# core/tasks.py
@task(queue_name="repository_scaffolding")
def push_ci_manifest(service_id: int) -> dict:
    """Push generated CI manifest to service repository via PR."""
    from core.ci_manifest import generate_github_actions_manifest
    from core.git_utils import parse_git_url
    from core.models import ProjectConnection, Service

    service = Service.objects.select_related("project", "ci_workflow").get(id=service_id)
    workflow = service.ci_workflow
    manifest_yaml = generate_github_actions_manifest(workflow)

    # Resolve SCM connection
    project_conn = ProjectConnection.objects.filter(
        project=service.project, is_default=True
    ).select_related("connection").first()
    connection = project_conn.connection
    plugin = connection.get_plugin()
    config = connection.get_config()

    # Parse repo URL
    parsed = parse_git_url(service.repo_url)
    repo_name = f"{parsed['owner']}/{parsed['repo']}"

    # Branch and file path
    branch_name = f"pathfinder/ci-manifest-{service.name}"
    file_path = f".github/workflows/{service.handler}.yml"

    # Create branch, write file, open PR
    plugin.create_branch(config, repo_name, branch_name, service.repo_branch)
    plugin.update_or_create_file(
        config, repo_name, file_path, manifest_yaml,
        f"chore: update CI workflow for {service.name}", branch_name
    )
    pr_result = plugin.create_pull_request(
        config, service.repo_url,
        title=f"Update CI workflow for {service.name}",
        body=f"Updated CI manifest from workflow: {workflow.name}",
        head=branch_name, base=service.repo_branch,
    )

    # Update service status
    service.ci_manifest_status = "synced"
    service.ci_manifest_pushed_at = timezone.now()
    service.ci_manifest_pr_url = pr_result.get("html_url", "")
    service.save(update_fields=[
        "ci_manifest_status", "ci_manifest_pushed_at", "ci_manifest_pr_url"
    ])
    return {"status": "success", "pr_url": pr_result.get("html_url")}
```

### Wizard Step: Workflow Selection Form

```python
# core/forms/services.py
class WorkflowSelectionForm(forms.Form):
    """Step: Select CI Workflow for the service."""
    ci_workflow = forms.ModelChoiceField(
        queryset=CIWorkflow.objects.none(),
        required=False,
        empty_label="No CI Workflow",
        widget=forms.Select(attrs={
            "class": "w-full px-3 py-2 bg-dark-bg border border-dark-border rounded-lg text-dark-text",
        }),
        label="CI Workflow",
        help_text="Select a CI Workflow approved for this project",
    )

    def __init__(self, *args, project=None, **kwargs):
        super().__init__(*args, **kwargs)
        if project:
            # Get approved workflows for this project
            ci_config = getattr(project, 'ci_config', None)
            if ci_config and ci_config.approve_all_published:
                self.fields["ci_workflow"].queryset = CIWorkflow.objects.filter(status="published")
            else:
                approved_ids = project.approved_workflows.values_list("workflow_id", flat=True)
                self.fields["ci_workflow"].queryset = CIWorkflow.objects.filter(
                    id__in=approved_ids, status="published"
                )
            # Pre-select default
            if ci_config and ci_config.default_workflow:
                self.fields["ci_workflow"].initial = ci_config.default_workflow
```

```python
def get_available_workflows_for_project(project):
```

## Open Questions

1. **Branch reuse on re-push**
   - What we know: Each push creates a branch `pathfinder/ci-manifest-{service.name}`. On re-push, the branch may already exist.
   - What's unclear: Should we force-update the existing branch or create a new uniquely-named branch?
   - Recommendation: Delete the old branch first (if PR is merged/closed), or force-push to the existing branch. If a PR is still open, update the existing branch in-place. The implementation should check for an existing open PR first.

2. **SCM connection for manifest push**
   - What we know: Services are created with a specific SCM connection. The push needs the same connection.
   - What's unclear: Should the Service store which connection was used, or always use the project's default?
   - Recommendation: Use the project's default SCM connection. If the service's repo is on a non-default connection, this could fail. Consider storing the connection ID on the Service model, or resolving it from the repo URL by matching against project connections.

3. **CIWorkflow `status` field initial value**
   - What we know: All existing workflows have no status. CONTEXT.md defers draft support.
   - What's unclear: Should the migration set existing workflows to `published`?
   - Recommendation: Yes. Default `status='published'` and migrate existing records to `published`. This maintains backward compatibility.
