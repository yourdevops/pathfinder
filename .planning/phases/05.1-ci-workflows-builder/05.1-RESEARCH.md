# Phase 5.1: CI Workflows Builder - Research

**Researched:** 2026-01-29
**Domain:** CI Workflows composer replacing Blueprints, GitHub Actions manifest generation, steps catalog scanning
**Confidence:** HIGH

## Summary

This phase replaces the existing Blueprints system with a new CI Workflows concept. The codebase already has mature patterns for Git repository scanning (GitPython), YAML parsing (PyYAML), background tasks (django-tasks), HTMX-based UI interactions, and semver handling. The new CI Workflows system reuses these foundations but introduces new domain models (StepsRepository, CIStep, RuntimeFamily, CIWorkflow, CIWorkflowStep) and a workflow composer UI.

The key technical challenge is the workflow composer -- a drag-and-drop/add-reorder-remove UI for composing CI steps filtered by runtime compatibility. The existing SessionWizardView pattern from django-formtools can be adapted for the multi-step workflow creation flow. Version constraint matching for runtime compatibility uses the already-installed `semver` library with a normalization layer for 2-part version strings (e.g., "3.11" becomes "3.11.0").

**Primary recommendation:** Build incrementally on existing patterns -- reuse `git_utils.py` for repo scanning, `tasks.py` for async scanning, HTMX for dynamic UI, and `semver` for version constraints. Replace Blueprint/BlueprintVersion models with new CI-specific models. Use PyYAML for both action.yml and runtimes.yml parsing.

### Pattern 3: Runtime Compatibility Checking
**What:** Check if a CI step is compatible with a selected runtime
**When to use:** Filtering steps in the workflow composer

```python
import semver

def normalize_version(version_str: str) -> str:
    """Normalize 2-part version (3.11) to 3-part semver (3.11.0)."""
    parts = version_str.split('.')
    while len(parts) < 3:
        parts.append('0')
    return '.'.join(parts[:3])

def is_step_compatible(step: CIStep, runtime_family: str, runtime_version: str) -> bool:
    """Check if a step is compatible with the given runtime."""
    constraints = step.runtime_constraints  # e.g., {"python": ">=3.10"}
    if not constraints:
        return True  # No constraints = compatible with all

    constraint = constraints.get(runtime_family)
    if constraint is None:
        return False  # Step doesn't support this runtime family at all

    if constraint == '*':
        return True  # Wildcard = all versions

    # Normalize both versions to semver format
    normalized_version = normalize_version(runtime_version)
    # Normalize constraint version part
    # Parse constraint like ">=3.10" -> operator ">=", version "3.10" -> "3.10.0"
    import re
    match = re.match(r'^([<>=!]+)(.+)$', constraint)
    if not match:
        return False
    op, ver = match.groups()
    normalized_constraint = f"{op}{normalize_version(ver)}"

    version = semver.Version.parse(normalized_version)
    return version.match(normalized_constraint)
```

import yaml

def generate_github_actions_manifest(workflow: CIWorkflow) -> str:
    """Generate GitHub Actions workflow YAML for a CI Workflow."""
    manifest = {
        'name': f'CI - {workflow.name}',
        'on': {
            'push': {
                'branches': ['main'],
            },
        },
        'jobs': {
            'build': {
                'runs-on': 'ubuntu-latest',
                'steps': [],
            },
        },
    }

    steps_list = manifest['jobs']['build']['steps']

    # Auto-inject: checkout
    steps_list.append({
        'name': 'Checkout',
        'uses': 'actions/checkout@v4',
    })

    # Auto-inject: ssp-notify-start (from docs)
    steps_list.append({
        'name': 'Notify SSP - Start',
        'uses': './ci-steps/ssp-notify-start',  # or full action reference
    })

    # User-composed steps
    for ws in workflow.workflow_steps.order_by('order'):
        step_entry = {
            'name': ws.step.name,
            'uses': f'./{ws.step.repository.name}/ci-steps/{ws.step.directory_name}',
        }
        if ws.input_config:
            step_entry['with'] = ws.input_config
        steps_list.append(step_entry)

    # Auto-inject: ssp-notify-complete
    steps_list.append({
        'name': 'Notify SSP - Complete',
        'uses': './ci-steps/ssp-notify-complete',
    })

    return yaml.dump(manifest, default_flow_style=False, sort_keys=False)

## Common Pitfalls

### Pitfall 1: Semver Normalization for Runtime Versions
**What goes wrong:** Runtime versions in runtimes.yml are "3.11", "3.12" (2-part), but `semver.Version.parse()` requires 3-part "3.11.0". Similarly, constraints like ">=3.10" fail because "3.10" is not valid semver.
**Why it happens:** runtimes.yml follows convention of short version numbers, semver library is strict.
**How to avoid:** Create a `normalize_version()` helper that pads to 3 parts. Apply to both runtime versions and constraint version strings.
**Warning signs:** `ValueError: X is not valid SemVer string` exceptions.

### Pitfall 2: Wildcard Constraint Handling
**What goes wrong:** The `semver.match()` function does not support `"*"` as a wildcard. Passing `"*"` raises an error.
**Why it happens:** semver only supports comparison operators (`>=`, `<=`, etc.).
**How to avoid:** Check for `"*"` before calling `semver.match()` and return True immediately. Document this in the compatibility check function.
**Warning signs:** Errors when processing steps with `runtimes: { python: "*" }`.

### Pitfall 3: Blueprint Foreign Key on Service Model
**What goes wrong:** The Service model has `blueprint = ForeignKey(Blueprint, on_delete=PROTECT)` and `blueprint_version = ForeignKey(BlueprintVersion, on_delete=PROTECT)`. Deleting Blueprint/BlueprintVersion models breaks existing Service records.
**Why it happens:** PROTECT on_delete prevents deletion, but dropping the model entirely causes migration issues.
**How to avoid:** The migration must either: (a) remove the FK fields from Service first, then drop Blueprint/BlueprintVersion tables, or (b) make the FK fields nullable first, set to null, then drop. Since "no backwards compatibility" applies, option (a) is cleaner. In a later phase (5.2), Service will get a new `ci_workflow` ForeignKey.
**Warning signs:** Migration errors about foreign key constraints.

### Pitfall 4: Steps Repository Directory Structure Assumption
**What goes wrong:** Assuming all steps are directly under `ci-steps/` when some repos might have different structures.
**Why it happens:** The docs specify `ci-steps/` but real repos may vary.
**How to avoid:** Scan for `action.yml` files specifically. Walk `ci-steps/` directory, look for subdirectories containing `action.yml`. Skip directories without it. Log warnings for unexpected structures.
**Warning signs:** Empty steps catalog after scanning a valid repo.

### Pitfall 5: YAML Output Key Ordering
**What goes wrong:** `yaml.dump()` with default settings alphabetically sorts keys, producing `jobs:`, `name:`, `on:` instead of the conventional `name:`, `on:`, `jobs:` order.
**Why it happens:** PyYAML default `sort_keys=True`.
**How to avoid:** Always use `yaml.dump(data, sort_keys=False)` for manifest generation. Use `OrderedDict` or plain `dict` (Python 3.7+ preserves insertion order).
**Warning signs:** Generated YAML looks correct but unconventional ordering confuses users.

### Pitfall 6: Removing Blueprints URL Namespace While Services Reference It
**What goes wrong:** The ServiceCreateWizard and ServiceListView reference Blueprint models. If Blueprint URLs and models are removed without updating Services views, the app crashes.
**Why it happens:** Services have FK to Blueprint. Views import Blueprint model.
**How to avoid:** Phase 5.1 must update the Service model (make blueprint FK nullable or remove it), update service views/templates to remove blueprint references, and update the service creation wizard (which currently requires selecting a blueprint). The service creation wizard should be temporarily simplified or disabled until Phase 5.2 reconnects services to CI workflows.
**Warning signs:** ImportError or TemplateDoesNotExist on any page after removing blueprints.

### Scanning runtimes.yml
```python
# In git_utils.py or a new ci_utils.py
import os
import yaml

def parse_runtimes_yml(repo_path: str) -> dict:
    """Parse runtimes.yml from a steps repository.

    Returns:
        Dict mapping family name to list of version strings.
        e.g., {"python": ["3.11", "3.12", "3.13"], "node": ["18", "20", "22"]}
    """
    runtimes_path = os.path.join(repo_path, 'runtimes.yml')
    if not os.path.exists(runtimes_path):
        return {}

    with open(runtimes_path, 'r') as f:
        data = yaml.safe_load(f) or {}

    result = {}
    for family, config in data.items():
        if isinstance(config, dict) and 'versions' in config:
            result[family] = [str(v) for v in config['versions']]
        elif isinstance(config, list):
            result[family] = [str(v) for v in config]

    return result
```

### Scanning action.yml Files
```python
def scan_ci_steps(repo_path: str) -> list:
    """Scan ci-steps/ directory for step definitions.

    Returns:
        List of dicts with step metadata from action.yml files.
    """
    steps_dir = os.path.join(repo_path, 'ci-steps')
    if not os.path.isdir(steps_dir):
        return []

    steps = []
    for entry in sorted(os.listdir(steps_dir)):
        step_dir = os.path.join(steps_dir, entry)
        if not os.path.isdir(step_dir):
            continue

        action_file = os.path.join(step_dir, 'action.yml')
        if not os.path.exists(action_file):
            action_file = os.path.join(step_dir, 'action.yaml')
            if not os.path.exists(action_file):
                continue

        with open(action_file, 'r') as f:
            metadata = yaml.safe_load(f) or {}

        pathfinder = metadata.get('x-pathfinder', {})

        steps.append({
            'directory_name': entry,
            'name': metadata.get('name', entry),
            'description': metadata.get('description', ''),
            'inputs': metadata.get('inputs', {}),
            'phase': pathfinder.get('phase', ''),
            'runtime_constraints': pathfinder.get('runtimes', {}),
            'tags': pathfinder.get('tags', []),
            'produces': pathfinder.get('produces'),
            'raw_metadata': metadata,
        })

    return steps
```

### HTMX Compatible Steps Filter
```python
# In views/ci_workflows.py
class CompatibleStepsView(LoginRequiredMixin, View):
    """HTMX endpoint: return steps compatible with selected runtime."""

    def get(self, request):
        family = request.GET.get('runtime_family', '')
        version = request.GET.get('runtime_version', '')

        if not family or not version:
            return HttpResponse('<p class="text-dark-muted">Select a runtime first.</p>')

        all_steps = CIStep.objects.all().order_by('phase', 'name')

        compatible = []
        incompatible = []
        for step in all_steps:
            if is_step_compatible(step, family, version):
                compatible.append(step)
            else:
                incompatible.append(step)

        html = render_to_string('core/ci_workflows/_compatible_steps.html', {
            'compatible_steps': compatible,
            'incompatible_steps': incompatible,
            'runtime_family': family,
            'runtime_version': version,
        }, request=request)
        return HttpResponse(html)
```

### Workflow Composer Step Reordering (Alpine.js)
```html
<!-- Alpine.js for step ordering in workflow composer -->
<div x-data="{
    steps: [],
    addStep(step) {
        this.steps.push({...step, order: this.steps.length});
        this.updateOrders();
    },
    removeStep(index) {
        this.steps.splice(index, 1);
        this.updateOrders();
    },
    moveUp(index) {
        if (index > 0) {
            [this.steps[index-1], this.steps[index]] = [this.steps[index], this.steps[index-1]];
            this.updateOrders();
        }
    },
    moveDown(index) {
        if (index < this.steps.length - 1) {
            [this.steps[index], this.steps[index+1]] = [this.steps[index+1], this.steps[index]];
            this.updateOrders();
        }
    },
    updateOrders() {
        this.steps.forEach((s, i) => s.order = i);
    }
}">
    <!-- Step list with up/down/remove buttons -->
    <template x-for="(step, index) in steps" :key="step.id + '-' + index">
        <div class="flex items-center gap-2 p-3 bg-dark-surface border border-dark-border rounded-lg mb-2">
            <span class="text-dark-muted text-sm" x-text="index + 1"></span>
            <span class="flex-1 text-dark-text" x-text="step.name"></span>
            <button @click="moveUp(index)" :disabled="index === 0">Up</button>
            <button @click="moveDown(index)" :disabled="index === steps.length - 1">Down</button>
            <button @click="removeStep(index)">Remove</button>
        </div>
    </template>
    <!-- Hidden input for form submission -->
    <input type="hidden" name="steps_json" :value="JSON.stringify(steps)">
</div>
```

## Open Questions

1. **Steps repository structure: flat ci-steps/ or nested?**
   - What we know: docs/ci-workflows.md shows `ci-steps/setup-python/action.yml` (one level deep)
   - What's unclear: Should scanning support deeper nesting? Multiple action.yml variants (action.yml vs pipe.yml) per step?
   - Recommendation: MVP supports only `ci-steps/<step-name>/action.yml` (GitHub Actions only). Other CI platforms deferred.

2. **How should the step `uses` reference look in generated manifest?**
   - What we know: GitHub Actions supports `uses: owner/repo/path@ref` for actions in repos
   - What's unclear: Whether steps should reference the steps repo directly or be copied into the service repo
   - Recommendation: Use `uses: <steps-repo-owner>/<steps-repo-name>/ci-steps/<step-name>@<commit-sha>` format in generated YAML. This references the steps repo directly.

3. **Per-step input configuration UI**
   - What we know: Each step has an `inputs` section in action.yml with type, default, required, description
   - What's unclear: How complex should the input configuration form be? Dynamic form per step?
   - Recommendation: Render a simple form per step based on inputs schema. Use HTMX to load step config form when a step is added to the workflow. Store configured inputs in CIWorkflowStep.input_config JSON field.

4. **What happens to existing Service records that reference Blueprints?**
   - What we know: Service has PROTECT FK to Blueprint. "No backwards compatibility" per CLAUDE.md.
   - What's unclear: Whether to delete existing services or just null out the FK
   - Recommendation: Make blueprint FK nullable, set existing values to NULL via migration. This preserves service records. In Phase 5.2, add ci_workflow FK to Service.
