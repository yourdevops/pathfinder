# Phase 6.8: Manifest and Plugin Interface - Context

**Gathered:** 2026-02-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Artifact discovery uses CI plugin API (not webhook payloads), CI variables are injected into manifests, a step validation API exists, dead manifest_path code is removed, and workflow runtimes are derived from steps with version constraints at the workflow level. Concrete version selection at the service level is deferred to a future phase.

Gaps addressed: GAP-13, GAP-17, GAP-18, GAP-19, GAP-20 from docs/ci-workflows/REMEDIATION.md

</domain>

<decisions>
## Implementation Decisions

### Workflow composer redesign (GAP-20)
- Engine is still selected on the wizard's first page (name, description, engine) — acts as a hard filter for compatible steps
- Runtime family and version are NO LONGER selected upfront — removed from WorkflowCreateForm
- Composer shows all active steps for the selected engine in the right-side list (current layout preserved)
- Optional runtime filter available in the step list for narrowing by runtime ecosystem
- Live constraint summary panel displayed above the composed steps — shows derived runtimes and version ranges, updates as steps are added/removed
- Constraints are purely derived from the intersection of included steps' runtime_constraints — no author narrowing in the composer
- Adding a step that conflicts with existing runtime constraints is blocked with an explanation (which existing step conflicts and why)
- Workflow stores derived `runtime_constraints` JSONField (computed from steps)

### Service-level version pinning (GAP-20 partial)
- **Deferred** to a future phase
- Phase 6.8 handles only the workflow side: derive constraints from steps, remove upfront runtime selection
- Concrete version pinning (e.g., "Python 3.12") at the Service level is not implemented in this phase

### CI variable provisioning (GAP-17)
- Variables provisioned on service creation, regardless of whether a workflow is assigned
- Only PTF_PROJECT and PTF_SERVICE provisioned — PTF_ENVIRONMENT skipped (environment context is a deployment-time concern)
- Provisioning via CI Plugin interface: `provision_ci_variables()` on CICapableMixin
- GitHubPlugin implements using GitHub repository variables API (`vars.PTF_PROJECT`, `vars.PTF_SERVICE`)
- Manifest includes static engine-native references (e.g., `${{ vars.PTF_PROJECT }}` for GitHub Actions) — identical across all services, does not affect manifest hash
- Provisioned variables displayed as read-only list in the Service CI tab
- Provisioning is non-blocking: service creation succeeds even if variable provisioning fails
- Provisioning status shown in CI tab (e.g., "Variables: provisioned" or "Variables: pending") with retry option

### Artifact discovery (GAP-13)
- `resolve_artifact_ref()` added to CICapableMixin plugin interface
- GitHubPlugin implements by querying GitHub API for actual container image reference and digest
- Called in `poll_build_details` instead of using webhook-extracted artifact_ref
- Stores resolved image ref (e.g., `ghcr.io/owner/repo:sha-abc123`) on Build.artifact_ref

### Step validation API (GAP-18)
- POST endpoint at `api/ci-workflows/steps/validate`
- Token-based authentication
- Reuses parsing logic from `discover_steps` and plugin `parse_step_file`
- Returns structured result: slug, name, phase, inputs, conflicts, warnings

### manifest_path cleanup (GAP-19)
- Remove `manifest_path` from CICapableMixin and GitHubPlugin
- Verify zero references before removing

### Claude's Discretion
- Step validation API authentication mechanism (token type, header format)
- Exact layout of the live constraint summary panel
- Error state handling for artifact resolution failures
- Retry mechanism for failed variable provisioning

</decisions>

<specifics>
## Specific Ideas

- Step list layout stays as-is (right-side panel in composer) — just remove the upfront runtime filter and add optional runtime filtering
- CI variable provisioning follows the design in docs/ci-workflows/steps-catalog.md "CI Integration — Variables" section
- Manifest references for variables are engine-native constants, not per-service values — hash verification integrity maintained

</specifics>

<deferred>
## Deferred Ideas

- Service-level concrete version pinning (e.g., "Python 3.12" per service) — future phase after 6.8
- PTF_ENVIRONMENT variable provisioning — deferred until deployment phase (environment is a deployment-time concern)
- Runtime config file management in service repos (`.python-version`, `.tool-versions`, `.nvmrc`) — related to service version pinning, also deferred

</deferred>
