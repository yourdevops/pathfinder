# Phase 6.8: Manifest and Plugin Interface - Research

**Researched:** 2026-02-17
**Domain:** CI plugin interface, workflow runtime derivation, CI variable provisioning, artifact resolution, step validation API
**Confidence:** HIGH

## Summary

Phase 6.8 addresses five remediation gaps (GAP-13, GAP-17, GAP-18, GAP-19, GAP-20) that evolve the CI plugin interface and workflow composition model. The changes span the plugin abstraction layer (`CICapableMixin`), the workflow creation/composer UX, manifest generation, and a new REST API endpoint.

The most impactful change is GAP-20: redesigning workflow creation to derive runtime constraints from steps rather than requiring upfront selection. This touches the `WorkflowCreateForm`, `WorkflowComposerView`, the `CIWorkflow` model, and composer templates. The remaining gaps are lower complexity: adding two new methods to `CICapableMixin` (GAP-13, GAP-17), removing dead code (GAP-19), and creating a standalone API endpoint (GAP-18).

All schema changes apply to empty tables (all CI data was previously deleted), so no data migrations are needed -- just `makemigrations`. The project uses Django 6.0.1, PyGithub 2.5+, and the existing plugin framework is mature enough to accept these additions cleanly.

**Primary recommendation:** Start with GAP-19 (dead code removal) and GAP-18 (step validation API) as quick wins, then GAP-13 (artifact discovery) and GAP-17 (CI variables), and finish with GAP-20 (composer redesign) as it has the broadest surface area.

### Pattern 2: Constraint Intersection (GAP-20)

**What:** Derive workflow runtime constraints from the intersection of all included steps' `runtime_constraints`.

**Logic:**
1. Collect all `runtime_constraints` from selected steps
2. For each runtime family appearing in ANY step, compute the intersection of all constraint ranges
3. A step with no `runtime_constraints` (or `"*": "*"`) is runtime-agnostic -- it does not restrict
4. A step that declares constraints for a runtime family it doesn't support creates a conflict

**Intersection algorithm for semver constraints:**
```python

    # If lower > upper, constraints conflict (return None)
    # Otherwise, return the tightest range as a string
```

**Storage:** `CIWorkflow.runtime_constraints` JSONField stores the result:
```json

    "python": ">=3.11",
    "node": ">=18"
}
```

### Pattern 3: Token-Based API Authentication (GAP-18)

**What:** Simple token authentication for the step validation API endpoint.

**Recommended approach:** Use Django's built-in session-free authentication via a custom token model and middleware-free token check in the view itself. This aligns with the existing codebase pattern (no DRF, no external auth library).

```python
# New model: ApiToken
class ApiToken(models.Model):
    key = models.CharField(max_length=64, unique=True, db_index=True)
    name = models.CharField(max_length=100)  # Human-readable label
    created_by = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE)
    created_at = models.DateTimeField(auto_now_add=True)
    last_used_at = models.DateTimeField(null=True, blank=True)
    is_active = models.BooleanField(default=True)

# View authentication:
@csrf_exempt
def step_validate_api(request):
    if request.method != "POST":
        return JsonResponse({"error": "Method not allowed"}, status=405)

    auth_header = request.headers.get("Authorization", "")
    if not auth_header.startswith("Token "):
        return JsonResponse({"error": "Authentication required"}, status=401)

    token_key = auth_header[6:]

        token.last_used_at = timezone.now()

        return JsonResponse({"error": "Invalid token"}, status=401)
```

### Anti-Patterns to Avoid

- **Baking per-service values into manifests:** The manifest MUST remain byte-identical for all services using the same workflow version. CI variables are engine-level, referenced via static expressions (e.g., `${{ vars.PTF_PROJECT }}`).
- **Adding runtime constraint UI to WorkflowCreateForm:** The decision explicitly removes upfront runtime selection. Don't add a replacement "constraint narrowing" field -- constraints are derived only.
- **Using DRF for a single API endpoint:** The project has no Django REST Framework dependency. Adding it for one endpoint is overkill. Use plain Django views with `JsonResponse`.
- **Blocking service creation on variable provisioning failure:** The decision explicitly states provisioning is non-blocking.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Semver constraint parsing | Custom parser | `semver` library (already installed) | Edge cases in pre-release handling, comparison operators |
| Token generation | Custom random strings | `secrets.token_hex(32)` | Cryptographically secure, standard library |
| YAML parsing for validation | Custom parser | `yaml.safe_load` (already used) | Battle-tested, handles edge cases |
| GitHub API calls | Raw HTTP requests | PyGithub (already used) | Token management, rate limiting, pagination handled |

**Key insight:** The existing `semver` library and `ci_manifest.py` already handle single-constraint compatibility checks. The new constraint intersection logic should build on `semver.Version.parse()` and the existing `normalize_version()` helper rather than re-implementing version comparison.

## Common Pitfalls

### Pitfall 1: manifest_path Removal Breaking Scaffolding
**What goes wrong:** Removing `manifest_path` from the plugin interface without updating scaffolding functions causes `AttributeError` when creating new services with CI workflows.
**Why it happens:** `core/git_utils.py` has two live call sites that were not identified in the original REMEDIATION.md gap description.
**How to avoid:** Update `scaffold_new_repository()` and `scaffold_existing_repository()` to use `manifest_id(workflow)` BEFORE removing the method. Run full test suite after removal.
**Warning signs:** Any `AttributeError: 'GitHubPlugin' object has no attribute 'manifest_path'` in logs.

### Pitfall 2: Constraint Intersection Producing Empty Sets
**What goes wrong:** Adding a step whose constraints don't intersect with existing steps should be blocked, but if the intersection logic has bugs, it silently produces an empty or invalid constraint set.
**Why it happens:** Semver constraint intersection is non-trivial (e.g., `>=3.10` and `<3.0` have no intersection). The wildcard (`"*"`) handling adds complexity.
**How to avoid:** Implement constraint intersection as a pure function with comprehensive unit tests. Test cases: overlapping ranges, disjoint ranges, wildcard + specific, multiple runtimes, single step.
**Warning signs:** Workflows with `runtime_constraints: {}` when they should have constraints.

### Pitfall 3: CI Variable Provisioning Race Condition
**What goes wrong:** Service creation triggers variable provisioning as a background task. If the user immediately assigns a workflow and pushes a manifest, the CI run may start before variables are provisioned.
**Why it happens:** Variable provisioning is async and non-blocking by design.
**How to avoid:** This is acceptable per the decision (non-blocking). The CI tab displays provisioning status. Steps should handle missing variables gracefully. Document this in the UI: "Variables may take a moment to provision."
**Warning signs:** CI builds failing with "variable not found" errors immediately after service creation.

### Pitfall 4: PyGithub Variables API Not Having Upsert
**What goes wrong:** The GitHub API has separate POST (create) and PATCH (update) endpoints for repository variables. Calling POST on an existing variable returns 409 Conflict.
**Why it happens:** GitHub's variables API is not idempotent by default.
**How to avoid:** Implement try/create-catch-409/update pattern, or list existing variables first and choose create vs. update.
**Warning signs:** `GithubException` with status 409 or 422 during provisioning.

### Pitfall 5: Composer UI State Desync with Constraint Derivation
**What goes wrong:** The constraint summary panel shows stale data because the Alpine.js component doesn't recalculate after step additions/removals.
**Why it happens:** Constraint calculation requires step metadata (runtime_constraints) that must be available client-side.
**How to avoid:** Pass step runtime_constraints as part of the `step_inputs_map` data structure (already passed to the composer template). Calculate constraints in JavaScript/Alpine.data() on every step add/remove.
**Warning signs:** Summary panel not updating when steps are dragged in/out.

### Pitfall 6: Artifact Resolution Returning Wrong Type
**What goes wrong:** `resolve_artifact_ref()` returns a GitHub API URL (like the current `extract_artifact_ref`) instead of an actual container image reference.
**Why it happens:** GitHub's workflow run `artifacts_url` endpoint returns build artifacts (zip files), not container images. Container images pushed to GHCR are discovered differently.
**How to avoid:** For GitHub Actions, the artifact ref for container images should come from the GitHub Packages/Container Registry API, not the workflow artifacts endpoint. Query `ghcr.io` packages for the repository.
**Warning signs:** `artifact_ref` values starting with `https://api.github.com/` instead of `ghcr.io/`.

### Example 2: Step Validation View Pattern

```python
@csrf_exempt
def step_validate_api(request):
    """POST /api/ci-workflows/steps/validate"""
    if request.method != "POST":
        return JsonResponse({"error": "Method not allowed"}, status=405)

    # Token auth
    auth_header = request.headers.get("Authorization", "")
    if not auth_header.startswith("Token "):
        return JsonResponse({"error": "Authentication required"}, status=401)

    token_key = auth_header[6:].strip()
    # ... validate token ...

    # Parse request body
    try:
        body = json.loads(request.body)
    except json.JSONDecodeError:
        return JsonResponse({"error": "Invalid JSON"}, status=400)

    engine = body.get("ci_engine", "")
    content_str = body.get("content", "")

    # Parse YAML content
    try:
        file_content = yaml.safe_load(content_str)
    except yaml.YAMLError as e:
        return JsonResponse({"valid": False, "errors": [str(e)]}, status=200)

    # Get CI plugin
    ci_plugin = get_ci_plugin_for_engine(engine)
    if not ci_plugin:
        return JsonResponse({"valid": False, "errors": [f"Unknown engine: {engine}"]}, status=200)

    # Parse and validate using existing plugin logic
    step_info = ci_plugin.parse_step_file(file_content)
    slug = ci_plugin.derive_step_slug(file_content, "")

    # Check conflicts
    conflicts = []
    if slug:
        existing = CIStep.objects.filter(engine=engine, slug=slug).first()
        if existing:
            conflicts.append(f"Slug '{slug}' already exists (from repository '{existing.repository.name}')")

    warnings = []
    if not step_info.get("phase"):
        warnings.append("No x-pathfinder.phase specified")
    if not slug:
        warnings.append("Could not derive slug from step definition")

    return JsonResponse({
        "valid": len(conflicts) == 0 and bool(slug),
        "step": {
            "name": step_info.get("name", ""),
            "slug": slug,
            "runtimes": step_info.get("runtime_constraints", {}),
            "phase": step_info.get("phase", ""),
            "produces": step_info.get("produces"),
        },
        "conflicts": conflicts,
        "warnings": warnings,
    })
```

### Example 3: Constraint Intersection Logic

```python
def compute_runtime_constraints(steps: list) -> dict:
    """Compute runtime constraints from a list of CIStep instances.

    Returns dict mapping runtime family -> constraint string.
    Returns None for a runtime family if constraints conflict.
    """
    from collections import defaultdict

    # Collect all constraints per runtime family
    family_constraints = defaultdict(list)

    for step in steps:
        if not step.runtime_constraints:
            continue
        for family, constraint in step.runtime_constraints.items():
            if family == "*":
                continue  # Runtime-agnostic, skip
            family_constraints[family].append((constraint, step))

    result = {}
    conflicts = []

    for family, constraint_pairs in family_constraints.items():
        constraints = [c for c, _ in constraint_pairs]

        # All wildcards means no constraint
        non_wildcard = [c for c in constraints if c != "*"]
        if not non_wildcard:
            result[family] = "*"
            continue

        # Find tightest lower and upper bounds
        intersected = intersect_semver_constraints(non_wildcard)
        if intersected is None:
            # Conflict -- record which steps conflict
            step_names = [s.name for _, s in constraint_pairs]
            conflicts.append({
                "runtime": family,
                "steps": step_names,
                "constraints": constraints,
            })
        else:
            result[family] = intersected

    return {"constraints": result, "conflicts": conflicts}
```

### GAP-17: Service Creation and Variable Provisioning

Service creation happens via the service wizard (multi-step form). The final step triggers `scaffold_repository` task. Variable provisioning should be triggered:
1. After service creation (when repo_url is known)
2. Non-blocking (background task or inline with error handling)

The `Service` model needs two new fields:
- `ci_variables_status`: CharField with choices like `pending`, `provisioned`, `failed`
- `ci_variables_error`: TextField for error messages

Provisioning should be called in `scaffold_repository` task (at `core/tasks.py:116`) after the repo is created/confirmed, since we need the repo to exist before setting variables.

### GAP-13: Artifact Resolution Strategy

For GitHub Actions with container images:
- GitHub Packages/Container Registry (GHCR) is the standard registry for GitHub Actions
- Container images are pushed to `ghcr.io/{owner}/{repo}:{tag}`
- The tag is typically `sha-{short_sha}` or a version tag
- PyGithub does not have a direct GHCR API wrapper -- need to query the GitHub Packages REST API

**Recommended approach:** Query the GitHub Packages API to find container packages for the repository, then match by the build's commit SHA to find the specific image tag.

```
GET /orgs/{org}/packages/container/{package}/versions
# or
GET /users/{username}/packages/container/{package}/versions
```

**Fallback:** If no container image is found (e.g., workflow produces a different artifact type), return empty string. The `produces` field on the package step indicates the expected artifact type.

### GAP-18: URL Route Integration

The step validation API endpoint needs to be registered. Current URL structure:
- Core URLs: `core/urls.py` has CI workflow routes under `ci-workflows/` namespace
- Webhooks: `pathfinder/urls.py` has webhook routes under `webhooks/`
- No `api/` prefix currently exists

**Recommended:** Add the API route in the main `pathfinder/urls.py`:
```python
path("api/ci-workflows/steps/validate", step_validate_api, name="step_validate_api"),
```

This matches the design doc specification and keeps API routes separate from UI routes.

## Open Questions

1. **PyGithub container package discovery**
   - What we know: GitHub Packages API exists for querying container images; PyGithub has some packages support
   - What's unclear: Whether PyGithub's packages API is mature enough or if raw REST calls are needed
   - Recommendation: Check PyGithub's `get_packages()` method; fall back to `requests` with token auth if needed. For Phase 6.8, implement a basic version that queries packages and returns empty string if no container image found. Full artifact resolution can be refined later.

2. **Constraint display format in summary panel**
   - What we know: Constraints are stored as `{"python": ">=3.11", "node": ">=18"}`
   - What's unclear: Exact UI layout for the summary panel (badges vs. table vs. list)
   - Recommendation: Use a compact badge-style display: `Python >=3.11 | Node >=18` with color-coded status (green for satisfied, red for conflicts). Register an `Alpine.data()` component for constraint calculation.

3. **API token management UI scope**
   - What we know: `ApiTokensView` exists as a placeholder; token model needs creation
   - What's unclear: How much token management UI is needed (create, list, revoke, or just create?)
   - Recommendation: Implement create, list, and revoke. Tokens are displayed once on creation (like GitHub PATs). Keep the UI minimal -- it's a settings page.
