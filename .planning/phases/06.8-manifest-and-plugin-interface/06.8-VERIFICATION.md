### Human Verification Required

#### 1. Composer constraint conflict blocking

**Test:** In the workflow composer, compose a workflow with a step that declares `python: ">=3.11,<3.12"` in its runtime constraints, then attempt to add a second step that declares `python: ">=3.13"`.
**Expected:** The "Add Step" action is intercepted by `canAddStep()`. An inline error appears explaining the incompatibility. The second step is not added to the composed list.
**Why human:** The Alpine.js `canAddStep()` client-side logic and resulting DOM mutation cannot be verified through static template or Python source analysis.

#### 2. CI variable provisioning end-to-end

**Test:** Create a service with a real GitHub connection and trigger repository scaffolding. Navigate to the service's CI tab.
**Expected:** The CI Variables section shows a green "Provisioned" badge with `PTF_PROJECT` and `PTF_SERVICE` displayed with their values. The GitHub repository should have these variables set under Settings > Secrets and Variables > Actions > Variables.
**Why human:** Requires a live GitHub connection, real API credentials, and a real repository.

#### 3. Step validation API token flow

**Test:** Create an API token via `/settings/api-tokens/`, copy the token value. Make a curl/Postman POST to `/api/ci-workflows/steps/validate` with and without the token.
**Expected:** Without token → 401. With invalid token → 401. With valid token + valid YAML body → 200 with `{valid: true, step: {...}, conflicts: [], warnings: []}`.
**Why human:** Runtime HTTP behavior of the live server, including session vs. API token auth path switching.
