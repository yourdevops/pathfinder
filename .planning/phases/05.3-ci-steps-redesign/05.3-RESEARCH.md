# Phase 05.3: CI Steps Redesign - Research

**Researched:** 2026-02-02
**Domain:** Django plugin architecture, CI engine abstraction, HTMX table filtering
**Confidence:** HIGH

## Summary

This phase refactors CI capabilities out of hardcoded core modules into the existing plugin system. The codebase currently has CI step scanning (`scan_ci_steps` in `core/git_utils.py`), manifest generation (`generate_github_actions_manifest` in `core/ci_manifest.py`), and manifest pushing (`push_ci_manifest` in `core/tasks.py`) all hardcoded to GitHub Actions. The redesign makes these engine-agnostic by defining a CI capability interface on `BasePlugin` and moving engine-specific logic into plugin implementations.

The existing `BasePlugin` class in `plugins/base.py` uses abstract methods with a singleton `PluginRegistry`. The `GitHubPlugin` in `plugins/github/plugin.py` has `category = "scm"` and `capabilities = [...]`. The redesign adds CI capabilities to this same plugin (SCM + CI in one plugin) via a mixin or additional abstract methods.

The UI portion replaces card layouts on Steps and Repositories pages with tables, removes the top-level Runtimes nav item, and adds HTMX-powered inline filter dropdowns (Engine, Runtime, Runtime Version) on the Steps page.

**Primary recommendation:** Use a `CICapableMixin` class that plugins opt into, keeping `BasePlugin` lean. Move `scan_ci_steps` to `core/ci_steps.py` as an engine-agnostic walker that delegates filename matching and parsing to plugin methods. Move `generate_github_actions_manifest` into `GitHubPlugin`. Add `engine` field to `StepsRepository` and `CIStep` models.

### Pattern 1: CICapableMixin
**What:** A mixin class that CI-capable plugins inherit alongside BasePlugin
**When to use:** Any plugin that provides CI engine capabilities
**Example:**
```python
# plugins/base.py
class CICapableMixin:
    """Mixin for plugins that provide CI engine capabilities."""

    @property
    def engine_name(self) -> str:
        """Return CI engine identifier, e.g., 'github_actions', 'bitbucket_pipelines'."""
        raise NotImplementedError

    @property
    def engine_display_name(self) -> str:
        """Human-readable engine name, e.g., 'GitHub Actions'."""
        raise NotImplementedError

    @property
    def engine_file_name(self) -> str:
        """Filename to discover in step repos, e.g., 'action.yml'."""
        raise NotImplementedError

    def parse_step_file(self, file_content: dict) -> dict:
        """Parse engine-native step file and extract x-pathfinder metadata."""
        raise NotImplementedError

    def generate_manifest(self, workflow) -> str:
        """Generate CI manifest YAML string for a workflow."""
        raise NotImplementedError

    def manifest_path(self, service) -> str:
        """Return the file path where manifest should be placed in repo."""
        raise NotImplementedError
```

### Pattern 2: Engine-Agnostic Step Discovery
**What:** `core/ci_steps.py` walks a cloned repo for files matching a given filename, delegates parsing to the plugin
**When to use:** During scan_steps_repository task
**Example:**
```python
# core/ci_steps.py
import os
import yaml

def discover_steps(repo_path: str, engine_file_name: str) -> list[dict]:
    """
    Walk repo_path looking for files named engine_file_name at any depth.
    Returns list of dicts with file path and parsed YAML content.
    No engine-specific logic here.
    """
    discovered = []
    for root, dirs, files in os.walk(repo_path):
        # Skip hidden directories
        dirs[:] = [d for d in dirs if not d.startswith('.')]
        if engine_file_name in files:
            file_path = os.path.join(root, engine_file_name)
            with open(file_path) as f:
                content = yaml.safe_load(f) or {}
            rel_dir = os.path.relpath(root, repo_path)
            discovered.append({
                'directory_path': rel_dir,
                'raw_content': content,
            })
    return sorted(discovered, key=lambda d: d['directory_path'])
```

### Pattern 4: Resolving CI and SCM Capabilities Separately
**What:** push_ci_manifest resolves the CI plugin (for manifest gen) and SCM plugin (for file push) independently
**When to use:** When pushing a manifest to a service repo
**Example:**
```python
# In refactored push_ci_manifest task:
# 1. Get CI capability from workflow's engine
ci_plugin = get_ci_plugin_for_engine(workflow_engine)
manifest_yaml = ci_plugin.generate_manifest(workflow)
manifest_file_path = ci_plugin.manifest_path(service)

# 2. Get SCM capability from project connection (unchanged)
scm_connection = get_default_scm_connection(service.project)
scm_plugin = scm_connection.get_plugin()
scm_plugin.update_or_create_file(config, repo_name, manifest_file_path, manifest_yaml, ...)
```

### Anti-Patterns to Avoid
- **Hardcoding engine filenames in core:** Currently `scan_ci_steps` hardcodes `action.yml` and `ci-steps/` directory. The new pattern delegates filename to the plugin.
- **Mixing CI and SCM concerns in push_ci_manifest:** Currently the task assumes GitHub for both manifest generation and file pushing. These must be separated.
- **Free-text engine field:** Per CONTEXT.md decision, the engine dropdown must be populated from installed CI-capable plugins, not free-text input.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTMX table filtering | Custom JavaScript filtering | HTMX `hx-get` + `hx-include` + `hx-target` | Already used in the app; server-side filtering is simpler and consistent |
| Runtime version dropdown cascading | Custom AJAX | HTMX `hx-get` triggered on change | Pattern already exists in RuntimeVersionsView |
| Plugin capability discovery | Custom registry | `PluginRegistry.all()` + `isinstance(plugin, CICapableMixin)` | Registry already exists; isinstance check is idiomatic Python |

**Key insight:** The existing plugin framework and HTMX patterns cover all the infrastructure needs. This phase is primarily about reorganizing code, not introducing new technologies.

## Common Pitfalls

### Pitfall 1: Breaking Existing Workflow Manifests
**What goes wrong:** Moving `generate_github_actions_manifest` into the plugin could break existing workflow detail views that call it directly.
**Why it happens:** Views and tasks import directly from `core/ci_manifest`.
**How to avoid:** Update all callers simultaneously. The function moves to `GitHubPlugin.generate_manifest()`, and callers resolve the plugin first. Since this is a redesign phase, do it in one sweep.
**Warning signs:** Import errors after moving the function.

### Pitfall 2: Circular Imports Between Plugins and Core
**What goes wrong:** Plugin implementations that need core models create circular imports.
**Why it happens:** `plugins/github/plugin.py` would need to import from `core.models` for workflow steps, while `core/tasks.py` imports from plugins.
**How to avoid:** Use lazy imports (import inside methods, not at module level) as already done in the codebase (see `core/tasks.py` patterns). The plugin's `generate_manifest` method should accept a workflow object, not import the model at module level.
**Warning signs:** `ImportError` during plugin autodiscovery.

### Pitfall 3: Migration Conflicts with Engine Field
**What goes wrong:** Adding `engine` field to `StepsRepository` and `CIStep` without a default value or data migration causes issues for existing records.
**Why it happens:** Existing rows have no engine value.
**How to avoid:** Add the field with `default="github_actions"` since all existing data is GitHub Actions. This is safe per the "no backwards compatibility" rule, but a data migration is cleaner.
**Warning signs:** Migration errors or empty engine fields after migration.

### Pitfall 4: Re-scan Wipe-and-Reimport vs CIWorkflowStep Foreign Keys
**What goes wrong:** Re-scan behavior (per CONTEXT.md: "wipe all steps from the repo and re-import fresh") can break CIWorkflowStep records that reference CIStep via ForeignKey with `on_delete=PROTECT`.
**Why it happens:** `CIStep` has `on_delete=models.PROTECT` from `CIWorkflowStep`, preventing deletion of steps that are in use.
**How to avoid:** Either (a) change re-scan to update-in-place for steps that still exist, only deleting truly removed ones, or (b) change `on_delete=PROTECT` to `on_delete=CASCADE` (but this silently destroys workflow steps). Option (a) is safer -- the current `scan_steps_repository` already uses `update_or_create` followed by a delete of removed steps, which is compatible with PROTECT. The CONTEXT.md says "wipe all steps and re-import fresh" but the current implementation already does the safer update_or_create approach. Keep the update_or_create pattern but make it engine-agnostic.
**Warning signs:** `ProtectedError` exception during re-scan.

### Pitfall 5: Directory Path vs Directory Name for Auto-Discovery
**What goes wrong:** The current `CIStep.directory_name` field stores a flat name (e.g., `setup-python`). With auto-discovery at any nesting depth, the directory path becomes `build/python/docker-build`.
**Why it happens:** The redesign allows arbitrary nesting, but the model field is `directory_name` with max_length=255.
**How to avoid:** Change `directory_name` semantics to store the relative path from repo root (e.g., `build/python/docker-build`). The field name could be renamed to `directory_path` for clarity, but since max_length=255 is sufficient and the unique_together constraint already scopes to repository, just document the change and update the scanning code.
**Warning signs:** Duplicate step entries or missing steps due to path handling.

### Helper to Get CI Plugin by Engine
```python
# plugins/base.py or core/ci_steps.py
def get_ci_plugin_for_engine(engine_name: str):
    """Find the installed plugin that provides a given CI engine."""
    for plugin in PluginRegistry.all().values():
        if isinstance(plugin, CICapableMixin) and plugin.engine_name == engine_name:
            return plugin
    return None

def get_available_engines() -> list[tuple[str, str]]:
    """Return list of (engine_name, display_name) for all installed CI-capable plugins."""
    engines = []
    for plugin in PluginRegistry.all().values():
        if isinstance(plugin, CICapableMixin):
            engines.append((plugin.engine_name, plugin.engine_display_name))
    return engines
```

## Open Questions

1. **Engine name convention**
   - What we know: CONTEXT.md says engine dropdown populated from installed plugins. The design doc uses lowercase strings like `github`, `bitbucket`, `jenkins`.
   - What's unclear: Should the engine name be `github` or `github_actions`? The plugin name is `github` but the CI engine is specifically "GitHub Actions" (GitHub the SCM vs GitHub Actions the CI engine).
   - Recommendation: Use `github_actions` as the engine name to distinguish from GitHub as an SCM. This allows a future where `GitHubPlugin` provides SCM capability and `engine_name = "github_actions"` for CI capability. The `engine_file_name` remains `action.yml`.

2. **Runtimes.yml handling in multi-engine world**
   - What we know: `parse_runtimes_yml` is currently engine-agnostic (just reads a YAML file). Runtimes are tied to a `StepsRepository`, not to an engine.
   - What's unclear: Should runtimes.yml parsing also move to the plugin, or stay in `core/ci_steps.py`?
   - Recommendation: Keep `parse_runtimes_yml` in `core/ci_steps.py` since the x-pathfinder runtimes format is engine-agnostic. The runtimes describe what programming languages the steps support, not anything engine-specific.

3. **RuntimeFamily model removal**
   - What we know: The Runtimes nav item is being removed. RuntimeFamily is currently used for the workflow creation form's runtime dropdown.
   - What's unclear: Should RuntimeFamily model be removed or kept?
   - Recommendation: Keep `RuntimeFamily` model. It's still needed for the workflow creation form's runtime family/version dropdowns. Only the dedicated Runtimes page and nav item are removed. The Steps page gets runtime filters that query RuntimeFamily.
