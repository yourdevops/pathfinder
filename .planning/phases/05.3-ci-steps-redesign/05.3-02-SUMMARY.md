# Phase 5.3 Plan 02: Refactor tasks, views, forms; remove RuntimesView

Plugin-based manifest generation in push task, engine dropdown on repo registration, steps catalog filtering by engine/runtime/version, RuntimesView and nav link removed.

## Completed Tasks

| # | Task | Commit | Key Changes |
|---|------|--------|-------------|
| 1 | Refactor push_ci_manifest task and clean ci_manifest.py | 950fe60 | Plugin-based manifest generation in push task; removed generate_github_actions_manifest from ci_manifest.py; updated views to use plugin |
| 2 | Update views, forms, URLs, and navigation | 78f35f3 | Engine dropdown on repo form; StepsCatalogView with filters; StepsTableView HTMX endpoint; RuntimesView removed; nav link removed |

## Changes Detail

### Task 1: push_ci_manifest + ci_manifest cleanup

**core/tasks.py (push_ci_manifest only):**
- Replaced `from core.ci_manifest import generate_github_actions_manifest` with `from plugins.base import get_ci_plugin_for_engine`
- Resolves CI plugin from first workflow step engine (fallback: `github_actions`)
- Uses `ci_plugin.generate_manifest(workflow)` instead of hardcoded function
- Uses `ci_plugin.manifest_path(service)` instead of hardcoded `.github/workflows/` path

**core/ci_manifest.py:**
- Removed `generate_github_actions_manifest` function entirely (83 lines)
- Removed `yaml` import
- Updated module docstring to reflect runtime-compatibility-only scope
- Retained: `normalize_version`, `is_step_compatible`, `get_compatible_steps`

**core/views/ci_workflows.py + core/views/services.py:**
- Updated WorkflowDetailView, WorkflowManifestView, and ServiceDetailView to use plugin-based manifest generation
- All three resolve engine from first workflow step, get CI plugin, call `ci_plugin.generate_manifest()`

### Task 2: Views, forms, URLs, navigation

**core/forms/ci_workflows.py:**
- Added `engine` ChoiceField to `StepsRepoRegisterForm` with DARK_SELECT styling
- Dynamic population in `__init__` via `get_available_engines()` from plugins.base
- Standardized existing field widget classes to use DARK_INPUT/DARK_SELECT constants

**core/views/ci_workflows.py:**
- `StepsRepoRegisterView.post()`: passes `engine=form.cleaned_data["engine"]` to create
- `StepsCatalogView`: rewritten with engine/runtime/runtime_version query param filtering; HTMX partial support for `steps-table-body` target
- `StepsTableView`: new HTMX endpoint returning filtered `_steps_table.html` partial
- `StepsRepoListView`: annotates `repo.engine_display` from plugin display name
- `RuntimesView`: deleted entirely
- Shared `_filter_steps()` helper extracts common filtering logic

**core/urls.py:**
- Removed `RuntimesView` import and `runtimes/` URL pattern
- Added `StepsTableView` import and `steps/table/` URL pattern (name: `steps_table`)

**core/templates/core/components/nav.html:**
- Removed Runtimes nav link from CI Workflows sidebar section
