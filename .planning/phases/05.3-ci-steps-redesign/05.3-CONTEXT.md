# Phase 5.3: CI Steps Redesign - Context

**Gathered:** 2026-02-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Refactor CI capabilities into the plugin system. Step discovery becomes engine-agnostic, manifest generation moves from core into plugin implementations, core/git_utils.py is cleaned to only generic git operations. UI updates: tables replace cards for Steps and Repositories pages, Runtimes nav item removed and replaced with inline filters on Steps page.

</domain>

<decisions>
## Implementation Decisions

### Engine registration flow
- Engine dropdown populated from installed plugins that have CI capabilities (not free-text)
- For now only GitHub Actions; Bitbucket Pipelines planned next
- Add engine field to the existing steps repository registration form (no form redesign)
- Plugin is required for scanning — no plugin for an engine means you can't register a repo for that engine
- Re-scan behavior: wipe all steps from the repo and re-import fresh (not update-in-place)

### Steps table design
- Table layout replaces current card layout
- Columns: name, phase, engine, runtimes, tags, repo (source repository)
- Click row for full step details

### Repositories table design
- Table layout replaces current card layout
- Columns: name, engine, URL, step count, last synced

### Steps page filtering
- Inline dropdowns above the table for Engine, Runtime, Runtime Version
- Table updates via HTMX when filters change

### Page structure
- Steps and Repositories remain separate pages (not combined with tabs)
- Top-level Runtimes nav item removed

### Claude's Discretion
- Plugin capability interface design (mixin vs abstract methods vs separate class)
- How to structure core/ci_steps.py internals
- Migration strategy for existing data
- Exact table styling and responsive behavior
- HTMX filter mechanics

</decisions>

- Design doc (`docs/ci-steps-redesign.md`) has detailed proposed structure for x-pathfinder metadata, plugin capabilities, and manifest push flow

</specifics>
