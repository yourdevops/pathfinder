# Phase 06.3: Security & Compliance Design -- Secrets, SLSA L3, SOX RBAC - Research

**Researched:** 2026-02-13
**Domain:** Security design documentation -- secrets management, SLSA provenance/signing, SOX-compliant RBAC
**Confidence:** HIGH (design docs phase; all three domains well-established with clear standards)

## Summary

This phase produces RFC-style design documentation for three security/compliance capabilities: secrets management (hybrid internal/external architecture), artifact provenance signing (SLSA Level 3 via Cosign/in-toto), and SOX-compliant RBAC (granular permissions with four-eyes approval). The output is docs, not code. All three domains have mature industry standards with well-documented patterns.

The secrets design extends Pathfinder's existing Fernet encryption infrastructure (`core/encryption.py`) to a new Secret entity with write-only semantics, while defining a `SecretsCapableMixin` plugin interface for external vault integration. The SLSA provenance design defines CI steps for signing and attestation using Cosign with KMS backends, specifies in-toto v1.0 predicate format, and extends Pathfinder's existing build verification model to include provenance metadata. The RBAC redesign replaces the current coarse owner/contributor/viewer model with granular CRUD permissions, predefined role bundles, three-tier scoping (system/project/environment), and an explicit approval workflow for production deployments.

**Primary recommendation:** Organize as a single `docs/security/` directory with separate documents per sub-domain, mirroring the `docs/ci-workflows/` and `docs/deployments/` structure. Each doc should be self-contained with clear model definitions, plugin interfaces, and interaction diagrams.

## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Secrets Management
- **Hybrid architecture:** Internal encrypted store as default, external vault integration as a connectable plugin
- **Separate namespace:** Secrets are a distinct entity from env vars -- managed independently, merged at deploy time. Separate RBAC possible
- **Write-only in UI:** Secret values are never visible once set -- not to admins, not to anyone. Can only be overwritten
- **Runtime injection:** Deploy snapshot stores secret references (names/paths), not values. Actual values resolved at runtime by the responsible deploy plugin. For external vault plugins, the External Secrets operator or vault agent handles rotation
- **System-wide encryption key:** Single Fernet key (extends current architecture). Key rotation documented as future concern, not designed in detail now
- **No migration needed:** No secrets exist in the system yet -- greenfield implementation
- **External vault targets:** Plugin interface designed for HashiCorp Vault + cloud-native secret managers (AWS Secrets Manager, GCP Secret Manager). Plugin interface must be generic enough to support all three

#### SLSA Provenance & Artifact Signing
- **CI generates, Pathfinder verifies:** Provenance generated by the CI engine via a CI step (added to the steps library). Pathfinder declares the expected attestation format and consumes metadata after build
- **Cosign with pluggable KMS backends:** Cosign as the signing tool. Customer-managed key support via AWS KMS, Vault Transit, etc.
- **DigiCert TSA for timestamping:** Use http://timestamp.digicert.com/ for RFC 3161 timestamps
- **Rekor disabled by default:** Transparency log opt-in only -- privacy concerns for enterprises who don't want artifact signatures in a public log
- **in-toto format only:** SLSA's native attestation format as primary goal
- **Opt-in per environment:** Attestation verification configurable per environment, enabled by default. Non-production can skip
- **Index after build, re-verify at deploy:** Pathfinder records signing metadata (artifact digest, signer identity, bundle location) at build time. Full cryptographic re-verification at deploy time (avoid TOCTOU gap)
- **SBOM designed in:** SBOM generation as a CI step, attached as attestation alongside provenance. Format (SPDX/CycloneDX) to be determined in research

#### SOX-Compliant RBAC
- **Granular CRUD permissions:** Every Pathfinder resource type (services, deployments, CI workflows, secrets, connections, etc.) has standard CRUD permissions
- **Predefined role bundles:** Pathfinder ships with predefined roles (curated permission sets per functionality and level). Replaces the simple owner/contributor/viewer model
- **Custom roles via Django config:** Custom roles defined in Django settings/fixtures for now. Config-as-Code (CasC) designed in a future phase. Not UI (would get messy fast)
- **Three-tier scoping:** System-wide roles + Project roles + Environment-specific overrides. Full hierarchy
- **Explicit approval workflow for production:** Production deployments require a separate user to approve. Approval request with expiry, notifications, audit trail. Four-eyes principle enforced

## Standard Stack

### Core Standards
| Standard | Version | Purpose | Why Standard |
|----------|---------|---------|--------------|
| SLSA | v1.0 | Supply chain security framework | Industry standard for artifact provenance levels |
| in-toto attestation | v1 | Attestation envelope format | SLSA's native format; DSSE envelope + predicate |
| SLSA Provenance predicate | v1.0 | Provenance metadata schema | `https://slsa.dev/provenance/v1` -- buildDefinition + runDetails |
| Cosign | 2.x | Container signing/attestation tool | Sigstore's standard; pluggable KMS, OCI-native storage |
| RFC 3161 | - | Timestamping protocol | Standard for trusted timestamps; DigiCert TSA specified |
| Fernet (cryptography) | existing | Symmetric encryption for internal secrets | Already used in Pathfinder for IntegrationConnection |

### Supporting Technologies
| Technology | Purpose | When Referenced |
|------------|---------|----------------|

| HashiCorp Vault Transit | KMS backend for Cosign signing | Customer-managed keys for enterprise signing |

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|

## Architecture Patterns

### Recommended Document Structure

```
docs/security/

  secrets.md                # Secrets management: model, encryption, plugin interface, UI
  provenance.md             # SLSA provenance: CI steps, attestation format, verification

```

**Rationale:** A single `docs/security/` directory is preferable to three separate directories because:
1. The three domains interact heavily (RBAC gates secret access; provenance verification respects RBAC; secrets are referenced in deployment which requires approval)
2. A shared README can describe the threat model and cross-cutting concerns once
3. Mirrors the pattern of `docs/ci-workflows/` (7 docs in one directory) and `docs/deployments/` (6 docs in one directory)
4. Three domains is fewer documents than ci-workflows -- a directory-per-domain would create unnecessary nesting

### Pattern 1: Secrets Model and Plugin Interface

**What:** Internal Secret entity with Fernet encryption, SecretsCapableMixin for external vault plugins
**Key design points:**

```
Secret:
  - id: UUID
  - project: FK Project
  - environment: FK Environment (nullable -- project-wide if null)
  - name: string (DNS-compatible, unique within scope)
  - encrypted_value: BinaryField (Fernet-encrypted)
  - source: enum (internal, external)
  - external_ref: string (nullable -- vault path for external secrets)
  - description: text
  - created_by, updated_by: string
  - created_at, updated_at: datetime
```

**Plugin interface:**
```python
class SecretsCapableMixin:
    """Mixin for plugins that provide external secrets management."""

    def resolve_secrets(
        self, config: dict, secret_refs: list[dict]
    ) -> dict[str, str]:
        """Resolve secret references to actual values at deploy time.

        Args:
            config: Plugin connection configuration
            secret_refs: List of {name, external_ref} dicts

        Returns:
            Dict mapping secret name to resolved value
        """

    def validate_secret_ref(self, config: dict, external_ref: str) -> bool:
        """Validate that an external secret reference is accessible."""

    def list_available_secrets(self, config: dict, path: str) -> list[str]:
        """List available secrets at a path in the external vault."""
```

**Interaction with deploy snapshot:**
- Deploy snapshot stores `secret_refs: [{name: "DB_PASSWORD", source: "internal", ref: "secrets/proj-a/prod/DB_PASSWORD"}, ...]`
- Internal secrets: Pathfinder resolves at deploy time via `core/encryption.py`
- External secrets: Deploy plugin's `SecretsCapableMixin.resolve_secrets()` handles resolution
- For Kubernetes: plugin creates ExternalSecret CR or injects via init container; no plaintext in Pathfinder

### Pattern 2: SLSA Provenance CI Step and Verification

**What:** CI step in the batteries-included steps library that generates SLSA provenance and signs with Cosign; Pathfinder verifies attestation after build

**CI step flow (runs in CI engine, not Pathfinder):**
```bash
# Step 1: Generate SLSA provenance predicate (JSON)
# The step generates buildDefinition + runDetails per SLSA v1.0
cat > provenance.json <<EOF
{
  "buildDefinition": {
    "buildType": "https://pathfinder.dev/ci-workflow/v1",
    "externalParameters": {
      "workflow": "$WORKFLOW_NAME",
      "version": "$WORKFLOW_VERSION",
      "source": {"uri": "$REPO_URL", "digest": {"sha1": "$COMMIT_SHA"}}
    },
    "resolvedDependencies": [...]
  },
  "runDetails": {
    "builder": {"id": "$CI_ENGINE_URL"},
    "metadata": {"invocationId": "$CI_RUN_ID", "startedOn": "...", "finishedOn": "..."}
  }
}
EOF

# Step 2: Sign and attest the container image
cosign attest \
  --predicate provenance.json \
  --type slsaprovenance1 \
  --key hashivault://signing-key \  # or awskms://, or keyless
  --tlog-upload=false \              # Rekor disabled by default
  --timestamp-server-url=http://timestamp.digicert.com \
  $IMAGE_REF

# Step 3: (Optional) Generate and attach SBOM
syft $IMAGE_REF -o cyclonedx-json > sbom.json
cosign attest \
  --predicate sbom.json \
  --type cyclonedx \
  --key hashivault://signing-key \
  --tlog-upload=false \
  --timestamp-server-url=http://timestamp.digicert.com \
  $IMAGE_REF
```

**Pathfinder verification flow (after build webhook):**
1. Fetch attestation: `cosign verify-attestation --type slsaprovenance1 --key <public-key> $IMAGE_REF`
2. Parse in-toto statement, extract provenance predicate
3. Verify buildType matches Pathfinder's expected type
4. Verify externalParameters.source.digest matches Build.commit_sha
5. Store signing metadata on Build record: `attestation_digest`, `signer_identity`, `bundle_ref`, `timestamp_verified`
6. At deploy time: full re-verification (repeat steps 1-4 with the image being deployed, not cached result)

### Pattern 3: Three-Tier RBAC with Granular Permissions

**What:** Replace owner/contributor/viewer with granular CRUD permissions, role bundles, and three-tier scoping

**Permission model:**
```
Permission = (resource_type, action)

Resource types: service, deployment, ci_workflow, secret, connection,
                environment, project, user, group, audit_log, steps_repo

Actions: create, read, update, delete, deploy, approve
```

**Role bundles (predefined, shipped with Pathfinder):**

System roles:
| Role | Description | Key Permissions |
|------|-------------|-----------------|
| `platform-admin` | Full system access | All permissions on all resources |
| `platform-operator` | Manage integrations and infrastructure | CRUD on connections, steps_repos, environments; read all |
| `security-auditor` | Read-only compliance view | Read all resources; read audit_log; export reports |
| `secrets-admin` | Manage secrets across projects | CRUD on secrets (system-wide) |

Project roles:
| Role | Description | Key Permissions |
|------|-------------|-----------------|
| `project-admin` | Full project control | All CRUD within project scope; manage members |
| `release-manager` | Deploy to production | Deploy + approve on all environments |
| `developer` | Build and deploy non-prod | CRUD services, CI workflows; deploy non-prod; create secrets |
| `viewer` | Read-only project access | Read all project resources |

Environment overrides:
| Override | Description | Key Permissions |
|----------|-------------|-----------------|
| `deploy-approver` | Can approve production deployments | approve on deployment (specific environment) |
| `secrets-reader` | Can view secret names (not values) in environment | read on secrets (specific environment) |

**Scoping resolution order:**
1. System role (highest priority, overrides all)
2. Environment-specific override (if set)
3. Project role (base project access)
4. Implicit authenticated user (read plugins, read own projects)

### Pattern 4: Production Approval Workflow

**What:** Four-eyes principle -- deployment to production requires explicit approval from a different user

**Approval model:**
```
DeploymentApproval:
  - deployment: FK Deployment (OneToOne)
  - requested_by: FK User
  - requested_at: datetime
  - approved_by: FK User (nullable)
  - approved_at: datetime (nullable)
  - rejected_by: FK User (nullable)
  - rejected_at: datetime (nullable)
  - expires_at: datetime
  - status: enum (pending, approved, rejected, expired)
  - reason: text (required for rejection)
  - emergency_override: bool (default false)
  - override_justification: text (required if emergency_override)
```

**Workflow:**
1. User requests deployment to production environment
2. Pathfinder creates Deployment (status: `pending_approval`) + DeploymentApproval (status: `pending`)
3. Users with `deploy-approver` override OR `release-manager` project role receive notification
4. Approver (must be different user) reviews and approves/rejects
5. On approval: Deployment transitions to `pending` (normal flow)
6. On rejection: Deployment transitions to `cancelled` with reason
7. On expiry: Deployment transitions to `cancelled` automatically

**Emergency override:** System admins can bypass approval with mandatory justification logged to audit trail. Emergency overrides are flagged in the UI and audit log with high-visibility indicators.

### Anti-Patterns to Avoid

- **Storing secret values in deployment snapshots:** Snapshot stores references only, never plaintext values
- **Trusting CI webhook payloads for attestation data:** Always fetch attestation from OCI registry independently (extends existing build verification pattern)
- **Allowing self-approval:** The four-eyes principle is violated if the requesting user can also approve -- enforce `approved_by != requested_by`
- **Hardcoding role permissions:** Permissions should be data-driven (fixtures/settings), not hardcoded in view logic, to enable future CasC
- **Caching attestation verification:** Re-verify at deploy time to prevent TOCTOU attacks -- the attestation could have been revoked between build and deploy

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Container signing | Custom signing protocol | Cosign with KMS backends | Industry standard; handles OCI storage, DSSE, key management |
| Attestation format | Custom JSON schema | in-toto v1 with SLSA provenance predicate | Interoperable; consumed by policy engines (Kyverno, OPA) |
| SBOM generation | Custom dependency scanning | Syft or Trivy in CI step | Established tools; CycloneDX/SPDX output |
| Timestamping | Custom timestamp service | RFC 3161 via DigiCert TSA | Legally recognized; cryptographically verifiable |
| Secret encryption | Custom AES implementation | Fernet from cryptography library | Already in use; authenticated encryption with HMAC |
| External secret sync | Custom vault polling | External Secrets Operator (K8s) or vault agent | Handles rotation, caching, error recovery |

**Key insight:** The signing and attestation toolchain (Cosign + in-toto + SLSA predicate) is a complete ecosystem. Pathfinder's role is to define the expected format and verify compliance -- not to implement signing primitives. The CI step does the heavy lifting; Pathfinder is the policy enforcement point.

## Common Pitfalls

### Pitfall 1: DigiCert TSA Certificate Chain Compatibility
**What goes wrong:** Cosign historically had issues verifying DigiCert's TSA certificate chain due to strict EKU (Extended Key Usage) validation on intermediate certificates.
**Why it happens:** DigiCert's intermediate certificates lacked explicit timestamping EKU; Cosign required it on all certificates in the chain.
**How to avoid:** Require Cosign >= 2.x with timestamp-authority library >= 1.2.8, which relaxed EKU validation. Document the full certificate chain requirement for `--timestamp-certificate-chain` during verification.
**Warning signs:** Verification fails with "certificate chain validation error" when using DigiCert timestamps.
**Confidence:** HIGH (verified via sigstore/cosign#3632 resolution)

### Pitfall 2: Secret References Leaking into Logs
**What goes wrong:** Secret names or references appear in operation logs, deployment snapshots, or error messages.
**Why it happens:** Developers log full deploy configurations or error context without filtering secret references.
**How to avoid:** Design the Secret model with `__repr__` that never reveals the value. Extend the existing sensitive field policy (from docs/logging.md) to cover secret names. The `env_vars_snapshot` already exists and is well-scoped -- secret references should follow the same pattern but in a separate `secret_refs_snapshot` field.
**Warning signs:** Secret names appearing in system logs or error messages.

### Pitfall 3: Self-Approval in Four-Eyes Workflow
**What goes wrong:** The system allows the deployment requester to also approve it, violating SOX segregation of duties.
**Why it happens:** Insufficient validation in the approval endpoint.
**How to avoid:** Enforce `approved_by != requested_by` as a database-level constraint or model validation. Design the approval UI to hide the approve button for the requesting user. Emergency override is a separate code path with mandatory justification.
**Warning signs:** Audit log shows same user in both `requested_by` and `approved_by` fields.

### Pitfall 4: Conflating Environment Variables and Secrets
**What goes wrong:** The boundary between env vars (visible, snapshotted) and secrets (write-only, resolved at runtime) becomes blurred, leading to secrets appearing in snapshots.
**Why it happens:** Both are key-value pairs injected into containers; developers treat them interchangeably.
**How to avoid:** Design document must clearly specify: env vars are in `env_vars_snapshot` (frozen, visible, auditable). Secrets are in `secret_refs_snapshot` (reference-only, values never in Pathfinder's database except for internal encrypted store). The merge happens at deploy time in the deploy plugin, not in Pathfinder core.
**Warning signs:** Deployment detail page showing secret values; secret values in audit log changes.

### Pitfall 5: Attestation Verification Scope Creep
**What goes wrong:** Attempting to verify attestations at every promotion boundary creates operational complexity and latency.
**Why it happens:** Over-interpretation of "verify at deploy time" -- re-verifying at every step rather than at the two critical points.
**How to avoid:** Verify at two points only: (1) after build (index metadata), (2) before deploy to target environment (re-verify the specific artifact being deployed). Promotion between environments does not need its own verification step -- the deploy to the target environment handles it.
**Warning signs:** Slow promotion workflows; verification failures blocking non-production deployments.

## Discretion Recommendations

### Attestation Verification Points
**Recommendation:** Verify at build ingestion (index metadata) and at deploy time (re-verify before execution). No separate verification at promotion boundaries -- the deploy-time verification at the target environment is sufficient. This avoids unnecessary latency while preventing TOCTOU attacks.
**Confidence:** HIGH

### CI Steps Library Supply Chain Verification
**Recommendation:** Trust the repository with branch protection (existing mechanism per docs/ci-workflows/steps-catalog.md). Do NOT sign individual steps. Branch protection (required reviews, no force push, no direct push) is already enforced at registration time. Adding step signatures would require a complex key management system for step authors with minimal added security over the existing branch protection + manifest hash verification chain.
**Confidence:** HIGH

### Document Structure
**Recommendation:** Single `docs/security/` directory with README.md + 3 domain-specific documents (secrets.md, provenance.md, rbac.md). See Architecture Patterns section for rationale.
**Confidence:** HIGH

### Deployment Review Integration
**Recommendation:** Address three findings from docs/deployments/review.md in these design docs; defer three to implementation:
- **Address now (in rbac.md):** "Authorization model insufficient for regulated industries" -- this is exactly what the SOX RBAC redesign solves
- **Address now (in provenance.md):** "Artifact signing absent from promotion chain" -- this is exactly what the SLSA provenance design solves
- **Address now (in secrets.md):** "God mode credential problem" -- the secrets design with external vault integration directly mitigates centralized credential risk
- **Defer to Phase 7:** Deployment timeouts, lifecycle hooks, deployment freezes -- these are implementation concerns, not security design

**Confidence:** HIGH

### Approval Expiry Duration
**Recommendation:** Default 4 hours. Rationale:
- Too short (e.g., 30 min) creates friction -- approver may be in a meeting
- Too long (e.g., 24h) defeats the purpose -- conditions may change overnight
- 4 hours covers a half-day work window; configurable per environment
- Production environments may want shorter (2h); staging may want longer (8h) or disable entirely

**Confidence:** MEDIUM (reasonable default; will need user feedback)

### Emergency Override Mechanism
**Recommendation:** System admins (`platform-admin` role) can bypass approval with:
1. Mandatory justification text field (minimum 20 characters)
2. Emergency flag set on the DeploymentApproval record
3. Audit log entry at CRITICAL level
4. Visual indicator in deployment history (red "Emergency Override" badge)
5. Weekly compliance report includes all emergency overrides

No separate "break glass" role -- system admins already have full access. The justification and audit trail provide the compliance paper trail.
**Confidence:** HIGH

### SBOM Format Choice
**Recommendation:** CycloneDX (JSON format).
**Rationale:**
- CycloneDX is better suited for security-focused use cases (vulnerability identification, dependency analysis) -- which is Pathfinder's primary concern
- SPDX is stronger for licensing compliance -- less relevant for a deployment platform
- CycloneDX natively supports VEX (Vulnerability Exploitability eXchange), hashing, and dependency trees
- Cosign natively supports `--type cyclonedx` for attestation
- Syft supports both formats; CycloneDX JSON is the more commonly consumed format in security tooling
- SPDX is not excluded -- if customers need it, a second CI step can generate SPDX alongside CycloneDX

**Confidence:** HIGH

### Cosign Verification in Pathfinder (Python pseudocode)

```python
# Source: Pathfinder design -- how Pathfinder verifies attestations after build
import subprocess
import json

def verify_artifact_attestation(image_ref: str, public_key_path: str) -> dict:
    """Verify SLSA provenance attestation on a container image.

    Called after build completion (index metadata) and before
    deployment (re-verify to prevent TOCTOU).

    Returns dict with verification result and extracted metadata.
    """
    result = subprocess.run(
        [
            "cosign", "verify-attestation",
            "--type", "slsaprovenance1",
            "--key", public_key_path,
            "--timestamp-certificate-chain", "/etc/pathfinder/digicert-tsa-chain.pem",
            image_ref,
        ],
        capture_output=True, text=True,
    )

    if result.returncode != 0:
        return {"verified": False, "error": result.stderr}

    # Parse in-toto statement from output
    statements = [json.loads(line) for line in result.stdout.strip().split("\n")]
    provenance = statements[0]["payload"]  # base64-decoded in-toto statement

    return {
        "verified": True,
        "build_type": provenance["predicate"]["buildDefinition"]["buildType"],
        "source_digest": provenance["predicate"]["buildDefinition"]["externalParameters"]["source"]["digest"],
        "builder_id": provenance["predicate"]["runDetails"]["builder"]["id"],
        "invocation_id": provenance["predicate"]["runDetails"]["metadata"]["invocationId"],
    }
```

### Secret Model Extension of Existing Encryption

```python
# Source: Extends core/encryption.py pattern already in Pathfinder
# Secrets use the same Fernet key as IntegrationConnection.config_encrypted

class Secret(models.Model):
    """A secret value within a project, optionally scoped to an environment.

    Secret values are write-only: once set, they can only be overwritten,
    never read back through the UI or API. Values are Fernet-encrypted
    at rest using the same system-wide key as IntegrationConnection.
    """
    project = models.ForeignKey(Project, on_delete=models.CASCADE, related_name="secrets")
    environment = models.ForeignKey(
        Environment, on_delete=models.CASCADE, null=True, blank=True,
        related_name="secrets",
        help_text="If null, secret is project-wide (available in all environments)"
    )
    name = models.CharField(max_length=63, validators=[dns_label_validator])
```

### Approval Workflow Database Constraint

```python
# Source: Django model validation pattern for four-eyes principle
class DeploymentApproval(models.Model):
    deployment = models.OneToOneField(Deployment, on_delete=models.CASCADE)
    requested_by = models.ForeignKey(User, on_delete=models.PROTECT, related_name="+")
    approved_by = models.ForeignKey(User, on_delete=models.PROTECT, null=True, related_name="+")
    status = models.CharField(max_length=20, choices=[
        ("pending", "Pending Approval"),
        ("approved", "Approved"),
        ("rejected", "Rejected"),
        ("expired", "Expired"),
    ])
    expires_at = models.DateTimeField()
    emergency_override = models.BooleanField(default=False)
    override_justification = models.TextField(blank=True)

    def clean(self):
        if self.approved_by and self.approved_by == self.requested_by:
            if not self.emergency_override:
                raise ValidationError("Approver must be different from requester (four-eyes principle)")
        if self.emergency_override and not self.override_justification:
            raise ValidationError("Emergency override requires justification")
```

## Integration with Existing Pathfinder Design

### How Secrets Interact with Existing Systems

1. **Environment Binding (docs/deployments/environment-binding.md):** Currently states "secrets must come from external sources." The secrets design changes this to "secrets come from Pathfinder's internal store OR external sources via plugin." The `env_vars_snapshot` pattern is preserved -- a parallel `secret_refs_snapshot` stores references without values.

2. **Existing Fernet infrastructure (core/encryption.py):** The Secret model reuses the existing `encrypt_config`/`decrypt_config` functions and the same `PTF_ENCRYPTION_KEY`. No new encryption infrastructure needed.

3. **Plugin architecture (plugins/base.py):** `SecretsCapableMixin` follows the established pattern of `CICapableMixin` and `DeployCapableMixin`. External vault plugins implement this mixin alongside their primary category.

### How SLSA Provenance Extends Build Authorization

1. **Build Authorization (docs/ci-workflows/build-authorization.md):** The existing manifest hash verification ensures the CI pipeline is authorized. SLSA provenance adds a second layer: the artifact itself is cryptographically attested. Both checks must pass for production deployment.

2. **Build model extension:** New fields on Build: `attestation_verified` (bool), `attestation_digest` (string), `signer_identity` (string), `provenance_bundle_ref` (string). These complement the existing `verification_status` (Verified/Draft/Unauthorized).

3. **Steps catalog (docs/ci-workflows/steps-catalog.md):** The "Sign and Attest" step is added to the batteries-included repo under `security/sign-attest/action.yml` (phase: package).

### How RBAC Replaces Current Permissions

1. **Current RBAC (docs/rbac.md):** System roles (admin, operator, auditor, user) + project roles (owner, contributor, viewer). The new design preserves the two-tier concept but adds granular permissions and environment-specific overrides as a third tier.

2. **Current access control tables:** Every doc (services.md, environments.md, etc.) has an "Access Control" table mapping roles to actions. These tables will be regenerated from the new role bundles. The new tables will be more granular (e.g., "create service" vs "delete service" instead of just "manage services").

3. **Deployment lifecycle (docs/deployments/deployment-lifecycle.md):** Currently `triggered_by` is "auto" or username. The approval workflow adds a `pending_approval` state before `pending`, and the `triggered_by` + `approved_by` pair replaces the single-user attribution for production deployments.

## Open Questions

1. **Cosign binary distribution in CI environments**
   - What we know: Cosign is available as a GitHub Action (`sigstore/cosign-installer`), Docker image, and binary download
   - What's unclear: For Jenkins and other CI engines, how is cosign installed? Pre-baked in build images? Downloaded per-build?
   - Recommendation: Document as "CI engine provides cosign" -- each CI step implementation handles installation for its engine. The design doc specifies the interface, not the installation mechanism.

2. **Vault Transit key naming convention**
   - What we know: Cosign uses `hashivault://keyname` with `VAULT_ADDR` and `VAULT_TOKEN` env vars
   - What's unclear: Should Pathfinder enforce a naming convention for signing keys? Per-project? Per-environment?
   - Recommendation: Document recommended naming convention (`pathfinder-signing-{project}` or `pathfinder-signing-global`) but don't enforce it. The connection configuration stores the key URI.

3. **Permission migration from old roles to new roles**
   - What we know: Current code has hardcoded role checks (`has_system_role('admin')`, `project_role == 'owner'`)
   - What's unclear: How to migrate existing ProjectMembership data when role names change
   - Recommendation: Design doc should specify the mapping (owner -> project-admin, contributor -> developer, viewer -> viewer). Implementation phase handles migration. This is greenfield for new roles; existing roles can be mapped.
