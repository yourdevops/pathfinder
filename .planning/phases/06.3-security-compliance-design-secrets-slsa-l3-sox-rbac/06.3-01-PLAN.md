Create `docs/security/secrets.md` defining the complete secrets management design.

**Opening paragraph:** Secrets are sensitive configuration values (API keys, database passwords, private keys) that require stronger protection than standard environment variables. Pathfinder treats secrets as a separate namespace from env vars -- managed independently, merged at deploy time. Secret values are write-only: once set, they can never be read back through any interface.

**Section: Secret Model**
Define the Secret entity (design-level, not Django code):
```
Secret:
  - id: UUID
  - project: FK Project
  - environment: FK Environment (nullable -- project-wide if null)
  - name: string (DNS-compatible, unique within project+environment scope)
  - encrypted_value: binary (Fernet-encrypted, nullable for external secrets)
  - source: enum (internal, external)
  - external_ref: string (nullable -- vault path, ARN, or secret ID for external secrets)
  - description: text
  - created_by, updated_by: user reference
  - created_at, updated_at: datetime
```

Explain the scope inheritance: project-wide secrets (environment=null) are available in all environments. Environment-specific secrets override project-wide secrets of the same name. This parallels the env var cascade in docs/deployments/environment-binding.md.

**Section: Encryption Architecture**
- Internal secrets use Fernet symmetric encryption via the existing `core/encryption.py` infrastructure
- Same system-wide key (`PTF_ENCRYPTION_KEY` env var or `secrets/encryption.key` file)
- Key rotation is a documented future concern -- not designed in detail now (per locked decision)
- `encrypt_config`/`decrypt_config` functions already handle Fernet operations; Secret model extends this pattern

**Section: Write-Only Semantics**
Explain the write-only principle:
- Secret values are never visible once set -- not to admins, not via API, not in any UI
- UI shows secret name, source, description, last updated timestamp, and a masked placeholder
- Only actions available: create, overwrite (replace value), delete
- No "reveal" or "copy" functionality
- Internal resolution happens only inside deploy plugins at deploy time -- never exposed through views or API
- `__repr__` and `__str__` on the model must never include the value

**Section: Deploy-Time Resolution**
Explain how secrets flow into deployments:
1. Deploy snapshot stores `secret_refs_snapshot` (parallel to `env_vars_snapshot`): list of `{name, source, ref}` dicts. Never stores actual values.
2. At deploy time, the deploy plugin resolves references:
   - Internal secrets: Pathfinder decrypts via `core/encryption.py` and passes to plugin
   - External secrets: Plugin's `SecretsCapableMixin.resolve_secrets()` fetches from vault
3. For Kubernetes targets: plugin creates ExternalSecret CR or injects via init container; plaintext never stored in Pathfinder
4. The merge order: env vars (visible) + secrets (resolved) = final container environment. Secrets override env vars of the same name.

**Section: Plugin Interface -- SecretsCapableMixin**
Define the mixin interface (design-level pseudocode, following CICapableMixin and DeployCapableMixin patterns from plugins/base.py):

```
SecretsCapableMixin:
  resolve_secrets(config, secret_refs) -> dict[name, value]
    # Resolve external secret references to actual values at deploy time
    # Called by deploy pipeline, not exposed to UI/API

  validate_secret_ref(config, external_ref) -> bool
    # Validate that an external secret reference is accessible
    # Called when user creates/updates an external secret

  list_available_secrets(config, path) -> list[str]
    # List available secrets at a path in the external vault
    # Used by UI for autocomplete/browsing when configuring external secrets
```

Target implementations: HashiCorp Vault plugin, AWS Secrets Manager plugin, GCP Secret Manager plugin. The interface must be generic enough to support all three without provider-specific methods.

**Section: External Vault Integration Pattern**
Describe how an external vault plugin works:
- Plugin subclasses BasePlugin + SecretsCapableMixin (same pattern as CICapableMixin)
- Connection config stores vault URL, auth method, namespace/region
- resolve_secrets() is called at deploy time by the deploy pipeline
- For K8s: plugin can generate ExternalSecret resources instead of resolving values directly
- For Docker direct: plugin resolves values and passes as environment variables to container

**Section: Access Control**
Brief section cross-referencing docs/security/rbac.md:
- Secret CRUD permissions are part of the granular permission model
- `secrets-admin` system role for cross-project secret management
- Project roles determine secret access within a project (developer can create/update, viewer can see names only)
- Creating/updating secrets requires `create`/`update` permission on `secret` resource type

**Section: Addressing the "God Mode" Problem**
Explain how this design mitigates the centralized credential risk from the deployment review:
- External vault integration means production secrets never need to exist in Pathfinder's database
- Internal store is for teams without external vault infrastructure -- a stepping stone, not the recommended production pattern
- Credential scoping: external vault connections are per-environment, limiting blast radius
- Write-only semantics mean even a compromised Pathfinder UI cannot reveal secret values
- The deploy plugin resolves secrets at the last possible moment, minimizing the window of exposure
