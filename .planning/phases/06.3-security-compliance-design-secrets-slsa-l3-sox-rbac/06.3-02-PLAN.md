Create `docs/security/provenance.md` defining the complete SLSA provenance and artifact signing design.

**Opening paragraph:** Artifact provenance proves where a software artifact came from, how it was built, and that it has not been tampered with. Pathfinder uses SLSA (Supply-chain Levels for Software Artifacts) Level 3 as its provenance target: CI generates signed attestations using Cosign, and Pathfinder verifies them before allowing deployment. This is a second verification layer complementing the existing manifest hash verification in docs/ci-workflows/build-authorization.md.

**Section: Two-Layer Verification Model**
Explain how provenance complements existing build authorization:
- Layer 1 (existing): Manifest hash verification -- proves the CI pipeline definition is authorized (docs/ci-workflows/build-authorization.md)
- Layer 2 (new): Artifact attestation -- proves the artifact itself was produced by a verified build and has not been modified since
- Both layers must pass for production deployment. Non-production environments can opt out of Layer 2.

**Section: Responsibilities Split**
Define the CI-generates/Pathfinder-verifies model (per locked decision):
- CI engine is responsible for: generating provenance predicate, signing the artifact, attaching attestations to OCI registry
- Pathfinder is responsible for: declaring expected attestation format, fetching and verifying attestations after build, storing verification metadata, re-verifying at deploy time
- Pathfinder never signs artifacts -- it is the verification/policy enforcement point

**Section: Signing Toolchain**
Specify the tools and standards (per locked decisions):
- **Cosign 2.x** with `--type slsaprovenance1` (v1.0 format, NOT the deprecated `slsaprovenance` which generates v0.2)
- **KMS backends:** `hashivault://` (HashiCorp Vault Transit), `awskms://` (AWS KMS), `gcpkms://` (GCP KMS), or keyless signing via Sigstore
- **DigiCert TSA:** `--timestamp-server-url=http://timestamp.digicert.com` for RFC 3161 timestamps
- **Rekor:** `--tlog-upload=false` by default. Opt-in via environment configuration for teams that want transparency log entries. Note: many regulated enterprises cannot publish to public transparency logs.
- **Minimum versions:** Cosign >= 2.x, timestamp-authority >= 1.2.8 (required for DigiCert TSA certificate chain compatibility -- see sigstore/cosign#3632)

**Section: Attestation Format**
Define the in-toto v1 attestation structure with SLSA v1.0 provenance predicate:

```
DSSE Envelope:
  payloadType: "application/vnd.in-toto+json"
  payload: (base64-encoded Statement)
  signatures: [...]

Statement:
  _type: "https://in-toto.io/Statement/v1"
  subject:
    - name: <image-ref>
      digest: {sha256: <digest>}
  predicateType: "https://slsa.dev/provenance/v1"
  predicate:
    buildDefinition:
      buildType: "https://pathfinder.dev/ci-workflow/v1"
      externalParameters:
        workflow: <workflow-name>
        version: <workflow-version>
        source:
          uri: "git+<repo-url>"
          digest: {sha1: <commit-sha>}
      resolvedDependencies: [...]
    runDetails:
      builder:
        id: <ci-engine-url>
      metadata:
        invocationId: <ci-run-id>
        startedOn: <timestamp>
        finishedOn: <timestamp>
```

Explain: `buildType` is Pathfinder-specific so Pathfinder can identify its own attestations. `externalParameters` captures exactly the inputs that determine the build. The `source.digest.sha1` must match the Build record's `commit_sha`.

**Section: CI Step -- Sign and Attest**
Describe the batteries-included CI step for the steps library (conceptual, not full YAML):
- Step name: "Sign and Attest"
- Phase: `package` (runs after build, before notification)
- Location in steps repo: `security/sign-attest/action.yml`
- x-pathfinder metadata: runtimes `"*": "*"` (runtime-agnostic -- works with any build)
- Inputs: `image_ref` (required), `kms_key` (optional -- keyless if omitted), `rekor_upload` (default: false)
- Actions: Install Cosign, generate provenance predicate JSON, sign image, attach provenance attestation
- Requires `COSIGN_KEY` or OIDC token for keyless signing

**Section: SBOM Attestation**
Define SBOM as a companion attestation alongside provenance (per locked decision -- "SBOM designed in"):
- Format: CycloneDX JSON (recommended over SPDX for security/vulnerability context -- native VEX support, Cosign `--type cyclonedx` compatibility)
- Generation: Syft or Trivy in a separate CI step (or combined with sign-and-attest step)
- Attached to the same OCI image as a separate attestation: `cosign attest --type cyclonedx --predicate sbom.json`
- Pathfinder can index SBOM metadata (component count, vulnerability summary) but full SBOM analysis is out of scope
- SPDX is not excluded -- teams needing it can add a second generation step

**Section: Pathfinder Verification Flow**
Define the two verification points (per locked decision):

**Point 1: Build Ingestion (after build webhook)**
1. Build record created from CI webhook (existing flow)
2. Pathfinder fetches attestation from OCI registry: `cosign verify-attestation --type slsaprovenance1 --key <public-key> <image-ref>`
3. Parse in-toto statement, extract provenance predicate
4. Verify `buildType` matches `https://pathfinder.dev/ci-workflow/v1`
5. Verify `source.digest.sha1` matches `Build.commit_sha`
6. Store on Build record: `attestation_verified` (bool), `attestation_digest` (string), `signer_identity` (string), `provenance_bundle_ref` (string), `timestamp_verified` (bool)
7. If attestation missing or invalid: `attestation_verified = false` (does not block build recording, only blocks production deploy)

**Point 2: Deploy Time (before deployment execution)**
1. If target environment has attestation verification enabled (opt-in per environment, enabled by default):
2. Re-verify attestation on the specific image being deployed (not cached result)
3. If verification fails: deployment blocked with clear error message
4. This prevents TOCTOU attacks -- attestation could have been revoked between build and deploy

**Section: Environment Configuration**
Describe the per-environment opt-in (per locked decision):
- `require_attestation_verification`: boolean, default `true`
- When enabled: deployments must have `attestation_verified = true` on the Build record AND pass re-verification at deploy time
- When disabled: deployments proceed without attestation check (useful for non-production, development environments)
- Recommended: enabled for production and staging, disabled for development

**Section: Build Model Extensions**
List the new fields needed on the Build model (design-level, not Django code):
```
Build (extensions):
  - attestation_verified: bool (default false)
  - attestation_digest: string (nullable -- SHA256 of the attestation payload)
  - signer_identity: string (nullable -- key ID or OIDC subject)
  - provenance_bundle_ref: string (nullable -- OCI reference to attestation)
  - timestamp_verified: bool (default false)
  - sbom_attached: bool (default false)
```
These complement the existing `verification_status` (Verified/Draft/Unauthorized) from build authorization.

**Section: Supply Chain Trust Model**
Address the discretion item about CI steps library verification:
- Trust the steps repository with branch protection (do NOT sign individual steps)
- Branch protection (required reviews, no force push) is enforced at steps repository registration time (existing mechanism per docs/ci-workflows/steps-catalog.md)
- The trust chain: branch protection -> authorized workflow version -> manifest hash -> artifact attestation
- Each layer verifies the output of the previous layer; signing individual steps would add complexity without meaningful security benefit

**Section: Addressing "Artifact Signing Absent" Finding**
Cross-reference docs/deployments/review.md finding:
- Review identified: no cryptographic guarantee that the artifact deployed to production is the same one that passed tests in staging
- This design addresses it: every artifact gets a signed attestation at build time, re-verified before every deployment
- The promotion chain now has cryptographic integrity: build -> sign with Cosign -> attach SLSA provenance -> verify at deploy time
