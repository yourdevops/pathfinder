**File 1: docs/deployments/plugin-interface.md**

Create following the pattern of `docs/ci-workflows/plugin-interface.md`.

**Opening paragraph:** Deploy plugins implement the `DeployCapableMixin` to register deployment methods with Pathfinder. This is analogous to `CICapableMixin` for CI engines (see [CI Plugin Interface](../ci-workflows/plugin-interface.md)).

**Section: DeployCapableMixin** — Pseudocode interface:

```python
class DeployCapableMixin:
    def deploy_methods(self) -> list[DeployMethod]:
        """Return supported deployment methods."""

    def execute_deploy(
        self, config: dict, method: str, artifact_ref: str,
        env_vars: dict, deploy_config: dict
    ) -> DeployResult:
        """Execute a deployment. Returns result with status and metadata."""

    def check_health(self, config: dict, deploy_ref: str) -> HealthResult:
        """Check deployment health. Returns pass/fail with details."""

    def get_deploy_status(self, config: dict, deploy_ref: str) -> str:
        """Poll current deployment status."""
```

**Section: DeployMethod Registration** — How a plugin declares what methods it supports:

```python
class DeployMethod:
    name: str           # e.g., "docker-direct"
    executor: str       # "platform" or "gitops"
    artifact_types: list[str]  # e.g., ["container-image"]
    config_schema: dict # JSON schema for deploy_config
```

**Section: Docker Plugin Implementation (MVP)** — Example showing how DockerPlugin implements DeployCapableMixin. Docker direct method: pull image, create/start container, bind env vars, expose ports. Show pseudocode for `execute_deploy` and `check_health` methods.

**Section: Future Plugin Examples** — Brief (3-5 lines each) showing how K8s and GitOps plugins would implement the same interface. Mark as "Future" — design-contract only.

**File 2: docs/deployments/environment-binding.md**

**Opening paragraph:** When a deployment is triggered, Pathfinder resolves the target environment's deploy connection and builds a configuration snapshot. This document covers how connections are selected and how environment variables cascade into the deploy-time snapshot.

**Section: Connection Selection** — How Pathfinder determines which deploy connection to use:
- Environment has connections grouped by plugin type (see [environments.md](../environments.md)).
- Each plugin type has a default connection within the environment.
- The deploy method's plugin type determines which connection is used.
- If no compatible connection exists, the deployment is blocked.

**Section: Environment Variable Cascade** — Cross-reference to existing cascade docs, then specify deployment-specific behavior:
- Cascade order: Project -> Environment -> Service -> (deployment snapshot)
- Per locked decision: "Snapshot at deploy time. Env vars frozen when deployment starts. Changing vars requires re-deploy. Predictable and auditable."
- The snapshot is the FULL merged result, stored on the Deployment record as `env_vars_snapshot`.
- Locked vars from upstream levels are included in the snapshot as-is (cannot be overridden).
- The snapshot is immutable — it represents exactly what was deployed.

Edge cases:
- If a project-level locked var changes between deployments, the new deployment gets the new value. Old deployments retain their snapshot.
- If an environment var is deleted between deployments, new deployments won't have it. Old deployments retain their snapshot.

**Section: Secrets** — Brief acknowledgment (NOT a full design):
- Pathfinder stores configuration values, not secrets.
- Secrets must come from external sources (Vault, K8s Secrets, cloud secret managers).
- Deploy plugins may resolve secret references at execution time (e.g., Docker plugin reads from Vault, K8s plugin creates ExternalSecret CR).
- Designing a secrets management system is out of scope for MVP.

**Section: Resources** — Brief acknowledgment (NOT a full design):
- Environment resources (databases, caches, queues) are a future concern.
- Current model: resources are provisioned externally, their outputs registered as environment variables.
- No resource provisioning or binding system in MVP.

Target length: 50-70 lines. Cross-reference environments.md for connection model and projects.md for env var merge rules.
