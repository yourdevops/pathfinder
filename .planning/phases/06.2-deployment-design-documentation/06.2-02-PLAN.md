Create `docs/deployments/deployment-lifecycle.md` following the pattern of `docs/ci-workflows/build-lifecycle.md`.

**Opening paragraph:** A deployment is triggered by a successful build (auto-deploy to default environment) or by a user action (manual promotion to another environment). Pathfinder creates a Deployment record, resolves the target environment's deploy connection, snapshots configuration, and delegates execution to the deploy plugin.

**Section: Triggering a Deployment** — Two trigger paths:

1. **Auto-deploy (build success):** When a build completes with status `success` and verification_status `verified` (or `draft` for non-production), Pathfinder automatically creates a deployment to the project's default environment. No human in the loop. Per locked decision: "merge to main -> build succeeds -> deploy starts immediately."
   - Edge case: If default environment has no compatible deploy connection, auto-deploy is skipped with a warning.
   - Edge case: If a deployment to (service, default-environment) is already running, the new deployment enters `pending` and waits.

2. **Manual deploy (promotion):** User triggers deployment from UI, selecting environment and build. See [Promotion](promotion.md) for the full promotion workflow.

**Section: Deployment Status State Machine** — ASCII art following `docs/ci-workflows/versioning.md` pattern:

```
PENDING (deployment created, queued)
  |
  |-- start --> RUNNING (plugin executing deployment)
  |                |
  |                |-- health_check --> HEALTH_CHECK (waiting for health verification)
  |                |                      |
  |                |                      |-- pass --> SUCCESS (deployment complete)
  |                |                      |
  |                |                      +-- fail --> FAILED (health check failed)
  |                |
  |                +-- error --> FAILED (execution error)
  |
  +-- cancel --> CANCELLED (before execution started)
```

Per locked decision: "Real-time updates — live status progression in UI (pending -> running -> health check -> success/failed), similar to existing build tracking pattern."

**Section: Deployment Record Model** — Pseudocode following build-lifecycle.md pattern:

```python
class Deployment:
    service = FK(Service)
    build = FK(Build)
    environment = FK(Environment)
    deploy_connection = FK(EnvironmentConnection)

    class Status(TextChoices):
        PENDING = "pending"
        RUNNING = "running"
        HEALTH_CHECK = "health_check"
        SUCCESS = "success"
        FAILED = "failed"
        CANCELLED = "cancelled"

    status = CharField(choices=Status.choices)
    artifact_ref = CharField()        # snapshot from build
    env_vars_snapshot = JSONField()    # frozen at deploy time
    error_message = TextField()

    triggered_by = CharField()        # "auto" or username
    created_at = DateTimeField()
    started_at = DateTimeField()
    completed_at = DateTimeField()
```

Per locked decision: "Env vars frozen when deployment starts. Changing vars requires re-deploy."

Explain `env_vars_snapshot`: The full merged env var cascade (Project -> Environment -> Service) is resolved and frozen at deployment creation time. The snapshot is stored on the Deployment record for auditability. Subsequent changes to upstream vars do not affect running or completed deployments.

**Section: Concurrent Deployment Rules**

Per locked decision: "Services are independent — concurrent deploys to different services in same environment are fine. Same service+environment blocked (only one active deployment at a time)."

- Only one non-terminal deployment (pending, running, health_check) per (service, environment) pair.
- Concurrent deployments to different services in the same environment proceed independently.
- If a deployment is pending and a new deployment is requested for the same (service, environment), the request is rejected until the current deployment reaches a terminal state.

**Section: Failure Handling**

Per locked decision: "Alert + manual action. Mark deployment as failed, notify user. User decides whether to rollback, fix-forward, or investigate. No auto-rollback."

- On failure: status transitions to FAILED, error_message is populated, user is notified.
- No automatic retry or rollback. User manually decides the next action.

Create `docs/deployments/promotion.md`. This document has no direct CI workflows analogue — promotions are unique to the deployment domain.

**Opening paragraph:** Promotion is the process of deploying an artifact that was already deployed to one environment into another environment. Pathfinder follows trunk-based development: code on main is treated as production-ready. The first deployment is always automatic to the project's default environment. Subsequent deployments to other environments are manual promotions.

**Section: Auto-Deploy to Default Environment**

Per locked decision: "Fully automatic — merge to main -> build succeeds -> deploy starts immediately, no human in the loop."

- Trigger: Build with status `success` and appropriate verification_status completes.
- Target: Project's default environment (see [environments.md](../environments.md) for default environment rules).
- The auto-deploy creates a Deployment record with `triggered_by="auto"`.
- Build verification gate: Only `verified` builds auto-deploy. `draft` builds auto-deploy only if the project has "Allow Drafts" enabled AND the default environment is non-production.
- `unauthorized` builds never auto-deploy.

**Section: Manual Promotion**

Per locked decision: "Role-based only — anyone with the right project role can promote. No separate approval workflow."

- **Non-production environments:** Contributor, Owner, or Admin can promote.
- **Production environments:** Only Owner or Admin can promote.
- Promotion reuses the artifact from the source build. No rebuild.

Per locked decision (rollback): "Re-deploy previous known-good artifact. Simple, artifact-focused. No git-revert model."

- Rollback = manual promotion of a previous known-good build to the same environment.
- No special "rollback" action. User selects a previous successful build and promotes it.
- The new deployment creates a fresh Deployment record (not modifying the failed one).

**Section: Environment Ordering**

Per locked decision: "Recommended by environment priority but not enforced. First deployment MUST go to the project's default environment."

- Environments have an `order` field (e.g., dev=10, staging=20, prod=30).
- The UI presents environments in order, suggesting the promotion path.
- The ordering is NOT enforced — a user can skip staging and promote directly to production if they have the required role.
- **Exception:** The first deployment of a service MUST go to the default environment (via auto-deploy on build success). After that, any environment in any order.
- This aligns with the recommendation-only approach used by GitOps platforms (ArgoCD, Flux) rather than the sequential enforcement of tools like Spinnaker.

**Section: Promotion Workflow Summary**

Concise numbered steps:
1. Build succeeds -> auto-deploy to default environment (automatic)
2. User views deployment history, selects a successful build
3. User selects target environment from the promotion UI
4. Pathfinder checks role authorization (contributor for non-prod, owner for prod)
5. Pathfinder creates new Deployment record with env var snapshot for target environment
6. Deploy plugin executes in target environment

**Section: Artifact Reuse**

Per existing `docs/deployments.md`: "Promotions to other Environments are manual via Pathfinder UI/API and use reusable artifacts from the original build."

- Promotion reuses the artifact_ref from the source build. The artifact is NOT rebuilt.
- If the artifact type is incompatible with the target environment's deploy connection, the promotion is blocked with an explanation (see [deployment-methods.md](deployment-methods.md) for artifact matching).
- A tag with release/version is pushed on the original commit that produced the artifact.

Target length: 60-90 lines. Cross-reference deployment-lifecycle.md for trigger details and deployment record model. Cross-reference deployment-methods.md for artifact matching.
