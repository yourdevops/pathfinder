## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Promotion & Workflow

- **Auto-deploy to default environment**: Fully automatic -- merge to main -> build succeeds -> deploy starts immediately, no human in the loop
- **Promotion approval model**: Role-based only -- anyone with the right project role can promote (owner for production, contributor for non-production). No separate approval workflow.
- **Rollback strategy**: Re-deploy previous known-good artifact. Simple, artifact-focused. No git-revert model.
- **Promotion path ordering**: Recommended by environment priority but not enforced. First deployment MUST go to the project's default environment. Research industry standards for whether sequential enforcement is common or whether recommendation-only is the norm.
- **Failure handling**: Alert + manual action. Mark deployment as failed, notify user. User decides whether to rollback, fix-forward, or investigate. No auto-rollback.
- **Deployment status tracking**: Real-time updates -- live status progression in UI (pending -> running -> health check -> success/failed), similar to existing build tracking pattern.
- **Environment variable binding**: Snapshot at deploy time. Env vars frozen when deployment starts. Changing vars requires re-deploy. Predictable and auditable.
- **Concurrent deployments**: No locking needed across services. Services are independent -- concurrent deploys to different services in same environment are fine. Same service+environment blocked (only one active deployment at a time).

## Architecture Patterns

### Recommended Document Structure

```
docs/deployments/
  README.md                # Overview, boundaries, index, quick reference tables
  deployment-methods.md    # Method types, executor model, artifact matching
  deployment-lifecycle.md  # Trigger -> execute -> health check -> complete
  promotion.md             # Auto-deploy, manual promotion, rollback, ordering
  plugin-interface.md      # DeployCapableMixin, method registration, config
  environment-binding.md   # Connection selection, env var cascade, snapshots
  logging.md               # Audited entities, deployment operation logs
```

**Rationale for this specific breakdown:**

1. **`README.md`** -- Mirrors CI workflows README. Contains "What Deployments Do Not Do" section, documentation index, and deployment status quick-reference table. This is the entry point that orients the reader.

2. **`deployment-methods.md`** -- Covers the three deployment method tiers (Direct/Platform-executed, GitOps, CI-managed) from `docs/deployments/initial.md` plus artifact-to-deployment matching from `docs/deployments.md`. This is analogous to `workflow-definition.md` -- it defines what a deployment method IS and what types exist.

3. **`deployment-lifecycle.md`** -- Covers the full flow from trigger to completion: auto-deploy on build success, manual trigger via UI, status progression (pending -> running -> health_check -> success/failed), concurrent deployment blocking, and the Deployment record model. Analogous to `build-lifecycle.md`.

4. **`promotion.md`** -- Covers promotion workflow, rollback strategy, environment ordering, and the auto-deploy-to-default rule. This is deployment-specific content that has no direct CI workflows analogue -- promotions are the core differentiator of the deployment domain.

5. **`plugin-interface.md`** -- Covers the deploy plugin contract (like `CICapableMixin` but for deployments), method registration, how connections bind to environments, and config override. Analogous to CI workflows `plugin-interface.md`.

6. **`environment-binding.md`** -- Covers how a deployment resolves its target: connection selection, env var cascade (Project -> Environment -> Service -> Deployment snapshot), and secrets references. This consolidates environment-related content from `docs/environments.md` and `docs/deployments/initial.md`.

7. **`logging.md`** -- Covers deployment-specific audit entities and operation log models. Follows the pattern of `docs/ci-workflows/logging.md` which references `docs/logging.md` for the core logging framework.

### Why 6 Documents (Not More, Not Fewer)

**Not fewer:** Combining lifecycle + promotion into one doc would produce an overly long document covering two conceptually distinct topics (how a deployment runs vs. how artifacts move between environments). The CI workflows docs establish a precedent of focused documents.

**Not more:** Splitting "deployment methods" into separate docs per method (one for Docker, one for K8s, one for GitOps) would fragment closely related content. Method types share the same executor model and artifact matching logic -- they differ only in implementation details, which belong in `plugin-interface.md`.

### Anti-Patterns to Avoid

- **Implementation leakage:** These are design docs, not implementation guides. Do not include Django model code, view code, or template code. Use pseudocode models (like CI workflows docs do) that describe WHAT the model contains, not HOW it's implemented.

- **Duplicating existing docs:** `docs/environments.md`, `docs/integrations.md`, and `docs/services.md` already define environment connections, env var cascade, and service models. Deployment docs should cross-reference these, not copy them. Only deployment-SPECIFIC aspects of these topics belong in deployment docs.

- **Scope creep into Phase 7:** The docs describe the design. They do not specify Django URL patterns, template layouts, or HTMX interaction patterns. Those are Phase 7 implementation decisions.

- **Over-specifying future methods:** Docker direct is the MVP deployment method. K8s, GitOps, and CI-managed should be documented at the design level (how they fit the model) but not at implementation depth. The plugin interface doc establishes the contract; individual plugin implementations are Phase 7+ concerns.

## Common Pitfalls

### Pitfall 1: Conflating Design Docs with Implementation Specs
**What goes wrong:** Design docs start specifying Django URLs, template names, or specific API response formats, blurring the line with Phase 7 implementation planning.
**Why it happens:** The temptation to be "thorough" leads to premature implementation decisions.
**How to avoid:** Each doc should answer "what does the system do and why?" not "how is it built?" The CI workflows docs are the bar -- they describe models with pseudocode, not Django ORM definitions.
**Warning signs:** References to specific template files, URL paths, or HTMX patterns.

### Pitfall 2: Ignoring Existing Material
**What goes wrong:** New docs are written from scratch, contradicting or duplicating content in `docs/deployments.md`, `docs/deployments/initial.md`, `docs/services.md`, or `docs/environments.md`.
**Why it happens:** Easier to write fresh than to carefully incorporate and restructure existing material.
**How to avoid:** The first task in any document should be identifying what existing material applies and how to restructure it. The existing `docs/deployments.md` has 70 lines of carefully considered design -- it should be decomposed into the new structure, not rewritten.
**Warning signs:** Contradictions between new deployment docs and existing docs. Deployment model in new docs differs from `docs/services.md` Deployment Model section.

### Pitfall 3: Under-Specifying Locked Decisions
**What goes wrong:** A locked decision like "env vars snapshot at deploy time" gets a single sentence instead of a clear specification with edge cases.
**Why it happens:** The decision seems simple, but implementers need to know: what happens to locked vars? What if project vars change between deployments? How is the snapshot stored?
**How to avoid:** Each locked decision should have its own subsection with: the rule, why it exists, edge cases, and a concrete example.
**Warning signs:** A locked decision covered in fewer than 3 sentences.

### Pitfall 4: Missing Boundary Between MVP and Future
**What goes wrong:** Docs describe K8s GitOps deployments at the same depth as Docker direct, even though only Docker is MVP.
**Why it happens:** The design domain is broad and interesting.
**How to avoid:** README's quick reference or each document's header should clearly mark what's MVP scope vs. future. Docker direct = full specification depth. Others = design-level coverage showing how they fit the model.
**Warning signs:** 200+ lines on ArgoCD Application CRs when Docker direct has 50 lines.

### Pitfall 5: Inconsistent Document Conventions
**What goes wrong:** Some docs use different table formats, heading levels, or notation than the CI workflows reference.
**Why it happens:** Different docs written in different sessions with drifting style.
**How to avoid:** Establish document conventions explicitly in the plan. Use CI workflows docs as the style guide. Every doc should have the same heading structure: title, overview paragraph, sections with tables, cross-references at the bottom.
**Warning signs:** One doc uses `###` for tables of contents while another uses `##`. One doc has a "See also" section while others have inline links.

## Findings: Promotion Path Ordering

The user locked decision states: "Recommended by environment priority but not enforced. First deployment MUST go to the project's default environment. Research industry standards for whether sequential enforcement is common or whether recommendation-only is the norm."

### Industry Analysis (MEDIUM confidence)

**Sequential enforcement IS common in mature tools:**
- Spinnaker provides environment constraints that ensure an artifact cannot be promoted to Environment B unless it has successfully deployed to Environment A. This is a core feature, not an afterthought.
- ArgoCD does NOT have built-in sequential promotion. Teams implement it via notification hooks (on staging success, auto-commit to prod values), but it requires custom automation.
- Kubernetes-native tools (Flux, ArgoCD) generally do not enforce ordering -- they treat each environment as independent.

**The split in the industry:**
- **Enforcement tools** (Spinnaker, Harness, Octopus Deploy): Built-in sequential gates. Common in regulated industries and large enterprises.
- **Recommendation tools** (ArgoCD, Flux, most GitOps platforms): No built-in enforcement. Ordering is a convention, not a constraint.
- **Platform engineering trend (2025-2026)**: Moving toward automated, policy-driven enforcement at the platform level. But this is "Policy-as-Code" (OPA, Kyverno) rather than hard-coded sequential gates.

**Recommendation for the design doc:** The locked decision (recommended, not enforced) aligns with the GitOps/recommendation camp. The first-deployment-to-default rule provides a minimal safety gate without the complexity of full sequential enforcement. Document this as a deliberate design choice: Pathfinder recommends promotion order via UI indicators (environment `order` field) and enforces only the first-deployment constraint. This matches the developer self-service philosophy where guardrails guide rather than block.

## Findings: Resource Model and Secrets Depth

Per "Claude's Discretion" -- determine appropriate level of detail for environment resources and secrets.

**Recommendation:** Minimal depth. Resources and secrets are explicitly out of MVP scope per `docs/deployments/initial.md` ("Tier 1 -- Manual registration (build first)" is the starting point) and per REQUIREMENTS.md (no DPLY requirement for resources or secrets management). The deployment docs should:

1. Acknowledge that resources and secrets exist as future concerns
2. Document the current env var cascade as the mechanism for configuration
3. Note that secrets must come from external sources (consistent with `docs/services.md`: "Only configuration values are supported. Secrets must come from external sources")
4. Not design a resource binding or secrets management system

This keeps the docs focused on what Phase 7 will actually implement.

1. **Should `docs/deployments.md` be replaced or kept?**
   - What we know: The new `docs/deployments/README.md` will cover all content from `deployments.md` in a better-organized form
   - What's unclear: Whether to delete the old file or keep it as a redirect

2. **Should `docs/services.md` Deployment Model section be updated?**
   - What we know: The services.md Deployment Model predates the locked decisions and has some misalignments
   - What's unclear: Whether updating services.md is in scope for Phase 6.2 (documentation phase)
   - Recommendation: Update `docs/services.md` Deployment Model to align with the new deployment docs. This is within scope for a documentation phase and prevents contradictions.

3. **Deployment status: include `health_check` as a separate state?**
   - What we know: Locked decision says "pending -> running -> health check -> success/failed". Existing `docs/services.md` has no health_check state.
   - What's unclear: Whether health_check is a distinct database state or a sub-phase of "running"
   - Recommendation: Document `health_check` as a distinct state. It provides meaningful UI feedback ("your container is running, now checking if it's healthy") and matches the locked decision literally. The implementation can decide whether it's a DB state or UI-only.
