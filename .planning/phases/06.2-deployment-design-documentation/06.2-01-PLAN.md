Create `docs/deployments/README.md` following the exact structure of `docs/ci-workflows/README.md`.

**Opening paragraph:** Define what a Deployment is in Pathfinder. A Deployment is a tracked operation that places a built artifact into a target environment. Pathfinder controls the full chain: credentials, execution, status, and audit trail. Deployments are triggered automatically on build success (to default environment) or manually via promotion.

**"What Deployments Do Not Do" section** (mirrors CI workflows boundary-setting):
- Manage CI secrets or build pipelines (that is a CI Workflow concern)
- Provision infrastructure (databases, networks, queues)
- Manage DNS or load balancer configuration
- Provide a container runtime or compute plane
- Replace existing CD systems (CI-managed deployments coexist)
- Auto-rollback on failure (user decides next action)
- Manage service secrets (secrets come from external sources)

1. "Deployment Status States" table (mirrors Build Verification States table):
| State | Meaning | Trigger |
|-------|---------|---------|

| **Health Check** | Waiting for health verification | Execution complete |

**Opening paragraph:** Pathfinder supports multiple deployment methods through the plugin system. Each deploy plugin registers a Connection and one or more Deployment Methods. Methods differ by executor type and depth of Pathfinder control.

**Section: Executor Types** (from existing deployments.md, expanded):
Table with two types:
| Executor | Who Does the Work | Pathfinder Role | Example |
|----------|-------------------|-----------------|---------|
| `platform` | Pathfinder worker calls target API | Full control: credentials, execution, status, rollback | Docker direct, K8s direct, Helm |
| `gitops` | External controller syncs from repo | Partial control: generates manifests, pushes to repo | K8s gitops, ArgoCD |

**Section: Method Types** with three subsections:

1. **Direct (Platform-Executed)** — Full specification depth for Docker. Pathfinder calls target APIs directly using Connection credentials. Docker: generate compose/run config, deploy via socket/TCP. K8s: generate manifests, apply via K8s API (design-contract depth only, mark as "Future").

2. **GitOps (Split Responsibility)** — Design-contract depth only. Pathfinder generates manifests and pushes to a git repo. External controller reconciles. Mark as "Future" — describe the pattern and how it fits the model, not implementation detail.
