# Phase 06.5: Workflow and Build Model Hardening - Research

**Researched:** 2026-02-16
**Domain:** Django model changes, view logic, template updates, plugin interface refactoring
**Confidence:** HIGH

## Summary

This phase addresses six gaps (GAP-08, GAP-09, GAP-10, GAP-14, GAP-15, GAP-16) from the REMEDIATION.md document. The work splits cleanly into two logical groups: (1) CIWorkflow hardening (explicit engine, step ordering validation, archived status) and (2) Build model hardening (engine-agnostic field rename, revoked verification status, manifest_id categorization, status mapping moved to plugin).

The codebase is well-structured for these changes. All existing CI data has been deleted per REMEDIATION.md, so schema changes apply to empty tables -- no data migrations needed, just `makemigrations`. The changes touch `core/models.py`, `core/tasks.py`, `core/views/ci_workflows.py`, `core/views/services.py`, `plugins/base.py`, `plugins/github/plugin.py`, and several templates.

**Primary recommendation:** Execute as two plans -- Plan 1 for CIWorkflow changes (engine field, ordering validation, archived status), Plan 2 for Build model changes (ci_run_id rename, revoked status, manifest_id categorization, map_run_status plugin interface).

## Architecture Patterns

### Pattern 1: Engine Derivation (Current Anti-Pattern)

**What:** The engine is currently derived from the first step's engine in at least 5 locations:
- `core/tasks.py:538` (`verify_build`) -- `first_step.step.engine if first_step else "github_actions"`
- `core/tasks.py:751-752` (`push_ci_manifest`) -- same pattern
- `core/views/ci_workflows.py:572-573` (`WorkflowComposerView.post`) -- same pattern
- `core/views/ci_workflows.py:666` (`WorkflowDetailView.get`) -- same pattern
- `core/views/ci_workflows.py:730-731` (`WorkflowManifestView.get`) -- same pattern
- `core/views/ci_workflows.py:825-826` (`PublishVersionView.post`) -- same pattern
- `core/views/ci_workflows.py:939` (`ForkWorkflowView.post`) -- same pattern
- `core/views/services.py:425-426` (`ServiceDetailView.get_context_data` CI tab) -- same pattern

**Fix:** Add `CIWorkflow.engine` CharField, set at creation in `WorkflowComposerView.post()`. Replace ALL derivation sites with `workflow.engine`.

### Pattern 2: Build Categorization (Current Implementation)

**What:** `ServiceDetailView` builds tab (line 449-451) categorizes by `workflow_name` string matching:
```python

other_builds_qs = all_builds.exclude(workflow_name=current_workflow_name)
```

**Fix:** Replace with `manifest_id` categorization per build-authorization.md:

### Pattern 3: Verification Status Mapping (Current Implementation)

**What:** `verify_build` task (line 576-577) maps revoked versions to "unauthorized":
```python
else:  # revoked
    verification_status = "unauthorized"
```

**Fix:** Map to `"revoked"` instead of `"unauthorized"`.

### Anti-Patterns to Avoid
- **Retroactive validation:** Step ordering validation must NOT retroactively reject existing workflows. Validate only on save (POST), not on load (GET).
- **Hardcoded fallback:** After adding `CIWorkflow.engine`, do NOT leave fallback to `"github_actions"` in the derivation sites. Use `workflow.engine` exclusively.
- **Template duplication for badges:** Use a consistent badge pattern for the new "revoked" verification status. Do not invent new styling -- follow the existing verified/draft/unauthorized badge patterns.

### GAP-09: Step Ordering Validation

**Location (`core/views/ci_workflows.py` line 509-569):** In `WorkflowComposerView.post()`, after parsing `steps_data`, before creating the workflow or saving steps.

**Validation logic:**
```python
def validate_step_ordering(steps_data):
    """Validate setup-before-use rule.

    For each step with runtime_constraints, verify a setup step
    for the same runtime appears earlier in the list.
    """
    provided_runtimes = set()
    errors = []

    for i, step_entry in enumerate(steps_data):
        ci_step = CIStep.objects.get(uuid=step_entry.get("id"))

        # Setup steps provide runtimes
        if ci_step.phase == "setup":
            for runtime_name in (ci_step.runtime_constraints or {}):
                provided_runtimes.add(runtime_name)

        # Non-setup steps with runtime constraints need prior setup
        if ci_step.phase != "setup" and ci_step.runtime_constraints:
            for runtime_name in ci_step.runtime_constraints:
                if runtime_name not in provided_runtimes:
                    errors.append(
                        f'Step "{ci_step.name}" requires {runtime_name} runtime, '
                        f'but no setup step for {runtime_name} appears before it.'
                    )
    return errors
```

Note: Steps with `*` as runtime or empty runtime_constraints skip validation. The `is_step_compatible` function in `core/ci_manifest.py` already handles runtime constraint checking -- reuse its patterns.

**Error display:** Return the composer template with an error message. Use Django's `messages.error()` framework (already used in this view).

```python
def map_run_status(self, status: str, conclusion: str | None) -> str:
    if status in ("queued", "in_progress"):
        return "running"
    if status == "completed" and conclusion:
        conclusion_map = {
            "success": "success",
            "failure": "failed",
            "cancelled": "cancelled",

        return conclusion_map.get(conclusion, "pending")
    return "pending"
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Field rename migration | Manual SQL ALTER TABLE | Django `RenameField` via `makemigrations` | Django handles column rename correctly for SQLite |
| Choice field expansion | Custom migration | Just add the choice to the choices list + `makemigrations` | Django choice changes create simple migrations |
| Step ordering validation | Complex graph solver | Simple linear scan tracking provided runtimes | The rule is "setup-before-use", not a DAG -- linear scan is sufficient |

**Key insight:** All schema changes are simple field additions, renames, or choice list expansions. Django's migration system handles all of these. Since tables are empty, no data migration logic is needed.

## Common Pitfalls

### Pitfall 1: Missing engine field on edit
**What goes wrong:** The composer edit path might not set the engine correctly when saving
**Why it happens:** Edit mode updates an existing workflow -- the engine should NOT be changed on edit
**How to avoid:** In `WorkflowComposerView.post()`, only set `engine` on create (when `workflow_uuid` is empty). On edit, do not touch the `engine` field.
**Warning signs:** Workflow engine changes after editing steps

### Pitfall 2: Forgetting to propagate engine field to select_related
**What goes wrong:** N+1 queries when accessing `workflow.engine` in loops
**Why it happens:** Adding a direct field does not cause N+1 (it is already on the model). However, removing `select_related("step")` calls when engine derivation is removed could cause issues if those select_related calls were needed for other fields.
**How to avoid:** When removing engine derivation code, audit the surrounding select_related calls. Only remove the ones that were solely for engine derivation.

### Pitfall 3: SQLite RENAME COLUMN gotcha
**What goes wrong:** Django's `RenameField` on SQLite creates a new table, copies data, and drops the old one
**Why it happens:** SQLite's ALTER TABLE is limited
**How to avoid:** This is fine since the Build table is empty per REMEDIATION.md. Just run `makemigrations` and it will work.

### Pitfall 4: Template caching with new verification status
**What goes wrong:** "revoked" builds show as "unauthorized" in the UI
**Why it happens:** The template has no branch for "revoked" -- falls through to the else/default
**How to avoid:** Add the `{% elif build.verification_status == "revoked" %}` block in the template BEFORE the unauthorized check.

### Pitfall 5: Archived workflow still usable by existing services
**What goes wrong:** Archived workflows break existing services
**Why it happens:** Filtering too aggressively -- excluding archived workflows from all queries
**How to avoid:** Only filter out archived in `get_available_workflows_for_project()` and the new service wizard. Do NOT filter in `ServiceDetailView` CI tab, version management, or manifest push.

### Pitfall 6: Build categorization null manifest_id
**What goes wrong:** Builds created before the manifest_id field was populated have `manifest_id=""`, causing all historical builds to show under "Other"
**Why it happens:** Older builds may not have `manifest_id` set
**How to avoid:** Since all CI data was deleted per REMEDIATION.md, this should not be an issue. But as a safeguard, fall back to `workflow_name` categorization when `manifest_id` is empty.

### Step Ordering Validation in Composer

```python
# Add after steps_data parsing, before workflow create/update
def _validate_step_order(steps_data):
    """Validate setup-before-use rule for workflow steps."""
    provided_runtimes = set()
    errors = []

    for step_entry in steps_data:
        try:
            ci_step = CIStep.objects.get(uuid=step_entry.get("id"))
        except CIStep.DoesNotExist:
            continue

        if ci_step.phase == "setup":
            for rt in (ci_step.runtime_constraints or {}):
                provided_runtimes.add(rt)
        elif ci_step.runtime_constraints:
            for rt in ci_step.runtime_constraints:
                if rt != "*" and rt not in provided_runtimes:
                    errors.append(
                        f'"{ci_step.name}" requires {rt} runtime setup. '
                        f'Add a setup step for {rt} before this step.'
                    )
    return errors
```

### Archive/Unarchive View

```python
class WorkflowArchiveView(OperatorRequiredMixin, View):
    """Toggle workflow archived status."""

    def post(self, request, workflow_name):
        workflow = get_object_or_404(CIWorkflow, name=workflow_name)
        if workflow.status == "archived":
            workflow.status = "published"
            messages.success(request, f'Workflow "{workflow.name}" restored.')
        else:
            workflow.status = "archived"
            messages.success(request, f'Workflow "{workflow.name}" archived.')
        workflow.save(update_fields=["status"])
        return redirect("ci_workflows:workflow_detail", workflow_name=workflow.name)
```

## Open Questions

1. **Should the WorkflowComposerView.post() pass engine via hidden input or derive from POST data?**
   - What we know: The `WorkflowCreateForm` already captures `engine` and passes it to the composer via URL params. The composer template would need a hidden `engine` input to POST it back.
   - Recommendation: Add a hidden `<input name="engine" value="{{ engine }}">` to the composer form. In edit mode, use `workflow.engine` for the hidden input value.

2. **Should archived workflows still show in the workflow list?**
   - What we know: REMEDIATION.md says "filter out archived workflows in the service wizard step and in `get_available_workflows_for_project()`". It does not explicitly say to hide from the workflow list.
   - Recommendation: Show archived workflows in the list with a visual "Archived" badge, but filter them out of service onboarding (wizard and available_workflows).

3. **What color for the "revoked" badge?**
   - What we know: The design doc groups revoked with unauthorized. The current template already has an amber/warning badge for when a verified build has a revoked version (lines 45-50).
   - Recommendation: Use orange (`bg-orange-500/20 text-orange-300`) to distinguish from unauthorized (red) and draft (amber). This signals "once valid, now revoked" rather than "never valid".
