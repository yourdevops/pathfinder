# Phase 4.1: Replace UUID URLs with Slugs - Research

**Researched:** 2026-01-26
**Domain:** Django URL routing, model validation, database constraints
**Confidence:** HIGH

## Summary

This phase replaces UUID-based URLs with human-readable name-based URLs across Project, Environment, Blueprint, Group, and Connection entities. The approach uses the existing `name` field directly in URLs (no separate slug field), with DNS-compatible validation (RFC 1123 label format).

Key findings:
- Django's built-in `slug` path converter is NOT suitable because it allows underscores and uppercase letters, while DNS-compatible names require lowercase only with hyphens
- A custom path converter is required to enforce DNS-compatible format in URLs
- Model-level validation using `RegexValidator` ensures data integrity at creation time
- The existing `unique_together` on Environment (project, name) is correct; other entities already have `unique=True` on name
- No data migration needed - context confirms existing data conforms to the required format

**Primary recommendation:** Create a custom `DnsLabelConverter` path converter and add `RegexValidator` to all name fields. Update URL patterns, views, and permission mixins to use name-based lookups.

## Architecture Patterns

### DNS-Compatible Name Validator Pattern

Create a reusable validator for DNS label format:

```python
# core/validators.py
import re
from django.core.validators import RegexValidator

# RFC 1123 label: lowercase letters, numbers, hyphens
# Can start with letter or digit (RFC 1123 relaxed RFC 952's letter-only requirement)
# Max 63 chars, no leading/trailing hyphens
DNS_LABEL_REGEX = r'^[a-z0-9][a-z0-9-]{0,61}[a-z0-9]$|^[a-z0-9]$'

dns_label_validator = RegexValidator(
    regex=DNS_LABEL_REGEX,
    message='Name must be DNS-compatible: lowercase letters, numbers, and hyphens only. '
            'Max 63 characters, no leading/trailing hyphens.',
    code='invalid_dns_label'
)
```

**Source:** Django validators documentation, RFC 1123 specification

### Custom URL Path Converter Pattern

Django's built-in `slug` converter matches `[-a-zA-Z0-9_]+` which allows uppercase and underscores. Create a custom converter:

```python
# core/converters.py
class DnsLabelConverter:
    """URL path converter for DNS-compatible names (RFC 1123 label format)."""
    regex = r'[a-z0-9][a-z0-9-]{0,61}[a-z0-9]|[a-z0-9]'

    def to_python(self, value):
        return value

    def to_url(self, value):
        return value
```

Register in urls.py:
```python
from django.urls import register_converter
from core.converters import DnsLabelConverter

register_converter(DnsLabelConverter, 'dns')
```

**Source:** [Django URL dispatcher documentation](https://docs.djangoproject.com/en/6.0/topics/http/urls/)

### URL Pattern Structure

**Before (UUID-based):**
```python
path('<uuid:project_uuid>/', ProjectDetailView.as_view(), name='detail'),
path('<uuid:project_uuid>/environments/<uuid:env_uuid>/', ...)
```

**After (name-based with custom converter):**
```python
path('<dns:project_name>/', ProjectDetailView.as_view(), name='detail'),
path('<dns:project_name>/environments/<dns:env_name>/', ...)
```

### Anti-Patterns to Avoid

- **Using built-in `slug` converter:** Allows uppercase and underscores, breaks DNS compatibility
- **Adding a separate slug field:** Redundant when `name` already serves this purpose
- **Auto-generating slugs:** Creates rename complexity; immutable names are simpler
- **Case-insensitive lookups:** Database collation varies; enforce lowercase at input

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| URL name validation | Custom regex in every view | Custom path converter | Centralized, automatic 404 on invalid format |
| Model name validation | Form-only validation | Model-level `RegexValidator` | Enforced at all entry points (forms, API, shell) |
| Uniqueness checking | Manual `exists()` queries | `unique=True` / `UniqueConstraint` | Database-enforced, race-condition safe |
| Error messages | Custom per-field messages | `violation_error_message` on constraint | Consistent, DRY |

**Key insight:** Model-level validation is critical because data can enter through forms, management commands, shell, or future APIs. Form validation alone is insufficient.

## Common Pitfalls

### Pitfall 1: Forgetting URL kwarg renames in templates

**What goes wrong:** Templates use old kwarg names (`project_uuid`) after URL patterns change to new names (`project_name`)
**Why it happens:** Mass find/replace misses template `{% url %}` tags
**How to avoid:**
1. Search for ALL occurrences: `project_uuid`, `env_uuid`, `group_uuid`, etc.
2. Run `python manage.py collectstatic` and test all navigation paths
**Warning signs:** `NoReverseMatch` errors at runtime

### Pitfall 2: Forgetting `reverse()` calls in Python code

**What goes wrong:** Python views using `reverse()` with old kwarg names
**Why it happens:** Views often construct redirect URLs dynamically
**How to avoid:** Grep for `reverse('projects:` and update all occurrences
**Warning signs:** `NoReverseMatch` errors on form submissions

### Pitfall 3: Environment lookup without project scoping

**What goes wrong:** Looking up environment by name alone, missing project scope
**Why it happens:** Environment names are only unique within their project
**How to avoid:** Always use `get_object_or_404(Environment, name=env_name, project=self.project)`
**Warning signs:** Wrong environment returned when same name exists in multiple projects

### Pitfall 4: Missing validator on model field

**What goes wrong:** Invalid names can be created via management commands or shell
**Why it happens:** Form validation doesn't cover all entry points
**How to avoid:** Add `RegexValidator` to model field definition, not just forms
**Warning signs:** Database contains non-DNS-compatible names

### Pitfall 5: Breaking existing bookmarks/links

**What goes wrong:** Users have bookmarked UUID-based URLs that now 404
**Why it happens:** No redirect from old URLs to new URLs
**How to avoid (CONTEXT decision):** This is explicitly acceptable - names are immutable, old UUID links are abandoned. No redirect middleware needed.
**Warning signs:** User reports of broken links (expected, acceptable)

### View with Name-Based Lookup

```python
# core/views/projects.py
class EnvironmentDetailView(LoginRequiredMixin, ProjectViewerMixin, TemplateView):
    def get_context_data(self, **kwargs):
        context = super().get_context_data(**kwargs)
        env = get_object_or_404(
            Environment,
            name=kwargs.get('env_name'),  # Changed from uuid
            project=self.project
        )
        context['environment'] = env
        return context
```

## State of the Art
