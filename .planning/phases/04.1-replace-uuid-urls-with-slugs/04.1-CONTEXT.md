# Phase 4.1: Replace UUID URLs with Slugs - Context

**Gathered:** 2026-01-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace UUID-based URLs with human-readable name-based URLs across all entities. Projects, Environments, Blueprints, Groups, and Connections use their existing `name` field in URLs (no separate slug field). User URLs retain UUIDs for privacy. Names are immutable after creation.

</domain>

<decisions>
## Implementation Decisions

### ID Strategy
- UUIDs remain as internal primary keys for database relationships
- **DO NOT add a slug field** — use the existing `name` field directly in URLs
- Foreign keys reference UUIDs, URLs reference `name`
- This keeps FK relations stable while URLs are human-readable

### Immutability
- All entity names are immutable after creation
- To "rename" an entity: delete and recreate
- No rename logic, no redirect handling, no reconciliation needed
- This matches infrastructure tool patterns (Terraform, Kubernetes)
- Rationale: Names propagate to deployment manifests, container names, DNS — renaming would require drift detection and reconciliation

### Name Field as Slug
- Format: `lowercase-with-dashes`
- Example: `my-cool-project`, `production`, `python-fastapi`
- **No separate slug field** — `name` field IS the URL identifier
- DNS-compatible format is stricter than Django's slug format — no conversion needed
- Add DNS-compatible validation to existing `name` fields
- Human-friendly display text belongs in Description field
- Existing data in DB already conforms to this format — no migration of values needed

### Validation Rules
- DNS-compatible (RFC 1123 label format)
- Allowed characters: `a-z`, `0-9`, hyphens
- Max length: 63 characters
- No leading or trailing hyphens
- Can start with letter or digit (RFC 1123 relaxed RFC 952's letter-only requirement)
- Fits Kubernetes naming, DNS subdomain rules, container naming

### Collision Handling
- Duplicate names rejected with clear error message
- No auto-suffixing (e.g., `my-api-2`)
- Message: "A [entity] named '[name]' already exists. Choose a different name."

### Entity-Specific URL Patterns
- Project: `/projects/{name}/`
- Environment: `/projects/{project-name}/environments/{env-name}/`
- Blueprint: `/blueprints/{name}/`
- Group: `/groups/{name}/`
- Connection: `/connections/{name}/`
- User: `/users/{uuid}/` (privacy — keeps UUID)

### Claude's Discretion
- Exact regex pattern for DNS-compatible validation
- Database index strategy for name lookups

</decisions>

- "Encourage thoughtful behavior by setting rigid rules" — immutability signals names matter
- Kubernetes/Terraform mental model — name is identity, description is for humans
- Input validation should show clear expectation that only slug format is allowed
