    2. **Update `core/consumers.py`** -- implement full `render_updates()`:
       The method must render all OOB-targetable sections and concatenate them into a single WS message. HTMX silently ignores OOB swaps for IDs not present in the current DOM, so it's safe to always send all sections.

       **OOB strategy (IMPORTANT -- no double-wrapping):**
       Plan 02 partials each own their outer div with a stable `id` (e.g., `<div id="dashboard-stats-row">`). The partial conditionally adds `hx-swap-oob="true"` when the template context includes `oob=True`. The consumer MUST NOT add an extra wrapper div. Instead, the consumer renders each partial with `{"oob": True, ...other_context}` and concatenates the raw rendered HTML strings. The partial's own root div becomes the OOB element.

       **Do NOT use a `build_oob_html()` wrapper function that adds `<div id="..." hx-swap-oob="true">` around rendered output.** The partials already handle this via `{% if oob %}hx-swap-oob="true"{% endif %}` on their root element.

       **render_updates(state) -- @sync_to_async:**
       Build template context from state dict, then render each section with `oob=True`:

       ```python
       @sync_to_async
       def render_updates(self, state):
           ctx = self.build_template_context(state)
           ctx["oob"] = True
           parts = []

           # Dashboard sections
           if ctx.get("total_builds", 0) == 0:
               parts.append(render_to_string("core/services/_dashboard_empty.html", ctx))
           else:
               parts.append(render_to_string("core/services/_stats_row.html", ctx))

           parts.append(render_to_string("core/services/_recent_builds.html", ctx))
           parts.append(render_to_string("core/services/_ci_pipeline_card.html", ctx))

           # Builds tab (always sent; HTMX ignores if element not in DOM)
           parts.append(render_to_string("core/services/_builds_tab.html", ctx))

           # CI manifest status
           parts.append(render_to_string("core/services/_ci_manifest_status.html", ctx))

           # Scaffold badge (only when relevant)
           if state.get("scaffold_status") in ("pending", "running", "failed"):
               parts.append(
                   f'<span id="scaffold-badge" hx-swap-oob="true">'
                   f'{self.render_scaffold_badge(state)}</span>'
               )

           return "".join(parts)
       ```

       Note: `_builds_tab.html` needs its own `id` + OOB conditional for WS push. If it doesn't already have a root `<div id="builds-tab-content">`, the existing `id="builds-tab-content"` div should get the `{% if oob %}hx-swap-oob="true"{% endif %}` pattern added (same as dashboard partials). If the builds tab partial already has a different structure, wrap its content in the OOB div. For the builds tab, since the consumer sends the first page with default sort, add `oob=True` to context and ensure the template's root element has the conditional attribute.

       For CI manifest status: extract the status section of `_ci_tab.html` into a small partial `_ci_manifest_status.html` with root `<div id="ci-manifest-status" {% if oob %}hx-swap-oob="true"{% endif %}>`, OR add the OOB id + conditional directly to the relevant section within `_ci_tab.html`. Choose whichever is cleaner given the existing template structure.

       **Template context building:**
       The consumer must build the same context variables that `ServiceDetailView.get_context_data` provides for the details tab. Create a helper method `build_template_context(state)` that constructs:
       - `service`, `project` objects (fetch in get_current_state)
       - `total_builds`, `last_build`, `success_rate`, `avg_build_time_seconds`, `recent_builds`
       - `ci_workflow`, `ci_workflow_version`
       - `builds` (for builds tab -- paginated queryset, first page)
       - `has_running_builds`, `has_any_builds`
       - `show_empty_state`

       Note: For the builds tab OOB swap, send just the first page of builds with default sort. Users who have navigated to specific pages/filters will get a full refresh -- this is acceptable per the locked decision "full list refresh when any build changes status".

       Alpine.data('wsStatus', function() {
           return {
               connected: false,
               init: function() {
                   var self = this;
                   document.addEventListener('htmx:wsOpen', function() {
                       self.connected = true;
                   });
                   document.addEventListener('htmx:wsClose', function() {
                       self.connected = false;
                   });
                   document.addEventListener('htmx:wsError', function() {
                       self.connected = false;
                   });
               }
           };
       });

       In `base.html`, add vanilla JS listeners in the existing nonce'd script block (after the htmx:wsAfterMessage listener from Task 1):
       ```javascript
       var wsEverConnected = false;
       document.addEventListener('htmx:wsOpen', function() {
           wsEverConnected = true;
           var wsWarning = document.getElementById('ws-fallback-warning');
           if (wsWarning) wsWarning.classList.add('hidden');
       });
       document.addEventListener('htmx:wsClose', function() {
           if (wsEverConnected) {
               var wsWarning = document.getElementById('ws-fallback-warning');
               if (wsWarning) wsWarning.classList.remove('hidden');
           }
       });
       ```

       This vanilla JS approach avoids CSP event name parsing issues entirely. The `wsStatus` Alpine component handles the sidebar dot (simpler scope), and the vanilla JS handles the fallback banner (reliable, no Alpine dependency).
