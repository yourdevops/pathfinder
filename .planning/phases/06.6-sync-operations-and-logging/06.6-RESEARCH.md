# Phase 06.6: Sync Operations and Logging - Research

**Researched:** 2026-02-16
**Domain:** Django models, webhook handling, background tasks, GitHub branch protection API, sync logging UI
**Confidence:** HIGH

## Summary

This phase implements three gaps from the remediation plan: branch protection validation (GAP-05), repository sync triggers (GAP-06), and sync operation logging (GAP-07). The current codebase has a fully working `scan_steps_repository` task with per-file SHA detection, slug-based identity, change classification, and archival -- all from Phase 06.4. What is missing is: (1) structured logging of sync operations into database models, (2) webhook and scheduled triggers for automatic syncing, (3) branch protection validation before and during sync, and (4) a sync history UI on the repo detail page.

The existing webhook infrastructure (`core/views/webhooks.py`) provides a solid pattern for the steps repo webhook handler. The `configure_webhook` method on `GitHubPlugin` already supports creating webhooks with specific event types. PyGithub's `BranchProtection` class exposes all needed properties (`allow_force_pushes`, `allow_deletions`, `enforce_admins`, `required_pull_request_reviews.required_approving_review_count`). For the scheduled daily scan, the project does not have a periodic task framework; the simplest approach is a Django management command invocable via system cron or container scheduler, matching the existing pattern suggested for health checks.

**Primary recommendation:** Build this in 3 plans: (1) sync log models + integration into `scan_steps_repository`, (2) webhook handler + scheduled scan command + branch protection validation, (3) sync history UI on repo detail page.

### Pattern 1: Sync Log Integration into scan_steps_repository
**What:** Wrap the existing scan logic to create `StepsRepoSyncLog` at start, create `StepSyncEntry` records for each step action during scan, and finalize the log with status/timing at end.
**When to use:** Every call to `scan_steps_repository` -- manual, webhook-triggered, or scheduled.
**Example:**
```python
# Source: Design from docs/ci-workflows/logging.md
from django.utils import timezone

@task(queue_name="steps_scan")
def scan_steps_repository(repository_id: int) -> dict:
    repository = StepsRepository.objects.get(id=repository_id)

    # Create sync log at start
    previous_sha = repository.last_scanned_sha or ""
    sync_log = StepsRepoSyncLog.objects.create(
        repository=repository,
        commit_sha="",  # Will be set after clone
        previous_sha=previous_sha,
        status="running",
        started_at=timezone.now(),
        protection_valid=False,
    )

    try:
        # ... clone repo, get HEAD SHA ...
        sync_log.commit_sha = head_sha

        # Check if SHA unchanged (skip)
        if head_sha == previous_sha:
            sync_log.status = "skipped"
            sync_log.completed_at = timezone.now()
            sync_log.save()
            return {"status": "skipped"}

        # Branch protection check
        protection_result = check_branch_protection(repository)
        sync_log.protection_valid = protection_result["valid"]
        if not protection_result["valid"]:
            StepSyncEntry.objects.create(
                sync_log=sync_log,
                step_slug="",
                action="skipped",
                severity="error",
                message=protection_result["message"],
            )

        # ... existing scan logic, but create StepSyncEntry for each action ...
        for raw_step in raw_steps:
            # On create:
            StepSyncEntry.objects.create(
                sync_log=sync_log, step_slug=slug,
                action="added", severity="info",
                message=f"New step discovered: {name}",
            )
            # On update:
            StepSyncEntry.objects.create(
                sync_log=sync_log, step_slug=slug,
                action="updated", severity="info",
                message=f"Step updated ({change_type} change)",
            )

        # Finalize
        sync_log.status = "success"  # or "partial" if errors
        sync_log.completed_at = timezone.now()
        sync_log.save()
    except Exception as e:
        sync_log.status = "failed"
        sync_log.completed_at = timezone.now()
        sync_log.save()
```

### Pattern 2: Separate Webhook Endpoint for Steps Repo
**What:** A dedicated webhook endpoint at `/webhooks/steps-repo/` that handles push events to steps repositories, distinct from the build webhook at `/webhooks/build/`.
**When to use:** When GitHub pushes to a steps repository's default branch.
**Example:**
```python
# Source: Existing pattern from core/views/webhooks.py (build_webhook)
@csrf_exempt
def steps_repo_webhook(request: HttpRequest) -> HttpResponse:
    """Handle push events for steps repositories."""
    if request.method != "POST":
        return HttpResponse(status=200)

    try:
        payload = json.loads(request.body)
    except json.JSONDecodeError:
        return HttpResponse(status=200)

    # Only process push events
    event_type = request.headers.get("X-GitHub-Event", "")
    if event_type != "push":
        return HttpResponse(status=200)

    # Identify steps repository by repo URL
    repo_url = payload.get("repository", {}).get("html_url", "")
    repository = StepsRepository.objects.filter(git_url__contains=repo_url).first()
    if not repository:
        return HttpResponse(status=200)

    # Only process pushes to default branch
    ref = payload.get("ref", "")
    if ref != f"refs/heads/{repository.default_branch}":
        return HttpResponse(status=200)

    # Verify signature using connection's webhook secret
    # ... signature verification ...

    # Enqueue scan task
    scan_steps_repository.enqueue(repository_id=repository.id)
    return HttpResponse(status=200)
```

### Pattern 3: Management Command for Scheduled Daily Scan
**What:** A Django management command that enqueues scan tasks for all repositories, callable via system cron.
**When to use:** Daily scheduled scan (e.g., `0 2 * * * python manage.py scan_all_steps_repos`).
**Example:**
```python
# Source: Matches existing pattern from core/tasks.py (schedule_health_checks)
# core/management/commands/scan_all_steps_repos.py
from django.core.management.base import BaseCommand

class Command(BaseCommand):
    help = "Enqueue scan tasks for all steps repositories"

    def handle(self, *args, **options):
        from core.models import StepsRepository
        from core.tasks import scan_steps_repository

        repos = StepsRepository.objects.all()
        for repo in repos:
            scan_steps_repository.enqueue(repository_id=repo.id)
            self.stdout.write(f"Enqueued scan for {repo.name}")
        self.stdout.write(self.style.SUCCESS(f"Enqueued {repos.count()} scans"))
```

### Pattern 4: Branch Protection Check via Plugin Interface
**What:** Add `check_branch_protection()` to `CICapableMixin` and implement in `GitHubPlugin` using PyGithub's branch protection API.
**When to use:** On repository registration and at the start of each sync.
**Example:**
```python
# Source: PyGithub BranchProtection API (verified from installed package source)
# plugins/github/plugin.py
def check_branch_protection(self, config: dict, repo_name: str, branch: str) -> dict:
    """Check branch protection rules on a repository branch.

    Returns dict with:
        valid: bool -- True if all required rules are in place
        rules: dict -- individual rule results
        message: str -- human-readable summary
    """
    try:
        g = self._get_github_client(config)
        repo = g.get_repo(repo_name)
        branch_obj = repo.get_branch(branch)

        if not branch_obj.protected:
            return {
                "valid": False,
                "rules": {},
                "message": f"Branch '{branch}' has no protection rules configured",
            }

        protection = branch_obj.get_protection()

        rules = {
            "no_force_push": not protection.allow_force_pushes,
            "no_deletions": not protection.allow_deletions,
            "enforce_admins": protection.enforce_admins,
            "required_reviews": False,
        }

        try:
            reviews = protection.required_pull_request_reviews
            if reviews and reviews.required_approving_review_count >= 1:
                rules["required_reviews"] = True
        except Exception:
            pass  # No review requirement configured

        valid = all(rules.values())

        # Build message for failures
        failures = [k for k, v in rules.items() if not v]
        if failures:
            message = "Missing protection rules: " + ", ".join(failures)
        else:
            message = "All branch protection rules are satisfied"

        return {"valid": valid, "rules": rules, "message": message}
    except Exception as e:
        return {
            "valid": False,
            "rules": {},
            "message": f"Failed to check branch protection: {e}",
        }
```

### Anti-Patterns to Avoid
- **Reusing the build webhook endpoint for steps repo events:** The build webhook identifies services by repo URL; steps repo push events have a completely different identification flow. Use separate endpoints.
- **Blocking sync on failed branch protection:** Log a warning and let the admin decide. The design says "offer to configure automatically if the plugin supports it" -- but do NOT silently skip the entire sync. Mark `protection_valid=False` on the sync log and continue.
- **Creating StepSyncEntry records outside the scan task:** All sync entries must be created within the atomic scan operation to maintain log consistency.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Branch protection API | Custom REST calls to GitHub | PyGithub's `branch.get_protection()` | Already handles auth, pagination, error codes |
| Webhook signature verification | New HMAC implementation | Existing `verify_github_signature()` in `webhooks.py` | Already tested, timing-safe |
| Periodic task scheduling | Custom recurring task framework | Management command + system cron | Matches existing project pattern, no new dependency |
| Steps repo identification from webhook | Complex URL matching | `StepsRepository.objects.filter(git_url=...)` with URL normalization | Direct FK lookup is simpler |

**Key insight:** The project already has all the building blocks -- webhook handling, signature verification, task enqueueing, CI plugin interface. This phase is primarily about connecting them together with proper logging.

## Common Pitfalls

### Pitfall 1: Webhook URL Matching for Steps Repositories
**What goes wrong:** Steps repo `git_url` may be stored as `https://github.com/owner/repo` while the webhook payload provides `https://github.com/owner/repo` (without `.git`), or vice versa.
**Why it happens:** Different contexts normalize URLs differently.
**How to avoid:** Normalize both URLs before comparison. Use `parse_git_url()` from `core/git_utils.py` to extract `owner/repo` and compare that tuple.
**Warning signs:** Webhook received but no scan triggered; logs show "No steps repository found for webhook."

### Pitfall 2: Concurrent Scan Prevention
**What goes wrong:** A webhook arrives while a manual scan is already running, causing duplicate scans that conflict.
**Why it happens:** No guard against concurrent scans for the same repository.
**How to avoid:** Check `repository.scan_status == "scanning"` before enqueuing a new scan. If already scanning, skip with a log message.
**Warning signs:** Duplicate `StepsRepoSyncLog` entries with overlapping time ranges.

### Pitfall 3: Branch Protection API Permissions
**What goes wrong:** `get_protection()` raises 403 Forbidden because the GitHub App or PAT lacks `admin:repo` permission.
**Why it happens:** Branch protection read requires `administration:read` permission on GitHub Apps or `repo` scope on PATs.
**How to avoid:** Catch the exception gracefully, set `protection_valid=False`, log a clear warning about missing permissions. Do not let this block the sync entirely.
**Warning signs:** All syncs show `protection_valid=False` even for protected branches.

### Pitfall 4: Webhook Registration Timing
**What goes wrong:** Webhook is registered on repo creation but the steps_repo_webhook URL endpoint does not exist yet (e.g., during incremental deployment).
**Why it happens:** Webhook URL must be routable at registration time.
**How to avoid:** Register the webhook endpoint URL in `pathfinder/urls.py` BEFORE implementing the webhook registration logic in `StepsRepoRegisterView`. Or better: register webhooks lazily on first successful scan.
**Warning signs:** GitHub shows webhook delivery failures (404) in webhook settings.

### Pitfall 5: StepsRepoSyncLog Foreign Key on Deleted Repository
**What goes wrong:** Deleting a `StepsRepository` cascades and deletes all sync logs, losing audit trail.
**Why it happens:** Using `on_delete=CASCADE` on the repository FK.
**How to avoid:** Use `on_delete=CASCADE` is acceptable here since the design doc does not require retaining sync history after repo deletion. But consider whether the audit trail via django-auditlog is sufficient.
**Warning signs:** N/A unless audit requirements change.

### Pitfall 6: SHA Comparison for "Skipped" Status
**What goes wrong:** The sync log says "skipped" but actually the SHA did change -- just the stored `previous_sha` was stale.
**Why it happens:** Need a reliable "last successfully scanned SHA" field on `StepsRepository`.
**How to avoid:** Add a `last_scanned_sha` field to `StepsRepository` (or derive from the most recent successful `StepsRepoSyncLog`). Update it only after successful scan completion.
**Warning signs:** Repeated "skipped" sync logs despite repository changes.

```python
# Source: Design from docs/ci-workflows/logging.md
class StepsRepoSyncLog(models.Model):
    """Log entry for each sync attempt on a steps repository."""

    STATUS_CHOICES = [
        ("running", "Running"),
        ("success", "Success"),
        ("partial", "Partial"),
        ("failed", "Failed"),
        ("skipped", "Skipped"),
    ]

    id = models.BigAutoField(primary_key=True)
    repository = models.ForeignKey(
        StepsRepository, on_delete=models.CASCADE, related_name="sync_logs"
    )
    commit_sha = models.CharField(max_length=40, blank=True)
    previous_sha = models.CharField(max_length=40, blank=True)
    status = models.CharField(max_length=20, choices=STATUS_CHOICES, default="running")
    started_at = models.DateTimeField()
    completed_at = models.DateTimeField(null=True, blank=True)
    protection_valid = models.BooleanField(default=False)
    trigger = models.CharField(
        max_length=20,
        choices=[("manual", "Manual"), ("webhook", "Webhook"), ("scheduled", "Scheduled")],
        default="manual",
    )
    steps_added = models.IntegerField(default=0)
    steps_updated = models.IntegerField(default=0)
    steps_archived = models.IntegerField(default=0)

    class Meta:
        db_table = "core_steps_repo_sync_log"
        ordering = ["-started_at"]

    def __str__(self):
        return f"Sync {self.repository.name} @ {self.commit_sha[:8]} ({self.status})"


class StepSyncEntry(models.Model):
    """Per-step detail within a sync operation."""

    ACTION_CHOICES = [
        ("added", "Added"),
        ("updated", "Updated"),
        ("archived", "Archived"),
        ("skipped", "Skipped"),
    ]
    SEVERITY_CHOICES = [
        ("info", "Info"),
        ("warning", "Warning"),
        ("error", "Error"),
    ]

    id = models.BigAutoField(primary_key=True)
    sync_log = models.ForeignKey(
        StepsRepoSyncLog, on_delete=models.CASCADE, related_name="entries"
    )
    step_slug = models.CharField(max_length=255, blank=True)
    action = models.CharField(max_length=20, choices=ACTION_CHOICES)
    severity = models.CharField(max_length=20, choices=SEVERITY_CHOICES, default="info")
    message = models.TextField(blank=True)

    class Meta:
        db_table = "core_step_sync_entry"
        ordering = ["id"]

    def __str__(self):
        return f"[{self.severity}] {self.step_slug}: {self.action}"
```

## Open Questions

1. **Should branch protection failure block registration entirely?**
   - What we know: Design doc says "block registration if protection is absent; allow admin override with warning"
   - What's unclear: The UX for the "admin override" -- is it a checkbox on the form, or a separate confirmation step?
   - Recommendation: Implement as a warning message with a "Register Anyway" confirmation button. Store `protection_valid=False` on the repo. Do not block scan operations.

2. **Should we offer auto-configuration of branch protection?**
   - What we know: Design doc mentions "offer to configure them automatically"
   - What's unclear: This requires `admin` permission on the repo, which the GitHub App/PAT may not have
   - Recommendation: Defer auto-configuration to a future iteration. For now, display clear instructions on what protection rules are needed and link to GitHub settings. The `edit_protection()` method exists on PyGithub's Branch class if needed later.

3. **Webhook secret for steps repo vs. service repo**
   - What we know: The webhook secret is stored on `IntegrationConnection.config_encrypted`. The steps repo uses a connection. Service repos use project connections.
   - What's unclear: Whether the same webhook secret should be used for both, or separate secrets per webhook type
   - Recommendation: Use the same connection's webhook_secret. The connection is shared between steps repos and service repos -- same GitHub App/PAT, same secret.

4. **How to identify steps repo from webhook when git_url format varies?**
   - What we know: `StepsRepository.git_url` stores the URL as entered by the user. Webhook payload provides `repository.html_url`.
   - What's unclear: Whether URLs will always match (e.g., `.git` suffix, HTTP vs HTTPS)
   - Recommendation: Use `parse_git_url()` on both sides to extract `(host, owner, repo)` tuple and match on that, not raw string comparison.
