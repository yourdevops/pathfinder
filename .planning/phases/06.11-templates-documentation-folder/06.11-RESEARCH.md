## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Manifest Schema
- Single `pathfinder.yaml` (or `.yml`) file for both templates and services
- `kind` field distinguishes: `kind: template` vs `kind: service`
- `required_vars` section uses name + description only (no types, no defaults, no required flags)
  - Example: `DATABASE_URL: "PostgreSQL connection string"`
- `runtimes` section uses YAML-native list format with version constraints
  - Example: `runtimes:\n  - python: ">=3.11"\n  - node: ">=20"`
- Runtimes used only for recommending compatible CI Workflows — no direct workflow references in manifest
- `kind` field noted as extensible for future `component` kind, but component schema not designed in this phase

#### Variable Lifecycle
- Pathfinder reads `pathfinder.yaml` on every build webhook at that commit SHA
- Variables from manifest layer ON TOP of existing env_vars system (separate concepts: manifest declares WHAT is needed; env_vars provides VALUES)
- New variable detected → deployment blocked until all required variables have values configured (hard gate)
- Variable removed from manifest → configured values kept, marked as "stale/no longer required" in UI; user cleans up manually

#### Document Structure
- Mirror `docs/ci-workflows/` structure: README.md, manifest-schema.md, variable-lifecycle.md, scaffolding.md, template-registration.md
- Separate `examples.md` file with complete pathfinder.yaml samples for different stacks (Python FastAPI, Node.js, etc.)
- Clean-slate design — no references to old Blueprints system
- `scaffolding.md` covers full flow redesign (not just integration with existing Jinja2 mechanism)

#### Template Scope
- Manual registration: operator provides git URL, Pathfinder reads pathfinder.yaml from it
- Git tags = versions: each tag is a selectable template version (like old Blueprints versioning)
- Scaffolded repos keep `pathfinder.yaml` with `kind: service` — variables section stays for build-time detection
- Runtimes section stays in scaffolded service repos for decorative/informational purposes

### Claude's Discretion
- Exact YAML schema field naming conventions
- Internal cross-references between doc files
- Level of detail in scaffolding flow redesign
- How to document the build-time manifest reading mechanism

### Deferred Ideas (OUT OF SCOPE)
- Components/Resources (IaC templates for env components like databases) — future phase, possibly monorepo structure
- Auto-discovery of templates from connected repos — not in scope, manual registration only
- Type hints / defaults / required flags on variable declarations — kept minimal for now (name + description)
</user_constraints>

## Summary

Phase 06.11 is a pure documentation phase. It creates `docs/templates/` with the same multi-file RFC structure as `docs/ci-workflows/`. No code is written. The output is six Markdown files that design the Templates system end-to-end: manifest schema, variable lifecycle, scaffolding flow redesign, template registration, and examples.

The core innovation of the new Templates design over the old Blueprints system is that `pathfinder.yaml` is a **living manifest** — it travels with the repo, transitions from `kind: template` to `kind: service` after scaffolding, and is re-read at every build to discover required variables. This is fundamentally different from the old `ssp-template.yaml` which was registration-only metadata. The new design ties variable discovery into the build loop rather than the wizard step.

The `docs/ci-workflows/` directory is the direct structural reference. Its six-file shape (README + five topic files) maps cleanly onto the six files the user specified. The depth of documentation in `ci-workflows/` provides a clear quality bar: each file is self-contained with flow diagrams, data models, state machines, and cross-references.

**Primary recommendation:** Author all six files in one pass with consistent cross-reference anchors, since the planner will break them into separate tasks. Establish the YAML schema in `manifest-schema.md` first — every other document references it.

## Architecture Patterns

### Recommended Document Structure

```
docs/templates/
├── README.md                   # Concept, non-goals, doc table, quick reference
├── manifest-schema.md          # Full pathfinder.yaml field reference
├── variable-lifecycle.md       # Required vars: detection, enforcement, stale cleanup
├── scaffolding.md              # Full wizard flow + post-scaffold repo state
├── template-registration.md    # Registration, git-tags-as-versions, sync
└── examples.md                 # Complete pathfinder.yaml samples
```

### Pattern 1: README Structure (mirror ci-workflows exactly)

The README follows a locked 4-section format:

```markdown
# Templates

[concept definition paragraph — what Templates ARE, what pathfinder.yaml IS]

## What Templates Do Not Do

- [bullet list of explicit non-goals]

## Documentation

| Document | Description |
|----------|-------------|
| [file.md](file.md) | [one-line description] |

## Quick Reference

[tables with key concepts, states, or field summaries]
```

**Non-goals to document (verified from CONTEXT.md decisions):**
- Templates do not define CI Workflows — that is a CI Workflows concern
- Templates do not inject CI steps or generate CI manifests
- Templates do not manage runtime version selection for builds — runtimes are a hint for CI Workflow recommendation only
- Templates are not auto-discovered from connected repos — registration is manual
- Component/Resource templates (IaC) are not in scope

### Pattern 2: Manifest Schema Document Structure

The `manifest-schema.md` is the foundation. All other docs refer to it. It should cover:

1. **The unified `pathfinder.yaml` concept** — one file, two kinds
2. **Top-level fields** with precise YAML examples
3. **`kind: template`** — what fields apply, what is required vs optional
4. **`kind: service`** — what fields apply after scaffolding, what Pathfinder reads at build time
5. **`required_vars` schema** — map format (name: description), no types, no defaults
6. **`runtimes` schema** — list format with version constraint strings
7. **Extensibility note** — `kind` is extensible for future `component`, not designed here

**Canonical `kind: template` schema:**

```yaml
kind: template
name: python-fastapi          # DNS-compatible identifier
description: "FastAPI service with async PostgreSQL"
runtimes:
  - python: ">=3.11"
required_vars:
  DATABASE_URL: "PostgreSQL connection string"
  SECRET_KEY: "Django/app secret key"
  REDIS_URL: "Redis connection string for caching"
```

**Canonical `kind: service` schema (post-scaffolding):**

```yaml
kind: service
name: order-service           # Same as service name in Pathfinder
description: "Order processing microservice"
runtimes:
  - python: ">=3.11"
required_vars:
  DATABASE_URL: "PostgreSQL connection string"
  SECRET_KEY: "Django/app secret key"
  REDIS_URL: "Redis connection string for caching"
```

**Key design choice to document:** `name` in `kind: service` is the service handler name. Pathfinder uses this for correlation when reading the manifest at build time.

### Pattern 3: Variable Lifecycle — The Hard Gate

This is the central new behavior in the Templates design. It bridges the manifest (what vars are needed) and the env_vars system (what values exist). Document as a state machine:

```
required_vars detected at build time
       |
       v
For each var in required_vars:
  Does a value exist in env_vars for this service?
  ├── YES → var satisfied
  └── NO  → var UNSATISFIED → deployment BLOCKED
       |
       v
All vars satisfied?
  ├── YES → deployment allowed to proceed
  └── NO  → deployment blocked with list of missing vars
```

**Stale variable flow (var removed from manifest):**

```
required_var removed from pathfinder.yaml
       |
       v
Next build read detects var no longer in required_vars
       |
       v
Existing env_var value kept in database (not deleted)
       |
       v
UI marks var as "stale / no longer required"
       |
       v
User manually removes from env_vars settings
```

**Critical distinction to document clearly:**
- `required_vars` in manifest = DECLARATION of what keys must exist (names + descriptions only)
- `env_vars` in Pathfinder = VALUES provided by operator/user (the existing cascade system)
- These are separate systems. The manifest does not contain values. env_vars does not contain requirements.

### Pattern 4: Build-Time Manifest Read (mirror build-lifecycle.md pattern)

Document this as a flow triggered by the build webhook:

```
Build webhook received at commit SHA
       |
       v
Pathfinder fetches pathfinder.yaml at that commit SHA
(using SCM plugin fetch_file_at_commit, same pattern as manifest_content fetch in CI workflows)
       |
       v
File present?
  ├── NO  → no variable discovery; deployment not blocked by manifest
  └── YES →
       v
Parse required_vars list
       |
       v
Diff against previously known required_vars for this service
       |
       v
New vars → add to service's "required but not configured" list
Removed vars → mark existing env_vars as stale
Unchanged → no action
       |
       v
Block deployment if any required_vars lack values in env_vars cascade
```

### Pattern 5: Template Registration and Versioning

Mirror the old Blueprint registration flow but with `pathfinder.yaml`:

```
Registration flow:
1. Operator provides git URL
2. Pathfinder clones/fetches (shallow) the HEAD of default branch
3. Pathfinder reads pathfinder.yaml from root
4. Validates: kind == template, name present, schema valid
5. Creates Template record with metadata from manifest
6. Fetches all git tags → each tag = a selectable version
7. Template is now available for use in Projects
```

**Git tags as versions:**
- Each git tag on the template repo = one selectable version
- Operator selects version during service creation wizard (same UX as old Blueprint version selector)
- Pathfinder reads pathfinder.yaml at the selected tag's commit SHA (not HEAD)
- Scaffolding uses the tagged commit's file tree

**Sync behavior:**
- Manual re-sync refreshes tag list and re-reads manifest from HEAD
- No automatic sync (not in scope)

### Pattern 6: Scaffolding Flow (full redesign)

The old scaffolding (Jinja2 substitution into cloned template) is replaced with a cleaner model. Document the full wizard flow as redesigned:

```
Wizard Page 1: Select Template + Version + Service Name + Project
       |
       v
Wizard Page 2: SCM — new repo or existing repo + branch config
       |
       v
Wizard Page 3: Configure required_vars initial values
  (Pathfinder reads required_vars from template manifest at selected tag)
  (Presents each var with its description; user provides initial values)
       |
       v
Wizard Page 4: Review + CI Workflow recommendation
  (Pathfinder reads runtimes from manifest, recommends compatible CI Workflows)
  (User confirms)
       |
       v
Scaffolding execution:
1. Fetch template repo at selected tag
2. Copy files to target repo (new branch or new repo)
3. Transform pathfinder.yaml: change kind from "template" to "service", set name to service handler
4. Commit and push (new repo: direct to main; existing repo: branch + PR)
5. Create Service record in Pathfinder
6. Store initial required_var values in service's env_vars
       |
       v
Post-scaffolding:
- Service repo has pathfinder.yaml with kind: service
- First build reads pathfinder.yaml → confirms required_vars, blocks or allows deployment
- Runtimes in service pathfinder.yaml are informational only post-scaffolding
```

**Key decision to document:** The manifest transformation during scaffolding — the template's `pathfinder.yaml` becomes the service's `pathfinder.yaml` with `kind` changed. This is the mechanism by which every service repo self-describes its requirements.

### Anti-Patterns to Avoid in the Documentation

- **Confusing manifest schema with old Blueprints schema:** Do not reference `ssp-template.yaml` field names (source.include, source.on_copy, ci.type, deploy.type). The new manifest is intentionally minimal.
- **Conflating required_vars with env_vars:** Document the separation explicitly. The manifest contains no values.
- **Implying runtime version selection in manifest:** Runtimes are version constraints for CI Workflow recommendation only. Concrete runtime version (e.g., python 3.12 specifically) is selected at the service/workflow level, not in the manifest.
- **Adding implementation detail to docs:** This is design documentation. Do not add Django model pseudocode, migration plans, or view implementation notes.

## Common Pitfalls

### Pitfall 1: Merging required_vars and env_vars into a single concept
**What goes wrong:** Documentation implies that `required_vars` in the manifest holds actual values, or that Pathfinder reads values from the manifest.
**Why it happens:** The old Blueprints schema had `variables` with `default:` values. The new schema deliberately removes defaults.
**How to avoid:** State explicitly in `variable-lifecycle.md` that the manifest contains only declarations (name + description). Values live exclusively in Pathfinder's env_vars system. Include a table showing the two systems side-by-side.
**Warning signs:** Any doc sentence that says "Pathfinder reads the value of X from pathfinder.yaml."

### Pitfall 2: Documenting the "kind: service" manifest as optional
**What goes wrong:** Documentation implies that service repos may or may not have a `pathfinder.yaml`, making variable lifecycle optional.
**Why it happens:** It's tempting to say "if the file exists, Pathfinder reads it" as a soft behavior.
**How to avoid:** Document that scaffolded repos ALWAYS have `pathfinder.yaml` because scaffolding creates it. Non-scaffolded (existing) repos joined to Pathfinder may not have one — document that case explicitly in `variable-lifecycle.md` (no manifest = no variable enforcement, deployment not blocked by this mechanism).
**Warning signs:** Any wording that makes variable enforcement seem conditional or optional for scaffolded services.

### Pitfall 3: Documenting runtimes as a CI Workflow selector
**What goes wrong:** Documentation implies that `runtimes` in `pathfinder.yaml` directly assigns a CI Workflow to the service.
**Why it happens:** The CONTEXT.md says "runtimes for CI recommendation" which could be read as automatic assignment.
**How to avoid:** Document runtimes as recommendation input only. The CI Workflow is assigned separately by the operator in the service settings. Runtimes narrow down compatible workflows in the recommendation UI — they do not select one.
**Warning signs:** Any sentence that says "Pathfinder selects the CI Workflow based on runtimes."

### Pitfall 4: Including implementation-specific details in design docs
**What goes wrong:** Design docs include Django model fields, migration notes, or view implementation details.
**Why it happens:** Researcher/planner temptation to document "how it will be built" alongside "what it should do."
**How to avoid:** Keep all six files at the design/behavior level. Model pseudocode (like in ci-workflows/versioning.md) is acceptable only if it clarifies the data shape, not the implementation.
**Warning signs:** Django-specific terms (ForeignKey, model method names, migration numbers) appearing in design docs.

### Pitfall 5: Breaking from ci-workflows document style
**What goes wrong:** Templates docs adopt a different heading hierarchy, table format, or cross-reference style than ci-workflows docs.
**Why it happens:** Each doc file gets written in isolation without checking the reference.
**How to avoid:** Open `docs/ci-workflows/README.md` and `docs/ci-workflows/versioning.md` before writing each file. Use identical heading depth, ASCII flow diagrams, table formatting.
**Warning signs:** H4 headings where ci-workflows uses H3, or prose paragraphs where ci-workflows uses bullet lists.

### Build-time read pattern (mirrors build-authorization.md verification flow)

```
On build webhook at commit SHA:
1. Fetch pathfinder.yaml at commit SHA from service repo (via SCM plugin)
2. If not present: skip variable enforcement for this build
3. Parse required_vars dict
4. Compare against service's previously known required_vars
5. Compute diff: added_vars, removed_vars, unchanged_vars
6. For added_vars: add to service's required list; check if values exist in env_vars cascade
7. For removed_vars: mark corresponding env_vars entries as "stale"
8. If any required_var lacks a value: set service's deployment_blocked flag
9. Build completes normally; deployment gate enforced separately at deploy time
```

## Open Questions

1. **How does Pathfinder fetch `pathfinder.yaml` at build time?**
   - What we know: The CI plugin already has `fetch_manifest_content(config, repo_name, manifest_id, commit_sha)` for CI manifests (docs/ci-workflows/plugin-interface.md). The same SCM plugin capability can fetch any file at a commit SHA.
   - What's unclear: Whether the design doc should specify the fetch mechanism (SCM plugin) or leave it abstract. Since implementation is a future phase, the design doc should specify behavior (read at commit SHA) and leave mechanism abstract.
   - Recommendation: Document as "Pathfinder reads pathfinder.yaml at the build's commit SHA from the service repository" without specifying which plugin method. Mentioning "via SCM plugin" is acceptable for orientation.

2. **What is the exact pathfinder.yaml file name — `pathfinder.yaml` or `pathfinder.yml`?**
   - What we know: CONTEXT.md says "pathfinder.yaml (or .yml)" — both are acceptable.
   - What's unclear: Whether the docs should define one canonical extension or document both.
   - Recommendation: Define `pathfinder.yaml` as canonical (matching the precedent of Kubernetes preferring `.yaml`). Note `.yml` as an accepted alias.

3. **What happens to `required_vars` at deployment time for a service with no pathfinder.yaml?**
   - What we know: Not all service repos will have a manifest (pre-existing repos onboarded without scaffolding). The user's vision focuses on scaffolded repos.
   - What's unclear: Whether to document the "no manifest" case explicitly or ignore it.
   - Recommendation: Document it briefly in `variable-lifecycle.md`: "If pathfinder.yaml is absent from the service repo, no variable enforcement applies. Deployment is not blocked by the manifest mechanism."

4. **Does the wizard's required_vars configuration page (Wizard Page 3) pre-populate descriptions from the manifest?**
   - What we know: The wizard reads required_vars from the manifest at the selected tag. Descriptions are available.
   - What's unclear: Whether the wizard shows descriptions as helper text during initial value entry.
   - Recommendation: Yes, document it this way in `scaffolding.md` — descriptions from the manifest are shown as helper text next to each var input field. This is natural UX and aligns with the "description" field's purpose.

5. **Are runtimes in `kind: service` validated against anything at build time?**
   - What we know: CONTEXT.md says runtimes in service repos are "decorative/informational purposes."
   - What's unclear: Whether they participate in CI Workflow recommendation for existing services (e.g., when re-assigning a workflow).
   - Recommendation: Document runtimes in `kind: service` as purely informational — they are not re-evaluated after scaffolding. CI Workflow assignment is a separate concern managed in service settings.
