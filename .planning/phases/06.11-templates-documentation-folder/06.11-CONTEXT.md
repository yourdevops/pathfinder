# Phase 6.11: Templates Documentation Folder - Context

**Gathered:** 2026-02-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Create `docs/templates/` documentation folder (structured like `docs/ci-workflows/`) that designs the Templates system end-to-end. Templates replace the old Blueprints concept for service scaffolding. A unified `pathfinder.yaml` manifest declares whether a repo is a template or a running service, what variables it requires, and what runtimes it uses. Documentation only — implementation follows in a separate phase.

</domain>

<decisions>
## Implementation Decisions

### Manifest Schema
- Single `pathfinder.yaml` (or `.yml`) file for both templates and services
- `kind` field distinguishes: `kind: template` vs `kind: service`
- `required_vars` section uses name + description only (no types, no defaults, no required flags)
  - Example: `DATABASE_URL: "PostgreSQL connection string"`
- `runtimes` section uses YAML-native list format with version constraints
  - Example: `runtimes:\n  - python: ">=3.11"\n  - node: ">=20"`
- Runtimes used only for recommending compatible CI Workflows — no direct workflow references in manifest
- `kind` field noted as extensible for future `component` kind, but component schema not designed in this phase

### Variable Lifecycle
- Pathfinder reads `pathfinder.yaml` on every build webhook at that commit SHA
- Variables from manifest layer ON TOP of existing env_vars system (separate concepts: manifest declares WHAT is needed; env_vars provides VALUES)
- New variable detected → deployment blocked until all required variables have values configured (hard gate)
- Variable removed from manifest → configured values kept, marked as "stale/no longer required" in UI; user cleans up manually

### Document Structure
- Mirror `docs/ci-workflows/` structure: README.md, manifest-schema.md, variable-lifecycle.md, scaffolding.md, template-registration.md
- Separate `examples.md` file with complete pathfinder.yaml samples for different stacks (Python FastAPI, Node.js, etc.)
- Clean-slate design — no references to old Blueprints system
- `scaffolding.md` covers full flow redesign (not just integration with existing Jinja2 mechanism)

### Template Scope
- Manual registration: operator provides git URL, Pathfinder reads pathfinder.yaml from it
- Git tags = versions: each tag is a selectable template version (like old Blueprints versioning)
- Scaffolded repos keep `pathfinder.yaml` with `kind: service` — variables section stays for build-time detection
- Runtimes section stays in scaffolded service repos for decorative/informational purposes

### Claude's Discretion
- Exact YAML schema field naming conventions
- Internal cross-references between doc files
- Level of detail in scaffolding flow redesign
- How to document the build-time manifest reading mechanism

</decisions>

<specifics>
## Specific Ideas

- "Any service repo can have a pathfinder.yaml that tells Pathfinder about itself"
- Variable discovery happens at build time: "Read this on every build, and if a list of variables has changed, reflect that in Pathfinder UI"
- Deployment enforcement: "a variable got added — require user to provide the value on next deployment of that Service Version"
- Runtimes for CI recommendation: "used for recommending a CI Workflow on the next step"
- The user envisions completing the end-to-end onboarding wizard: template selection → scaffolding → CI workflow recommendation → deployment

</specifics>

<deferred>
## Deferred Ideas

- Components/Resources (IaC templates for env components like databases) — future phase, possibly monorepo structure
- Auto-discovery of templates from connected repos — not in scope, manual registration only
- Type hints / defaults / required flags on variable declarations — kept minimal for now (name + description)

</deferred>
