**H1: Variable Lifecycle**

Opening paragraph (2-3 sentences): required_vars in pathfinder.yaml are declarations of what environment variables a service needs to run. Pathfinder reads these declarations at every build and enforces that all required variables have values configured before a deployment can proceed.

**H2: Two Systems**

A short section clarifying the separation — this is the most common point of confusion. Must include a table:

| System | What It Contains | Where | Who Manages |
|--------|-----------------|-------|-------------|
| required_vars (manifest) | Variable names and descriptions — no values | pathfinder.yaml in the service repo | Service developer |
| env_vars (Pathfinder) | Variable values per environment | Pathfinder database | Operator / platform team |

"Pathfinder enforces that every name declared in required_vars has a corresponding value in env_vars before allowing deployment. The manifest never contains values."

Cross-reference: "See [Manifest Schema](manifest-schema.md) for the required_vars field format."

**H2: Build-Time Discovery**

State: Pathfinder reads pathfinder.yaml at the commit SHA of each build using the SCM plugin. If the file is absent from the repo, no variable enforcement applies for that build — deployment is not blocked by the manifest mechanism.

**H2: Deployment Gate**

Document the hard gate: after the build-time read, if any required variable lacks a value in the env_vars cascade (project-level or environment-level), the service's deployment is blocked.

ASCII diagram:

```

   |             |
  YES            NO
   |             |
   v             v

              missing variables
```

State explicitly: "The deployment gate is hard — there is no override or bypass. The operator must configure values for all missing variables before deployment can proceed."

State: the gate applies at deploy time, not at build time. Builds always complete. The block is on the subsequent deployment step.

State: stale variables do not block deployment. The service continues to function. The stale marker is informational — it prompts the operator to clean up the now-unused value.

**H2: No Manifest**

Brief section (3-4 sentences): If pathfinder.yaml is absent from the service repo (e.g., a pre-existing service repo onboarded without scaffolding), no variable enforcement applies. Deployment is not blocked by the manifest mechanism. Pathfinder treats the absence of a manifest as "no required variables declared." Scaffolded repos always have pathfinder.yaml because scaffolding creates it (see [Scaffolding](scaffolding.md)).

**H2: Variable States**

Table summarizing the possible states of a required variable at any point:

| State | Meaning | Deployment |
|-------|---------|------------|
| Satisfied | Declared in manifest; value exists in env_vars cascade | Allowed |
| Unsatisfied | Declared in manifest; no value in env_vars cascade | Blocked |
| Stale | Previously declared; removed from manifest; value still in env_vars | Allowed (no gate) |

IMPORTANT constraints from CONTEXT.md (locked):
- Build-time read happens at commit SHA — not HEAD, not latest
- Variable removal marks existing env_var values as stale; does NOT delete them
- New variable = deployment blocked (hard gate, no override)
- Stale variable = deployment allowed (informational only)
- DO NOT imply values come from the manifest
- DO NOT make the gate sound optional

**H1: Template Registration**

Opening paragraph (2-3 sentences): Operators register templates by providing a git URL. Pathfinder reads pathfinder.yaml from the repository to extract template metadata. Git tags on the template repository become selectable versions in the service creation wizard.

**H2: Registration Flow**

```
        v

Each tag becomes a selectable version
        |
        v

```

**H2: Required Fields**

Table: For a git repository to be registerable as a template, pathfinder.yaml must contain:

| Field | Required | Value |
|-------|----------|-------|
| kind | Yes | Must be "template" |
| name | Yes | DNS-compatible identifier |
| description | No | Human-readable summary |
| runtimes | No | List of runtime constraints |
| required_vars | No | Map of variable declarations |

Cross-reference: "For the full pathfinder.yaml schema, see [Manifest Schema](manifest-schema.md)."

**H2: Git Tags as Versions**

Document the versioning model. Git tags on the template repository are the version system — there is no separate semver assigned by Pathfinder.

Key behaviors to document:
- Each git tag on the template repo becomes a selectable version entry in Pathfinder
- When creating a service, the operator selects a version (git tag) from the available list
- Pathfinder reads pathfinder.yaml at the selected tag's commit SHA (not HEAD)
- Scaffolding uses the file tree at the selected tag's commit SHA
- Tags can follow any naming convention (e.g., v1.0.0, v1, 2024-01, release-candidate)
- Pathfinder presents tags in reverse chronological order (newest first)

Include an example showing a tag list as it appears in the UI:

```
Template: python-fastapi
Available versions:
  v2.1.0   (latest)
  v2.0.0
  v1.3.2
  v1.3.1
  v1.3.0
  v1.2.0
```

Note: "A template with no git tags has only HEAD as its version source. Operators are encouraged to tag releases to give users stable, selectable versions."

**H2: Sync**

Document manual sync behavior. Operators can trigger a sync on a registered template to refresh it.

What sync does:
1. Re-fetches the repository from the git URL
2. Reads pathfinder.yaml from HEAD of the default branch
3. Updates Template record metadata (name, description, runtimes, required_vars) from the manifest
4. Fetches the current git tag list — new tags are added as available versions
5. Tags that no longer exist on the remote are not automatically removed (they remain visible but are flagged as unavailable)

What sync does NOT do:
- Does not automatically retrigger scaffolding for existing services
- Does not push manifest updates to existing service repos
- Does not auto-update services that were created from earlier versions

State: "There is no automatic sync. Operators trigger sync manually when they want to pick up new tags or updated metadata."

Cross-reference: "For how pathfinder.yaml fields affect template display, see [Manifest Schema](manifest-schema.md)."

**H2: Template Availability**

Brief section (3-4 sentences): Templates are available to all Projects by default once registered. (Note: access scoping is a future concern.) A template can be deregistered if no service was ever created from it. Deregistering a template that has been used to create services is blocked to preserve scaffolding history.

IMPORTANT constraints from CONTEXT.md (locked):

- Git tags = versions (like old Blueprints versioning model)
- Pathfinder reads pathfinder.yaml at the selected tag's commit SHA, not HEAD
- No automatic sync — manual only
- DO NOT reference old Blueprints (ssp-template.yaml, Blueprint model)
