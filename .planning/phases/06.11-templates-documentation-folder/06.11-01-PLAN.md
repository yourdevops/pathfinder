**H1: Manifest Schema**

Opening paragraph (2-3 sentences): pathfinder.yaml is a single file that identifies a repo as either a template or a running service. Pathfinder reads it from the repo root. The kind field determines which fields apply and how Pathfinder behaves.

**H2: Overview**

Brief paragraph on the unified-file concept. One file, two kinds. Templates carry it in template repos; services carry it after scaffolding transforms kind: template to kind: service.

**H2: Top-Level Fields**

Table with columns: Field | Required | Applies To | Description. Rows:
- kind | Yes | All | Identifies the repo type: "template" or "service". Extensible for future kinds.
- name | Yes | All | DNS-compatible identifier. For kind: service, matches the service name in Pathfinder and is used for correlation at build time.
- description | No | All | Human-readable description shown in the Pathfinder UI.
- runtimes | No | All | List of runtime constraints used to recommend compatible CI Workflows. See Runtimes section.
- required_vars | No | All | Map of variable declarations required for deployment. See Required Variables section.

**H2: kind: template**

Explanation that this identifies the repo as a template available for scaffolding. Include annotated YAML example (kind: template, name, description, runtimes, required_vars with 2-3 vars). Note that all top-level fields apply.

Include a complete annotated example:
```yaml
kind: template
name: python-fastapi
description: "FastAPI service with async PostgreSQL and Redis"
runtimes:
  - python: ">=3.11"
required_vars:
  DATABASE_URL: "PostgreSQL connection string (asyncpg format)"
  SECRET_KEY: "Application secret key for JWT signing"
  REDIS_URL: "Redis connection string for caching"
```

**H2: kind: service**

Explanation that this identifies the repo as a running service. Pathfinder reads this file at every build webhook at the commit SHA of that build to discover required variables. Include complete annotated example showing kind: service after scaffolding transformation.

Include a note: "Scaffolded repos always have pathfinder.yaml with kind: service — scaffolding transforms the template's kind: template into kind: service and sets name to the service handler name. See [Scaffolding](scaffolding.md)."

Note that runtimes in kind: service are informational only — they are not re-evaluated after scaffolding. CI Workflow assignment is managed separately in service settings.

```yaml
kind: service
name: order-service
description: "Order processing microservice"
runtimes:
  - python: ">=3.11"
required_vars:
  DATABASE_URL: "PostgreSQL connection string (asyncpg format)"
  SECRET_KEY: "Application secret key for JWT signing"
  REDIS_URL: "Redis connection string for caching"
```

**H2: Runtimes**

Document the list format with version constraints. Format: a YAML list where each entry is a single-key map of runtime name to semver constraint string.

Supported format:
```yaml
runtimes:
  - python: ">=3.11"
  - node: ">=20"
```

State explicitly: runtimes are used exclusively to recommend compatible CI Workflows during scaffolding and in the service settings UI. They do not select a CI Workflow — the CI Workflow is assigned separately by the operator. They do not enforce runtime versions at build time. Concrete runtime version selection (e.g., Python 3.12 specifically) happens at the CI Workflow level.

**H2: Required Variables**

Document the map format: keys are variable names (uppercase, underscore-separated by convention), values are description strings.

```yaml
required_vars:
  DATABASE_URL: "PostgreSQL connection string (asyncpg format)"
  SECRET_KEY: "Application secret key for JWT signing"
  REDIS_URL: "Redis connection string for caching"
```

Key design decisions to state explicitly:
- required_vars contains only declarations (name + description). It does not contain values.
- No types, no defaults, no required flags. Every declared variable is implicitly required.
- If required_vars is absent or empty, no variable enforcement applies.

Include a clear two-column table distinguishing the two systems:

| Concept | What It Stores | Where It Lives | Who Manages It |
|---------|---------------|----------------|----------------|
| required_vars (manifest) | Variable names and their descriptions | pathfinder.yaml in the repo | Template author / service developer |
| env_vars (Pathfinder) | Variable values for each environment | Pathfinder database | Operator / platform team |

Cross-reference: "For how Pathfinder enforces required_vars at deployment time, see [Variable Lifecycle](variable-lifecycle.md). For how operators register templates and manage versions, see [Template Registration](template-registration.md)."

**H2: Extensibility**

Note that kind is designed to be extensible. A future kind: component is planned for IaC component templates (databases, queues, etc.) but is not defined in this version of the schema. Only kind: template and kind: service are in scope.

**H2: File Location and Naming**

pathfinder.yaml must be in the repository root. The canonical extension is .yaml. The .yml extension is also accepted. If both are present, pathfinder.yaml takes precedence.

IMPORTANT constraints to honor from CONTEXT.md (locked decisions):
- required_vars uses name + description ONLY — no types, no defaults, no required flags
- runtimes uses YAML list format with version constraint strings
- runtimes recommend CI Workflows only — no direct workflow references in manifest
- kind is extensible but only template and service are in scope now
- DO NOT reference old Blueprints fields (ssp-template.yaml, source.include, ci.type, deploy.type)
