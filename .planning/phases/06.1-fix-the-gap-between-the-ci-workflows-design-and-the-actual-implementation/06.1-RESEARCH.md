# Phase 06.1: Fix CI Workflows Design-Implementation Gap - Research

**Researched:** 2026-02-11
**Domain:** CI Workflow Versioning, Build Verification, Plugin Interface Completion (Django 6.x)
**Confidence:** HIGH

## Summary

This phase closes the gap between the design documents in `docs/ci-workflows/` and the current implementation. The design docs are detailed and authoritative -- they specify exact model schemas, plugin method signatures, verification algorithms, and state machines. The current codebase has the foundational models (CIWorkflow, CIStep, Build) and task infrastructure but lacks the version lifecycle (CIWorkflowVersion model), build verification flow (hash computation + version matching), four missing plugin interface methods, deterministic manifest generation with version headers, and the forking mechanism.

The primary engineering challenge is integrating the new CIWorkflowVersion model into the existing CIWorkflow editing flow while maintaining the draft-single-slot constraint, and wiring up the verify_build background task to perform the 7-step verification flow described in build-authorization.md. The existing `poll_build_details` task must be modified to trigger verification after build completion.

**Primary recommendation:** Implement bottom-up -- model + migration first, then plugin methods, then manifest generation, then verification task, then UI layers. Each layer is testable independently.

### Pattern 1: Model + Migration for CIWorkflowVersion
**What:** New model per design doc schema in `docs/ci-workflows/versioning.md`
**Source:** `docs/ci-workflows/versioning.md` lines 16-31

The design doc specifies the exact model:
```python
class CIWorkflowVersion(models.Model):
    class Status(models.TextChoices):
        DRAFT = "draft"
        AUTHORIZED = "authorized"
        REVOKED = "revoked"

    workflow = models.ForeignKey(CIWorkflow, on_delete=models.PROTECT, related_name="versions")
    version = models.CharField(max_length=32, blank=True)  # semver, blank while draft
    status = models.CharField(max_length=16, choices=Status.choices)
    manifest_hash = models.CharField(max_length=64)  # SHA-256
    manifest_content = models.TextField()  # full manifest text
    author = models.ForeignKey(User, on_delete=models.SET_NULL, null=True)
    created_at = models.DateTimeField(auto_now_add=True)
    published_at = models.DateTimeField(null=True, blank=True)
    revoked_at = models.DateTimeField(null=True, blank=True)
```

**Constraints per design doc:**
- At most one row with `status=draft` per workflow
- `version` is unique within a workflow (enforced on publish)
- `manifest_hash` is immutable once `status=authorized`

**Claude's discretion additions recommended:**
- `revoked_by` (ForeignKey to User, null=True) -- for audit trail of who revoked
- `changelog` (TextField, blank=True) -- for publish modal changelog summary
- UniqueConstraint for single draft: `UniqueConstraint(fields=["workflow"], condition=Q(status="draft"), name="unique_draft_per_workflow")`
- UniqueConstraint for version: `UniqueConstraint(fields=["workflow", "version"], condition=~Q(version=""), name="unique_version_per_workflow")`

### Pattern 4: Manifest Path Discrepancy (CRITICAL)
**What:** Current `manifest_path(service)` returns wrong path
**Impact:** Must be corrected for verification to work

Current code:
```python
# plugins/github/plugin.py:115
def manifest_path(self, service) -> str:
    return f".github/workflows/{service.handler}.yml"
    # Returns: .github/workflows/{project-name}-{service-name}.yml
```

Design doc requires:
```python
def manifest_id(self, workflow) -> str:
    return f".github/workflows/ci-{workflow.name}.yml"
    # Returns: .github/workflows/ci-{workflow-name}.yml
```

The existing `manifest_path` should be updated/replaced by `manifest_id`. The filename should be based on the workflow name with `ci-` prefix, not the service handler. The `push_ci_manifest` task currently uses `manifest_path(service)` and must be updated to use `manifest_id(workflow)`.

### Pattern 5: Deterministic Manifest Generation
**What:** Current `generate_manifest()` must be made deterministic and include version header
**Source:** `docs/ci-workflows/build-authorization.md`

Current issues with `generate_manifest()`:
1. No version header (the YAML comment block)
2. No version parameter -- signature needs `generate_manifest(workflow, version=None)`
3. PyYAML `yaml.dump(sort_keys=False)` is deterministic for the same input dict, BUT YAML comments cannot be added via `yaml.dump`

**Solution:** Generate YAML body via `yaml.dump(sort_keys=False)`, then prepend the header comment block:
```python
header = (
    "# ==================================================\n"
    "# Managed by Pathfinder - DO NOT EDIT MANUALLY\n"
    f"# Workflow: {workflow.name}\n"
    f"# Version: {version}\n"
    "# ==================================================\n"
)
return header + yaml.dump(manifest, default_flow_style=False, sort_keys=False)
```

For draft manifests (no version number), use `# Version: draft` or omit the version line.

### Pattern 6: Verification Task Flow
**What:** Background task implementing the 7-step verification flow
**Source:** `docs/ci-workflows/build-authorization.md`

```
verify_build task:
1. Fetch authoritative build data from GitHub API (already done by poll_build_details)
2. Resolve Service's assigned CIWorkflow + manifest_id
3. Call plugin.fetch_manifest_content(repo, manifest_id, commit_sha)
4. Compute SHA-256 hash of fetched content
5. Look up hash in CIWorkflowVersion for that workflow:
   - Match + authorized -> Verified
   - Match + revoked -> Unauthorized
   - Match + draft -> Draft
   - No match -> Unauthorized
6. Store result on Build record (verification_status, manifest_hash, workflow_version FK)
7. Never recompute
```

**Queue:** Use `build_updates` queue (already exists in settings.py TASKS config).

**Trigger:** After `poll_build_details` completes and build status is terminal (success/failed/cancelled), enqueue `verify_build`.

### Pattern 7: Draft Auto-Creation on Edit
**What:** When user edits workflow steps/config, automatically create or update the draft CIWorkflowVersion
**Source:** CONTEXT.md decision: "Draft auto-created on workflow edit"

Current `WorkflowComposerView.post()` saves directly to CIWorkflow. The new flow:
1. On save, check if a draft CIWorkflowVersion exists for this workflow
2. If not, create one (status=draft, author=current user)
3. Regenerate manifest with `version=None` (draft)
4. Compute hash, store in draft version's manifest_hash and manifest_content
5. If draft exists, update its manifest_hash, manifest_content, author (take over)

### Anti-Patterns to Avoid
- **Recomputing verification status:** Design doc says "stored permanently, never recomputed" -- no recalculation on version revocation for existing builds
- **Trusting webhook payloads:** Webhook is notification-only; `verify_build` must fetch all data from GitHub API
- **Mutable authorized versions:** Once published, manifest_hash MUST NOT change
- **Using `manifest_path(service)` for verification:** Must use `manifest_id(workflow)` -- path is workflow-based not service-based

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SHA-256 hashing | Custom hash function | `hashlib.sha256(content.encode()).hexdigest()` | Standard library, no edge cases |
| Semver comparison | String comparison | `semver.Version.parse()` + comparison operators | Already in deps, handles pre-release/build metadata |
| Version validation | Custom regex | `semver.Version.parse()` with try/except ValueError | Handles all semver edge cases |
| YAML generation | Template strings | `yaml.dump(sort_keys=False)` + header prepend | Deterministic, handles escaping |
| Background tasks | Celery/custom queue | `django_tasks` with `@task(queue_name="build_updates")` | Already configured, matches existing patterns |
| File content at commit | Git clone + checkout | `PyGithub repo.get_contents(path, ref=sha).decoded_content` | Already available, no disk I/O |

**Key insight:** The design docs specify exact implementations for most components. The engineering work is integration and wiring, not algorithm design.

## Common Pitfalls

### Pitfall 1: Non-Deterministic YAML Output
**What goes wrong:** Different runs produce different bytes for the same logical manifest, breaking hash verification
**Why it happens:** YAML dict ordering, trailing whitespace, different newline styles
**How to avoid:** `yaml.dump(sort_keys=False)` preserves insertion order of OrderedDict/dict (Python 3.7+ dicts are ordered). Always use `\n` line endings. Strip trailing whitespace. Use `default_flow_style=False` consistently.
**Warning signs:** Same workflow definition produces different hashes across generations

### Pitfall 2: Manifest Path vs Manifest ID Confusion
**What goes wrong:** Builds verified against wrong file path, always showing Unauthorized
**Why it happens:** Current code uses `manifest_path(service)` which produces `{project}-{service}.yml`, but builds run `ci-{workflow}.yml`
**How to avoid:** Replace all manifest path usage with `manifest_id(workflow)`. The `push_ci_manifest` task must use `manifest_id` for the file path in the repo.
**Warning signs:** All builds showing as Unauthorized despite using Pathfinder-managed manifests

### Pitfall 3: Race Condition Between Build Completion and Verification
**What goes wrong:** Build marked complete but verification task hasn't run yet; UI shows empty verification status
**Why it happens:** Two separate async tasks (poll_build_details, verify_build) with no guaranteed ordering
**How to avoid:** `poll_build_details` should directly enqueue `verify_build` on terminal status. UI should show "Pending verification" when build is terminal but verification_status is blank.
**Warning signs:** Builds showing as complete with no verification badge

### Pitfall 4: Draft Slot Contention
**What goes wrong:** Two users editing the same workflow simultaneously, both creating drafts
**Why it happens:** No locking mechanism on the single draft slot
**How to avoid:** Use the UniqueConstraint on (workflow, status=draft). On save, use `update_or_create` with the constraint. The second save will update the existing draft (take-over behavior per design doc).
**Warning signs:** IntegrityError on draft creation

### Pitfall 5: Forgetting to Hash the Header
**What goes wrong:** Version number in header doesn't affect hash, multiple versions produce same hash
**Why it happens:** Hashing only the YAML body without the comment header
**How to avoid:** Hash the COMPLETE manifest content including the header comment block. The design doc explicitly states "The version string is embedded in the header. It is part of the file content and therefore part of the hash."
**Warning signs:** Different versions of same workflow producing identical hashes

### Pitfall 6: Migration for Existing Data
**What goes wrong:** Existing builds have no manifest_id or verification_status; existing workflows have no versions
**Why it happens:** New fields are added but old data not backfilled
**How to avoid:** Migration should: (1) Add all new fields with blank=True/null=True defaults, (2) Optionally create a data migration to set manifest_id on existing builds based on workflow_name convention, (3) Existing builds get verification_status="" (empty = "not verified, pre-versioning")
**Warning signs:** Existing builds breaking, null constraint violations

### Pitfall 7: PyGithub ContentFile Encoding
**What goes wrong:** Hash computed on base64-encoded content instead of actual file content
**Why it happens:** `get_contents()` returns base64 by default; must use `.decoded_content` which returns bytes
**How to avoid:** Use `content_file.decoded_content.decode('utf-8')` to get the string content for hashing
**Warning signs:** Hash never matches any CIWorkflowVersion

## Code Examples

### Hash Computation
```python
# Source: stdlib hashlib + design doc specification
import hashlib

def compute_manifest_hash(manifest_content: str) -> str:
    """Compute SHA-256 hash of manifest content (including header)."""
    return hashlib.sha256(manifest_content.encode("utf-8")).hexdigest()
```

# Source: docs/ci-workflows/build-authorization.md
def generate_manifest(self, workflow, version: str | None = None) -> str:
    """Generate deterministic manifest with version header."""
    import yaml

    yaml_body = yaml.dump(manifest, default_flow_style=False, sort_keys=False)

### Fetch Manifest Content via PyGithub
```python
# Source: PyGithub API (verified in installed package)
def fetch_manifest_content(
    self, config: dict, repo_name: str, manifest_id: str, commit_sha: str
) -> str | None:
    """Fetch manifest file content from repo at a specific commit."""
    g = self._get_github_client(config)
    repo = g.get_repo(repo_name)
    try:
        content_file = repo.get_contents(manifest_id, ref=commit_sha)
        if isinstance(content_file, list):
            return None  # Path is a directory, not a file
        return content_file.decoded_content.decode("utf-8")
    except Exception:
        return None  # File not found at that commit
```

### Verification Task
```python
# Source: docs/ci-workflows/build-authorization.md verification flow
@task(queue_name="build_updates")
def verify_build(build_id: int, connection_id: int, repo_name: str) -> dict:
    """Verify build manifest against authorized workflow versions."""
    build = Build.objects.get(id=build_id)

    # Skip if already verified or not terminal
    if build.verification_status or build.status not in ("success", "failed", "cancelled"):
        return {"skipped": True}

    service = build.service
    workflow = service.ci_workflow
    if not workflow:
        build.verification_status = "unauthorized"
        build.save(update_fields=["verification_status"])
        return {"status": "unauthorized", "reason": "no workflow assigned"}

    # Get plugin and compute manifest_id
    ci_plugin = get_ci_plugin_for_engine(engine)
    manifest_id = ci_plugin.manifest_id(workflow)

    # Fetch manifest content at build's commit
    content = ci_plugin.fetch_manifest_content(config, repo_name, manifest_id, build.commit_sha)
    if not content:
        build.manifest_id = manifest_id
        build.verification_status = "unauthorized"
        build.save(update_fields=["manifest_id", "verification_status"])
        return {"status": "unauthorized", "reason": "manifest not found"}

    # Compute hash
    manifest_hash = compute_manifest_hash(content)

    # Look up in CIWorkflowVersion
    version = CIWorkflowVersion.objects.filter(
        workflow=workflow, manifest_hash=manifest_hash
    ).first()

    if version:
        if version.status == "authorized":
            verification_status = "verified"
        elif version.status == "draft":
            verification_status = "draft"
        else:  # revoked
            verification_status = "unauthorized"
    else:
        verification_status = "unauthorized"
        version = None

    build.manifest_id = manifest_id
    build.manifest_hash = manifest_hash
    build.workflow_version = version
    build.verification_status = verification_status
    build.save(update_fields=[
        "manifest_id", "manifest_hash", "workflow_version", "verification_status"
    ])
```

### Version Suggestion Algorithm
```python
# Source: Claude's discretion - design doc says "suggests next patch version"
def suggest_next_version(workflow) -> str:
    """Suggest next version number for publishing."""
    latest = (
        CIWorkflowVersion.objects
        .filter(workflow=workflow, status="authorized")
        .exclude(version="")
        .order_by("-published_at")
        .first()
    )
    if not latest or not latest.version:
        return "1.0.0"
    try:
        v = semver.Version.parse(latest.version)
        return str(v.bump_patch())
    except ValueError:
        return "1.0.0"
```

### Publish Flow
```python
# Source: CONTEXT.md decision + docs/ci-workflows/versioning.md
def publish_version(draft_version, version_number: str, user) -> CIWorkflowVersion:
    """Publish a draft version."""
    # Validate version
    v = semver.Version.parse(version_number)  # raises ValueError if invalid

    # Check uniqueness
    if CIWorkflowVersion.objects.filter(
        workflow=draft_version.workflow, version=version_number
    ).exists():
        raise ValueError(f"Version {version_number} already exists")

    # Check ordering (must be greater than latest)
    latest = (
        CIWorkflowVersion.objects
        .filter(workflow=draft_version.workflow, status="authorized")
        .exclude(version="")
        .order_by("-published_at")
        .first()
    )
    if latest:
        latest_v = semver.Version.parse(latest.version)
        if v <= latest_v:
            raise ValueError(f"Version must be greater than {latest.version}")

    # Regenerate manifest with version number (makes hash unique per version)
    ci_plugin = get_ci_plugin_for_engine(engine)
    manifest_content = ci_plugin.generate_manifest(
        draft_version.workflow, version=version_number
    )
    manifest_hash = compute_manifest_hash(manifest_content)

    # Transition
    draft_version.version = version_number
    draft_version.status = CIWorkflowVersion.Status.AUTHORIZED
    draft_version.manifest_content = manifest_content
    draft_version.manifest_hash = manifest_hash
    draft_version.published_at = timezone.now()
    draft_version.save()

    return draft_version
```

## Key Integration Points

### 1. WorkflowComposerView.post() -> Draft Creation
Current: Saves directly to CIWorkflow + CIWorkflowStep
New: After saving, create/update draft CIWorkflowVersion with regenerated manifest and hash

### 2. poll_build_details -> verify_build Chain
Current: poll_build_details creates/updates Build, activates service on first success
New: After terminal status, enqueue verify_build task with build_id + connection_id + repo_name

### 3. push_ci_manifest -> Use Versioned Manifest
Current: Calls `ci_plugin.generate_manifest(workflow)` and `ci_plugin.manifest_path(service)`
New: Must use `manifest_id(workflow)` for file path, and generate manifest WITH version number from the service's pinned version

### 4. Service -> Workflow Version Pinning
Services need to track which CIWorkflowVersion they are currently using. This enables:
- Manifest push with correct version number
- Display of "version X.Y.Z is in use" in service CI tab
- Out-of-date detection when newer version is published

Recommendation: Add `ci_workflow_version` FK to Service model (nullable, SET_NULL on delete).

### 5. Builds Tab -> Verification Badge
Current: Build row shows status badge (success/failed/running)
New: Add verification badge next to status badge (Verified/Draft/Unauthorized)
