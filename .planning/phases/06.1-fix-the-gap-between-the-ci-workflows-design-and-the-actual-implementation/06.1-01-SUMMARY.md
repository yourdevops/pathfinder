## Accomplishments
- CIWorkflowVersion model with UniqueConstraints enforcing single-draft-per-workflow and unique-version-per-workflow
- Build model extended with manifest_id, manifest_hash, workflow_version FK, and verification_status for build authorization
- Service model extended with ci_workflow_version FK for version pinning and ci_manifest_push_method for PR vs direct push
- ProjectCIConfig.allow_draft_workflows BooleanField (default=False) as project-level gate for draft testing
- CICapableMixin extended with 4 new abstract methods (manifest_id, extract_manifest_id, get_manifest_id_pattern, fetch_manifest_content)
- GitHubPlugin implements all 4 new methods with MANIFEST_ID_PATTERN regex validation
- generate_manifest updated to accept optional version parameter and prepend deterministic header block

## Decisions Made
- CIWorkflowVersion uses TextChoices enum for status (draft/authorized/revoked) matching the state machine in versioning.md
- Build verification_status is a standalone CharField with choices, not directly tied to version status, allowing independent verification state
- Service ci_manifest_push_method defaults to "pr" (Pull Request) as the safer default
- manifest_id is workflow-name-based (`ci-{workflow.name}.yml`) rather than service-handler-based, per plugin-interface.md design
- generate_manifest version parameter defaults to None which renders as "draft" in the header comment
