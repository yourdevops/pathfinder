# Phase 6.1: Fix CI Workflows Design-Implementation Gap - Context

**Gathered:** 2026-02-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the gap between the CI Workflows design docs (`docs/ci-workflows/`) and the actual implementation. The core missing pieces are: CIWorkflowVersion model with full lifecycle (draft/authorized/revoked), build verification via manifest hash comparison, deterministic manifest generation with version headers, and plugin interface completion for manifest fetching. This enables Phase 7 (Deploy) to gate deployments on build verification status.

**What exists (built in 5.1, 5.2-partial, 5.3, 6):**
- CIWorkflow, CIStep, StepsRepository models with engine-agnostic plugin support
- Manifest generation via GitHubPlugin.generate_manifest()
- Push-to-repo via push_ci_manifest task
- Build model with webhook ingestion and poll_build_details
- Service.ci_workflow FK, ProjectApprovedWorkflow, ProjectCIConfig

**What's missing:**
- CIWorkflowVersion model (versioning lifecycle)
- Build verification flow (hash computation, version matching)
- Plugin methods: manifest_id, extract_manifest_id, get_manifest_id_pattern, fetch_manifest_content
- Deterministic manifest generation with version header
- Forking workflow support
- Version management UI (publish, revoke, version history)
- Build verification status display

</domain>

<decisions>
## Implementation Decisions

### Scope
- Full design parity with docs/ci-workflows/ — not just MVP
- **In scope:** Versioning lifecycle, build verification, manifest management, forking, draft testing, revocation
- **Deferred:** Auto-update policies (services always use pinned version, manual update), version cleanup automation (versions accumulate, manual deletion only)

### Workflow Versioning
- Draft auto-created on workflow edit — editing steps/config automatically creates or updates the single draft slot
- Publish flow via modal dialog — clicking "Publish" opens modal with version number input, changelog summary, and confirmation
- Version history as a tab on workflow detail page — sits alongside existing tabs
- Only operators can revoke versions — confirmation modal warns about impact on future builds

### Build Verification
- Webhook creates build record with initial data; background verify_build task fetches authoritative data from GitHub API — API response overwrites and becomes final source of truth
- Verification runs as separate background task — webhook marks build complete, verify_build task runs async (fetch API data, fetch manifest content, compute hash, set verification status)
- Verification status displayed as colored badge on build row — Verified=green, Draft=amber, Unauthorized=red
- Phase 6.1 computes and displays verification status only — deployment enforcement logic belongs to Phase 7 (Deploy)

### Manifest Management
- Manifest header per design doc — YAML comment block with "Managed by Pathfinder - DO NOT EDIT MANUALLY", workflow name, and version number
- Push method configurable per service — service settings include "Manifest push method" (PR or direct push)
- Publishing does NOT auto-push to services — user must explicitly push to each service after publishing a version (auto-update deferred)
- No skip-CI on manifest commits — let CI run on manifest pushes, producing verified builds

### Claude's Discretion
- Exact CIWorkflowVersion model fields beyond what's specified in design docs
- Migration strategy for existing workflows (creating initial versions)
- Fork UI placement and interaction pattern
- Background task queue configuration for verify_build
- Hash computation implementation details
- Version number suggestion algorithm in publish modal

</decisions>

<specifics>
## Specific Ideas

- Design docs are authoritative: `docs/ci-workflows/build-authorization.md`, `versioning.md`, `plugin-interface.md`
- Manifest header format exactly as specified in build-authorization.md
- CIWorkflowVersion state machine: DRAFT -> AUTHORIZED (publish) or delete (discard); AUTHORIZED -> REVOKED (revoke)
- Plugin interface must implement all 4 methods from docs/ci-workflows/plugin-interface.md: manifest_id, extract_manifest_id, get_manifest_id_pattern, fetch_manifest_content
- Build verification states: Verified, Draft, Unauthorized (per build-authorization.md)
- Pre-revocation builds keep Verified status with UI warning; post-revocation builds get Unauthorized
- Forking required when runtime set changes — per versioning.md change classification

</specifics>

<deferred>
## Deferred Ideas

- Auto-update policies (patch auto-update, manual minor/major) — future phase
- Version cleanup automation (content clearing after X days, version record deletion rules) — future phase
- Deployment gating enforcement — Phase 7 (Deploy)
- DeploymentMethod CD manifests (cd-*.yml pattern) — Phase 7

</deferred>
