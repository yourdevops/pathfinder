## Deviations from Plan

### Bugs Discovered During Verification

**1. [Rule 1 - Bug] Manifest file path showed service.handler pattern instead of manifest_id**
- **Found during:** Verification Step 8 (Manifest Header)
- **Issue:** Template displayed manifest path as `.github/workflows/{service.handler}.yml` instead of correct `ci-{workflow.name}.yml` pattern from manifest_id
- **Fix:** Changed template to use `{% with manifest_id=github_plugin.manifest_id value=workflow %}` for correct path display
- **Files modified:** core/templates/core/ci_workflows/workflow_detail.html
- **Verification:** Manifest tab shows correct ci-{name}.yml path
- **Commit:** 9e8c1f6

**2. [Rule 1 - Bug] Workflow name strip used [5:] instead of [3:] for "ci-" prefix**
- **Found during:** Verification Step 4 (Build Verification)
- **Issue:** Build classification logic stripped 5 characters from workflow name instead of 3 for "ci-" prefix, causing builds to land in "Other" tab with malformed names
- **Fix:** Changed `workflow_name[5:]` to `workflow_name[3:]` in create_or_update_build_from_webhook task
- **Files modified:** core/tasks.py
- **Verification:** Service builds tab now correctly shows workflow builds under proper service tabs
- **Commit:** cf78117

**3. [Rule 1 - Bug] push_ci_manifest regenerated manifest from current steps instead of using stored version content**
- **Found during:** Verification Step 2 (Publish Flow)
- **Issue:** push_ci_manifest called generate_manifest(steps) instead of using stored version.content, causing hash mismatches in build verification
- **Fix:** Changed push_ci_manifest to read manifest content from pinned CIWorkflowVersion.content field
- **Files modified:** core/tasks.py
- **Verification:** Pushed manifests now match published version content exactly, verification badges show "Verified"
- **Commit:** cf78117

**4. [Rule 1 - Bug] Service CI tab manifest preview always showed "Version: draft" instead of pinned version content**
- **Found during:** Verification Step 8 (Version Pinning)
- **Issue:** ManifestPreviewView always returned draft or authorized version, ignoring Service.ci_workflow_version FK
- **Fix:** Updated ManifestPreviewView to check service.ci_workflow_version first, then fall back to draft/authorized priority
- **Files modified:** core/templates/core/services/_ci_tab.html
- **Verification:** CI tab manifest preview now shows pinned version content when version is pinned
- **Commit:** f69b20c
